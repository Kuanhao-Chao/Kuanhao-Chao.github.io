[LiftOn](/posts/lifton/) ([Chao et al., 2025](https://doi.org/10.1101/gr.279620.124)) was built around a simple observation: DNA alignment and protein alignment fail in different ways. DNA lift-over with [Liftoff (Shumate and Salzberg, 2021)](https://doi.org/10.1093/bioinformatics/btaa1016) preserves gene structure well when two genomes are close, but it can introduce frameshifts, premature stops, and shifted exon boundaries as sequence diverges. Protein-to-genome alignment with [miniprot (Li, 2023)](https://doi.org/10.1093/bioinformatics/btad014) survives divergence better, but it can be fragmentary and does not naturally preserve the reference transcript structure.

LiftOn combines the two. For each gene, it compares the Liftoff model and the miniprot model against the reference protein, chains together the coding segments that best preserve that protein, and then runs an open-reading-frame rescue step to repair boundary problems. The first release showed that this dual-evidence strategy improves protein-coding annotation transfer across genomes and across species.

**LiftOn v1.0.9** keeps that scientific idea and extends the algorithm and execution engine for whole [RefSeq genomes (O'Leary et al., 2016)](https://doi.org/10.1093/nar/gkv1189), complete assemblies, and larger cross-species workloads. Its main contribution is making the dual-evidence method bounded, conservative, and testable at whole-genome scale.

## What changed

The first release was accurate, but it was engineered around chromosome-scale runs. Whole genomes exposed four practical problems. A few giant genes could drive full dynamic-programming alignment into tens of gigabytes of memory. Some full plant genomes crashed partway through, leaving partial annotations that looked usable but were not complete. The pipeline spent too much time in serial execution and disk round-trips. And the default lift covered protein-coding genes while quietly leaving pseudogenes, long non-coding RNA genes, tRNAs, rRNAs, and other gene-like features behind.

LiftOn v1.0.9 addresses those problems with four main updates.

First, it replaces the most memory-hungry alignments with an **anchor-windowed aligner**. LiftOn scores candidate genes by aligning their translated proteins back to the reference protein. A full [Needleman-Wunsch alignment (Needleman and Wunsch, 1970)](https://doi.org/10.1016/0022-2836(70)90057-4) with traceback is quadratic in memory, so one giant gene can dominate the entire run. The new aligner finds exact shared anchors, chains them into a consistent backbone, and runs full dynamic programming only inside the windows between anchors. For ordinary homologous genes, this keeps the same answer while bounding peak memory. On the largest memory cases, the practical change is dramatic: the benchmark shows controlled same-input runs using up to about 24x less memory, and the most memory-intensive four-way mouse subset drops from 44.6 GB in v1.0.8 to 0.76 GB in v1.0.9.

Second, it makes the DNA-plus-protein merge more conservative. The old LiftOn always emitted the merged model after protein maximization. That is usually right, but a local improvement can still perturb a downstream reading frame. LiftOn v1.0.9 now evaluates the merged model and the DNA-only model after ORF rescue, then emits the one with the better reference-protein identity. In other words, miniprot evidence is used when it improves the final transcript, not simply because it exists. On identical aligner inputs, that change has the shape we wanted: many improvements and almost no regressions, including 115 improved transcripts and 1 regressed transcript on the fruit-fly benchmark, and 83 improved with 0 regressed on mouse-to-rat.

Third, LiftOn now lifts **every gene-like feature type** by default. Instead of assuming only `gene` parents matter, it scans the reference annotation for top-level feature types that behave like genes: a parent record with transcript and exon children. That brings pseudogenes, non-coding RNA genes, tRNAs, rRNAs, snoRNAs, snRNAs, and related structured features into the lift. The change matters because these features are part of the annotation, not decoration. Dropping them means downstream analyses inherit a narrower genome map than the reference actually provides.

Fourth, the execution engine was rebuilt under a byte-identity contract. The two external aligners can run concurrently. The per-locus merge can run across worker threads. Some intermediate files can be streamed or passed in memory. miniprot now receives the requested thread count. These are engineering changes, so they should not silently change the annotation. LiftOn v1.0.9 protects that boundary with a 24-cell regression matrix: every combination of streaming, in-memory Liftoff handoff, thread count, and native bindings must reproduce the default GFF3 byte-for-byte. If a speed path changes one byte of output, it fails.

## What the benchmark shows

I evaluated LiftOn v1.0.9 the way the tool is actually used: seventeen whole-genome, end-to-end lift-overs, each tool running from scratch on a complete target assembly. The seventeen are organized as a four-tier divergence ladder — same-species (independent assemblies and cultivars of Arabidopsis, honeybee, rice, maize, tomato), close cross-species (human to gorilla, mouse to *Mus caroli*, tomato to potato), distant cross-species (dog to cat, human to macaque, human to marmoset), and very-distant (Arabidopsis to rice eudicot-to-monocot; human to zebrafish, chicken, and frog; Drosophila to honeybee), where genome-wide DNA synteny has all but disappeared. The comparison includes four tools: Liftoff alone, miniprot alone, LiftOn v1.0.8, and LiftOn v1.0.9. Every output is re-scored by the same evaluator rather than by each tool's own reporting: for coding transcripts, the evaluator translates the lifted CDS, aligns it to the reference protein, and computes mean protein identity. (A companion suite of single-chromosome subsets, used only to isolate the engine's speed and memory, lives in the technical report.)

On all seventeen whole genomes, LiftOn v1.0.9 produced a higher mean protein identity than both single-method baselines, and the set-mean margin over the better of the two grew with divergence — from +0.0007 on same-species Arabidopsis to +0.13 at the mammal-to-bird extreme (human to chicken). Two caveats keep that honest. First, against Liftoff — the DNA baseline LiftOn extends — the win is unambiguous: as the target diverges the DNA model frameshifts and Liftoff collapses from about 0.99 to 0.15–0.31 across the very-distant tier, while the protein model survives and the merge keeps whichever signal is locally better. Second, that headline margin is each tool's mean over its own recovered set; on the very-distant tier, where LiftOn and miniprot recover very different fractions of the genome, it is partly a denominator effect — the apples-to-apples reading (below) is more nuanced. Against the previous release, LiftOn v1.0.9 matched or beat v1.0.8 on the great majority of the genomes both completed — every same-species, cross-species, close, and distant pair — with only fractional dips on the three most extreme distances (human to zebrafish, chicken, and frog, each about 0.0002–0.002, on sets both recover at under 25 percent). On Arabidopsis, rice, maize, and two tomato pairs, v1.0.8 never produced a comparable number at all — it crashed mid-run (more below).

The largest practical result is not an accuracy decimal. It is that LiftOn v1.0.9 finishes full genomes that v1.0.8 abandoned — five of the seventeen. On full RefSeq *Arabidopsis*, v1.0.8 crashed after recovering 13,618 coding transcripts, about 28% of the reference set; LiftOn v1.0.9 completed the run and recovered 48,207, adding 34,589. On full rice it stopped at 32,933, about 77%; LiftOn v1.0.9 finished with 42,527, adding 9,594. On maize and two tomato genomes, v1.0.8 crashed with no usable whole-genome output at all, while LiftOn v1.0.9 completed each. A separate hardening fix matters here too: on the dog-to-cat lift, a single malformed transcript — one with inverted coordinates produced by the ORF-rescue step — used to abort the entire run partway through; LiftOn v1.0.9 now drops that one transcript with a logged warning and finishes the genome, 61,276 transcripts instead of 11,048. The lesson the whole-genome benchmark kept teaching is that one bad feature must never be allowed to sink an entire genome.

The same figure also shows the broader feature lift. On full *Arabidopsis*, LiftOn v1.0.9 recovers thousands of gene-like features that v1.0.8 omitted by design: 4,816 pseudogenes, all 3,878 lnc_RNA genes in the reference set, many more tRNAs and snoRNAs, and nearly all rRNA and antisense RNA records. This does not mean every nested product is solved. Mature miRNA recovery remains poor in this release because those records are nested under primary transcripts in a way the current transcript-centric logic does not yet handle well. But the parent gene-like features now come through, and the annotation is much less artificially coding-only.

Completeness still needs a careful reading, and the whole genomes make the trade-off concrete. On the same-species through distant genomes, every tool that finishes recovers most or all of the coding set. On the five very-distant genomes, miniprot recovers far more coding transcripts than LiftOn — roughly 67 to 86 percent versus LiftOn's 4.5 to 24 percent — because it searches directly with proteins and is not anchored to a DNA lift. LiftOn's goal is different: keep reference-consistent transcript structure and emit accurate gene models where a reliable lift exists. On raw recall alone, protein search wins on the hardest pairs; LiftOn instead prioritizes the identity and structure of the transcripts it recovers. (There is a smaller mirror of this trade-off on the close and distant mammalian pairs, where LiftOn recovers a fraction of a percent fewer transcripts than Liftoff in exchange for higher protein identity.)

The very-distant whole genomes sharpen the picture — and demand a careful reading of what the accuracy numbers mean. Where DNA synteny is essentially gone, Liftoff's DNA-only model collapses to between 0.15 and 0.31 mean protein identity while LiftOn v1.0.9 holds far higher, so against Liftoff its lead is largest exactly here. But the set-mean lead over the *best* single method (which at this distance is miniprot) is mostly a denominator effect: LiftOn recovers only the small DNA-anchored core (about 4.5 to 24 percent of coding transcripts) and averages over that easy set, while miniprot recovers far more (about 67 to 86 percent) and its mean is pulled down by the larger, harder set. The fair comparison is on the transcripts both tools place. Read that way, LiftOn's per-transcript edge over miniprot is real and peaks on the *distant* tier (dog-to-cat, human-to-macaque, human-to-marmoset, +0.023 to +0.032), but it does *not* hold at very-distant — on four of the five very-distant genomes miniprot is as accurate or slightly more so on the shared transcripts (human-to-chicken, human-to-frog, human-to-zebrafish, Arabidopsis-to-rice). What *is* a clean per-transcript win at extreme distance is LiftOn v1.0.9 over v1.0.8, which recover the same small set and so compare without that artifact. The honest summary: at extreme distance LiftOn delivers high-precision refinement of the recoverable core, not higher recall — and closing the recall gap is the role of the miniprot-only rescue path, which v1.0.9 now ships **on by default**. It runs as a separate pass after the main lift, emitting a model for each reference coding gene the DNA lift misses entirely, de-duplicated so it only adds genes; across eight distant/very-distant test pairs it recovers a net-positive set with zero lost and zero redundant models (for example, +555 genes on a nematode pair and +201 on *Drosophila*-to-*Anopheles*). The four-way benchmark above freezes it off (`--no-miniprot-rescue`) so the comparison isolates the merge engine apples-to-apples.

Output validity improves where scale stresses the pipeline. v1.0.9 adds a write-time containment normalization that repairs the duplicate-exon-ID and out-of-bounds-CDS errors LiftOn's chaining used to introduce — without touching any coding sequence, so protein identity is unchanged. Re-running every output through one validator for the [GFF3 specification](https://github.com/The-Sequence-Ontology/Specifications/blob/master/gff3.md), the released v1.0.9 now **validates clean (zero errors) on 9 of the 17 whole genomes** — including all five human cross-species lifts — and emits far fewer errors than v1.0.8 on every genome both versions produced (honeybee 103 to 0, Drosophila 150 to 0, dog-to-cat 159 to 15, Arabidopsis-to-rice 225 to 35), as well as fewer than the DNA-only Liftoff on every genome. The remaining errors sit on the plant genomes and are reference-inherited, not LiftOn's doing: they are organellar genes annotated with coding sequence directly under the gene (and duplicate organellar IDs), which the strict validator flags and LiftOn faithfully carries from references that themselves fail the same checks more (the Arabidopsis reference alone carries 159). A lift-over tool is only useful if its accuracy claims and its failure modes are both explicit.

Finally, the performance numbers explain why the new version changes practical use. On identical cached aligner inputs — a controlled comparison that isolates LiftOn's own code — v1.0.9 is faster end to end and much lighter on memory: on the most memory-intensive case it peaks at 0.76 GB against v1.0.8's 44.6 GB, a 59-times reduction, and runs up to 2.15 times faster. The memory reductions come from the windowed aligner, the speedups from concurrent aligners, per-locus threading, and less disk and database overhead. On a fresh whole-genome run v1.0.9 can do more total work than v1.0.8 because it lifts the expanded feature set, so wall time can rise; the decisive point is that the memory bound and the robustness fixes are what make those whole-genome runs finish at all.

## What changes for users

The basic command is unchanged:

`lifton -g reference.gff3 reference.fa target.fa -o lifton.gff3`

The defaults are more complete and more conservative. LiftOn v1.0.9 lifts all gene-like features by default, uses the best-of-outcome merge, and uses the memory-bounded aligner. If you need to reproduce the published LiftOn 1.x behavior, the relevant opt-out flags are `--gene-only`, `--legacy-merge`, `--full-dp-align`, and `--serial-aligners`.

For larger runs, `--threads N --locus-pipeline` enables the threaded per-locus path, while `--stream` and `--inmemory-liftoff` reduce intermediate file overhead. These speed paths are covered by the byte-identity matrix, so they are performance choices rather than analysis choices: the annotation should be byte-for-byte identical to the default output.

The main thing I want users to take from LiftOn v1.0.9 is that genome annotation lift-over needs both biological evidence and software discipline. The biological argument is still DNA plus protein: each signal catches errors the other misses. The engineering argument is now just as important: the tool has to finish full genomes, fit in realistic memory, lift the annotation types people expect, and make performance refactors prove that they did not change the science.
