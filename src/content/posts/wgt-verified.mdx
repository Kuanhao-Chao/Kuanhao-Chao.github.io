[WGT](/posts/wgt/), the Wheeler Graph Toolkit, was built around a useful fact: a special class of graphs — **Wheeler graphs** — can be indexed and searched with the same machinery that makes the Burrows–Wheeler Transform and the FM-index fast. That makes them a natural backbone for graph-based pangenome indexes. The catch is that deciding whether an arbitrary graph *is* a Wheeler graph is NP-complete, so WGT's recognizer, **Wheelie**, makes the problem tractable by first running a renaming heuristic that narrows each node's possible position, and only then handing the much smaller leftover problem to a permutation search or an SMT solver.

This update keeps that method exactly as it is. It is not a new algorithm; it is a verification and engineering pass on the existing one, with three goals: make the recognizer's verdicts **trustworthy** by checking them against an independent ground truth, make the full-range SMT solver **faster** with a leaner encoding, and add the ability to **repair** a graph that is not a Wheeler graph into one that is. The work was less about a headline result and more about putting a foundational tool on a firmer footing.

## What changed

A recognizer is only as useful as it is trustworthy: if it ever calls a non-Wheeler graph a Wheeler graph, every downstream conclusion built on that answer is unsupported. So the first and most important change is not a speedup — it is a way to *know* the answers are right.

First, I built an **independent brute-force oracle**. It is deliberately simple, so that it is obviously correct: for a graph with *n* nodes it enumerates all *n*! node orderings and, for each, checks the three Wheeler-graph axioms directly against the graph's actual edges. That is hopelessly slow — it is only usable up to about nine nodes — but on those small graphs it is an incontrovertible source of truth, computed by completely different code from the recognizer. With the oracle in hand I wrote a differential-testing harness that generates large families of graphs (De Bruijn graphs, tries, reverse-deterministic graphs, and random graphs, in both DNA and protein alphabets, plus hand-built edge cases with self-loops and parallel edges) and compares the recognizer's verdict to the oracle's on every one.

That comparison is what hardened the recognizer. It surfaced a handful of edge cases in the accept paths, and each was tightened: the permutation backend now explicitly rejects when its pruned search is exhausted rather than falling through; the fast path that settles an ordering by propagation alone now re-checks the axioms before accepting; an integer-overflow guard was added to the solver-dispatch heuristic; and the SMT path now guards degenerate inputs (empty or isolated-node graphs) that previously tripped the solver. After this hardening the recognizer agrees with the oracle on **every decided graph** — 2,447 of 2,447, with zero disagreements — and, on a corpus deliberately stuffed with non-Wheeler graphs to stress exactly the accept paths, it produces **zero** false-accepts across all of its backends.

Second, I rebuilt the **exponential reference implementation** into an honest decision procedure. WGT ships a reference implementation of the only previously proposed recognition algorithm (Gibney–Thankachan) so that the fast recognizer can be checked against an independent second opinion. To make that second opinion genuinely independent and decisive, I rewrote its core so that it enumerates orderings and verifies the three axioms against the graph's real edges — exactly mirroring the brute-force oracle — and returns a clean three-way verdict (Wheeler / not a Wheeler graph / undecided within its budget). It is now a trustworthy cross-check rather than a timing curiosity.

Third, I sharpened the **full-range SMT encoding**. WGT's `-f` mode skips the range-narrowing heuristic and hands the entire ordering space to the SMT solver — useful as an independent path and for graphs where you want no heuristic in the loop. The bottleneck there is how many constraints the encoding emits. The new encoding expresses the same axioms with far fewer terms: the cross-label "smaller label means earlier head" constraint drops from quadratic in the number of edges to roughly linear, and the within-label ordering constraint drops from quadratic in a label's edges to roughly quadratic only in its *distinct* endpoints. The two encodings are logically equivalent — they accept and reject exactly the same graphs — but the leaner one is much cheaper to build and to solve. The honest cost is memory: on graphs small enough that the old encoding never exploded, the new one uses about twice the RAM, because it trades a large number of constraints for a smaller number of auxiliary variables. It is a time-for-space trade on those graphs, not a free win, and the benchmark reports it as such.

Fourth, WGT can now **repair** a graph. Given a directed acyclic graph that is *not* a Wheeler graph, the new module unfolds it into the trie of its path-strings. A trie is always a Wheeler graph — you order its nodes by the strings spelled along incoming paths — so this always succeeds for a DAG, and crucially it preserves the **set of strings** the graph spells, exactly. Every repair self-verifies three ways: the recognizer accepts the result, the oracle accepts it when it is small enough, and the set of path-strings is confirmed identical to the input's. This turns "your graph is not a Wheeler graph" from a dead end into a constructive next step.

## What the benchmark shows

I evaluated each change against the question it is meant to answer, and kept those questions separate. Conflating them is how tools end up looking better than they are: a recognizer that returns answers quickly but sometimes returns the wrong one has not improved. So the results below are organized along three independent axes — **correctness** (is the verdict right?), **capability** (can a verdict be produced at all?), and **performance** (how fast, and how large, when it is right?).

On **correctness**, the picture is clean. On the reject-heavy corpus of non-Wheeler graphs — built specifically to probe the accept paths — the hardened recognizer produces zero false-accepts on every backend, and it agrees with the independent oracle on all 2,447 graphs both can decide, with no disagreements at all.

On **capability**, the rebuilt exponential reference now does something it could not before: it returns real verdicts where it used to run out of budget. On the subset of the benchmark small enough for the oracle to decide (931 graphs with at most nine nodes), the honest decision procedure decides all 931 — including seven non-Wheeler graphs built from real human gene orthologs that the previous reference could not classify within its budget. That is a capability gain, not a timing one: these are answers that simply did not exist before, regardless of how long you waited.

On **performance**, the bottom line is a median **~1.8× faster** full-range solve (up to 3.5×), with zero regressions — earned by handing Z3 a median **~13× smaller** formula, and without moving the size ceiling or hiding the memory cost. The idea is simple: we made the solver's homework smaller. The full-range path writes the whole node-ordering problem out as a logic formula and hands it to Z3, so the size of that formula matters. The old encoding wrote it as an all-pairs comparison — its size grows with the *square* of the number of edges. The new encoding brackets each group of edges into a small window instead, and the formula shrinks dramatically: we can count it exactly (it matches Z3's own assertion count to the atom), and on real gene graphs it drops by a median of about thirteen-fold — and far more on the largest. On the few-letter DNA alphabet the shrink is even asymptotic — the formula grows like $E^{1.6}$ rather than $E^2$ — while on the twenty-letter amino-acid alphabet it is a large constant-factor reduction. Building the problem then becomes nearly free (about fourteen times faster on the headline graph), and the total time roughly halves. The honest caveat is that **a smaller formula makes the solver faster, not fundamentally able to handle bigger graphs** — the size ceiling of the full-range path is set by the solver, and it does not move. The graph that scales is the *default* recognizer (next section), which never builds this formula at all.

Finally, two practical questions: how large a graph can the recognizer handle, and does repair actually work? Under a ten-minute-per-graph budget, the default heuristic-plus-SMT recognizer decides synthetic worst-case graphs of around 2,800 nodes — far past what the permutation backend or the full-range path reach, and orders of magnitude past the exponential reference's hard limit. And repair is robust: across 820 random DAGs, every one that needed repair was repaired and verified, with no failures.

## What changes for users

The commands are unchanged. The recognizer still runs on a DOT graph with `./bin/recognizer graph.dot` for the default heuristic-plus-SMT path, `-s p` for the permutation backend, and `-f` for the full-range SMT path; the `-f` runs are simply faster now. The generators and the visualizer are untouched. The one genuinely new entry point is repair: pointed at a DAG that the recognizer rejects, the `repair/wheelerize.py` module returns a Wheeler graph that spells the same set of strings, and checks its own work before handing it back.

The main thing I want to carry over from this pass is an ordering of priorities. The most consequential change is not the faster encoding or even the new repair capability — it is that the recognizer's verdicts are now checked against an independent ground truth and a trustworthy second opinion, so a "yes, this is a Wheeler graph" can be relied on. Speed and capability matter, but they come after correctness, because a faster wrong answer helps no one. Building a foundational tool means earning that trust, and then making it quick.
