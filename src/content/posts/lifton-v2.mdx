When we built [LiftOn](/posts/lifton/), the pitch was simple: DNA alignment and protein alignment fail in different ways, so don't choose between them — combine them, and let the protein decide what a correct gene ought to look like. LiftOn runs [Liftoff](https://doi.org/10.1093/bioinformatics/btaa1016) (which lifts genes by DNA alignment) and [miniprot](https://doi.org/10.1093/bioinformatics/btad014) (which aligns reference proteins to the genome), then reconciles their two annotations with a *protein-maximization* algorithm — chaining together whichever coding pieces, from either tool, best reconstruct the reference protein, and repairing the open reading frame afterward. That was the idea, and it worked.

But an idea that works on one chromosome isn't the same thing as a tool you can point at a genome and walk away from. The time since has gone into the quieter, less glamorous half of the job: making LiftOn hold up at the scale people actually want to use it. That scale has moved. Complete, telomere-to-telomere assemblies and whole pangenomes are now routine, and people don't run lift-over on a tidy test chromosome — they run it on entire RefSeq genomes, dozens of species at a time.

Pushed that hard, the first version showed its seams. It could crash partway through a full plant genome and hand you a half-finished annotation. A single giant gene like *titin* could drive it past forty gigabytes of memory. It carried protein-coding genes but silently left pseudogenes, long non-coding RNAs, and the rest of the non-coding annotation behind. And it ran on a single core while your genome sat there waiting. A method that works on a chromosome but falls over on the genome isn't finished.

**LiftOn 2.0** is that unfinished half — the same protein-maximization idea, rebuilt to run whole genomes quickly, lightly, completely, and without falling over. None of that means much as a list of adjectives, so the back half of this post is a twenty-dataset, four-way benchmark that measures every claim: LiftOn 2.0 against the previous release, and against each of its two underlying tools on their own.

## What changed under the hood

The first thing I built wasn't a feature — it was a guarantee. Reworking the guts of a tool is dangerous precisely because it's so easy to make it faster and subtly wrong at the same time, and a lift-over tool that silently corrupts one gene in a thousand is worse than useless, because you'll trust it. So before touching anything, I pinned the output. LiftOn has a number of optional fast paths — it can stream one aligner's output straight into the next stage instead of going through disk, hand a result off in memory, run on one to several threads, or drive its aligners through in-process language bindings. I wrote a test that runs the pipeline under *every combination* of those switches — two dozen of them — and demands that all of them produce a result that is **byte-for-byte identical** to the plain, default run. That matrix runs on every change. With it in place I could be ruthless about the internals, because anything that changed a single byte of output failed loudly and immediately, in continuous integration, before it ever reached a real genome.

The worst problem was memory. To score a lifted gene, LiftOn aligns its protein back to the reference, and an exact alignment is quadratic in length — the work and the memory both grow with the *square* of the protein. For an ordinary gene that's nothing; for a giant like *titin*, whose protein runs to tens of thousands of residues, a single alignment could swallow tens of gigabytes and take the whole run down with it. LiftOn 2.0 aligns in *anchored windows* instead: it finds short, confident exact matches between the two proteins, chains them into a consistent backbone, and then runs the expensive exact alignment only inside the small gaps between anchors. Where the two proteins genuinely correspond — the common case — this is exact, identical to the old full alignment; only in long, anchorless, divergent stretches does it fall back to a bounded approximation. On one mouse run, peak memory for the giant genes fell from roughly 45 GB to 2.6 GB. That single change is the difference between a job that needs a large dedicated compute node and one that finishes on a laptop.

Speed came from several directions at once, all of them within the byte-identity net. The two aligners LiftOn depends on are independent, so they now run *concurrently* instead of one after the other — Liftoff on the main thread, miniprot alongside it — and the wall-clock cost of the pair drops to the slower of the two. The expensive per-gene work in the merge stage is spread across worker threads, each one handling a locus while the alignment kernel releases the interpreter lock. miniprot is finally told how many cores it may use, instead of quietly defaulting to four no matter what you asked for. And a cluster of database queries that fired two or three times per gene were collapsed into one. None of this changes the answer; all of it makes the same run faster.

Two changes deliberately *do* change the output — for the better — and each keeps an escape hatch. The first is completeness: the original carried only the protein-coding `gene` hierarchy and stopped there, so pseudogenes, non-coding RNA genes, and structured mobile elements were left behind. LiftOn 2.0 detects every *gene-like* feature type present in the reference and lifts them all by default. The second is the merge itself. The old version always applied its combined, chained gene model; 2.0 keeps that model only when it scores *strictly higher* against the reference protein than the plain DNA lift does — so the merge can only ever help, never corrupt a gene that Liftoff already had right. Both behaviors are reversible with a flag, for anyone who needs to reproduce the published numbers exactly.

How does LiftOn know what counts as "gene-like"? It scans the reference once and picks out every top-level feature type with the *shape* of a gene — a parent that has transcript and exon children — and lifts those, while leaving purely structural records (whole-chromosome regions, bare alignment matches) alone. And the open-reading-frame rescue that gives LiftOn its name still runs on top of all of it: when a lifted gene comes through with a frameshift, a premature stop, or a damaged start codon, LiftOn searches the neighborhood for an alternative boundary or start site that recovers the largest possible stretch of the reference protein. The windowed aligner had to preserve that rescue *exactly* — same reading frame chosen, same boundary repaired — which is the harder half of the memory fix: it's easy to bound the work if you're allowed to change the answer, and the whole point was not to.

Finally, the crashes. Running LiftOn 2.0 across whole genomes turned up two — one I'd inherited from the original and one I introduced while making the completeness change above — and both are fixed. Each was the kind of bug you only ever hit at scale: a particular feature arrangement that doesn't occur on a test chromosome but is guaranteed to occur somewhere in a real genome. That's exactly why scale was the point, and why the benchmark below runs on whole genomes and not just chromosomes.

## What we measured

To make changes like these with any confidence, I had to know exactly what each one did to the output. So I built a benchmark harness and pointed it at twenty datasets arranged along a *divergence ladder* — from same-species lift-overs (human, mouse, rice, *Arabidopsis*, honeybee, carried between independent assemblies) out to increasingly distant cross-species pairs (mouse to rat, chicken to quail, *C. elegans* to *C. briggsae*, human to mouse) — and across two annotation databases, RefSeq and Ensembl. On every dataset I ran four tools side by side: **Liftoff** (DNA alignment alone), **miniprot** (protein alignment alone), the previous **LiftOn v1.0.8**, and **LiftOn 2.0**.

The scoring is deliberately tool-neutral. It would be easy — and meaningless — to let each tool report its own accuracy. Instead, every tool's output is re-scored the same way by the same code: take each lifted gene, translate its coding sequence, align that protein back to the reference protein, and measure percent identity. No tool grades its own homework. I report the mean of that identity over the coding transcripts each tool recovered. Sixteen of the datasets are a single representative chromosome — small enough to score every transcript quickly and to compare the two LiftOn versions on *identical* aligner inputs, which isolates the algorithm from any run-to-run noise in the aligners. The other four are whole genomes, the headline robustness test. One note on those: when a tool crashes partway through a genome, the accuracy it reports over the fraction it finished isn't comparable to a tool that finished the whole thing, so I keep crashed runs out of the head-to-head *accuracy* comparison and show them where they belong — in completeness, where finishing or not is the whole point.

A word on fairness, because benchmarks live and die by it. For the per-chromosome accuracy comparison between the two LiftOn versions I fed both the *same* cached aligner outputs, so any difference is the algorithm and nothing else — not a lucky re-run of Liftoff or miniprot. For completeness I let each version run its aligners end to end, because the whole point of 2.0's gene-like default is that it asks the aligners for more, and pinning the inputs would hide exactly the effect I wanted to measure. And throughout, I kept each tool's *self*-reported identity separate from the neutral re-score, as a check that the neutral scorer wasn't quietly favoring anyone. None of the headline numbers here depend on a tool's own accounting of how it did.

**It's the most accurate tool in the field — and the lead grows with distance.** On every one of the datasets where both LiftOn versions ran to completion, LiftOn 2.0 matches or beats v1.0.8; against the better of the two single-method baselines, it matches or beats them on 19 of 20. The aggregate picture is in Figure 1A: on same-species data all four tools cluster near perfect identity, as they should, but as the species diverge they fan out. Liftoff, working from DNA alone, falls off a cliff; miniprot, reading the more conserved protein, holds up far better; and LiftOn, free to use both, stays highest and pulls *further* ahead exactly where the problem is hardest. The per-dataset gains over the best single method (Figure 1B) tell the same story, sorted by divergence: small on the easy pairs, large on the hard ones — mouse to rat **+0.056**, chicken to quail **+0.050**, *C. elegans* to *C. briggsae* **+0.072**. There is one honest exception, and it's worth stating plainly: on human-to-mouse, miniprot's protein alignment beats LiftOn outright, by under a hundredth. At that evolutionary distance the DNA signal Liftoff contributes is so degraded (it manages only ~0.50 identity there) that leaning on protein alone would have been the better bet — a reminder that combining evidence helps most when *both* signals still carry information.

Two things in that improvement are easy to miss. First, the gains over v1.0.8 come almost entirely from the more conservative merge, and they are remarkably one-sided (Figure 1C): on identical inputs, fruit fly improved **115** transcripts and regressed exactly **one**; mouse-to-rat improved **83** and regressed **none**. That asymmetry is the point of the "keep the merged model only if it scores strictly higher" rule — it can fix a gene but can't break one. Second, the absolute numbers, tool by tool and dataset by dataset, are worth seeing directly rather than as an aggregate, so Figure 2 lays them all out.

**It finishes genomes the old version couldn't.** The single most important result in this whole release isn't an accuracy number — it's that LiftOn 2.0 *finishes*. Pointed at the full *Arabidopsis* RefSeq genome, the old version crashed partway through and completed barely a quarter of the annotation; on rice it stalled at about three-quarters. LiftOn 2.0 runs both to completion (Figure 3A), which by itself recovers **34,589** more coding transcripts on *Arabidopsis* and **9,594** more on rice. This is the difference in kind I keep coming back to: a tool that dies a quarter of the way through a genome isn't a little worse than one that finishes — it's unusable for that genome, and the partial file it leaves behind is a trap, because it looks like an annotation.

**It lifts the parts of the genome the old version ignored.** Because LiftOn 2.0 lifts every gene-like feature type by default, the annotations it produces are categorically more complete (Figure 3B). On *Arabidopsis* it carries **4,816** pseudogenes the old version didn't emit at all, and substantially more of the rest of the non-coding annotation — thousands more long non-coding RNA genes (1,197 → 3,878), hundreds more transfer RNAs (235 → 663) and small nucleolar RNAs (91 → 283), nearly all of the ribosomal and antisense RNA. The pattern repeats on rice (pseudogenes 0 → 1,593, long non-coding RNAs 4,487 → 6,271) and on the cross-species pairs. These aren't protein-coding genes, but they are real annotation, and a lift-over that drops them quietly truncates whatever you build next. One case stays honest about its limits: mature microRNAs, which sit nested inside another feature in a way LiftOn still doesn't handle well, come through at a dismal 1 in 428. The parent features are lifted; the mature products aren't. It's on the list.

**On raw completeness, it matches the DNA lifter — and miniprot still leads on distant pairs.** Completeness deserves a careful, honest look, because it's the one axis where LiftOn is not always the leader (Figure 4). On same-species and close cross-species data, all four tools recover essentially the same coding transcripts. But on the *distant* pairs, miniprot — which goes looking for genes by their protein — simply finds more of them than any DNA-based method can: on *C. elegans* to *C. briggsae* it recovers 74% of coding transcripts where Liftoff manages 22%. LiftOn builds on Liftoff's DNA lift and adds a miniprot-rescue pass for genes a strict one-to-one map misses, so it recovers at least as many transcripts as Liftoff — on this pair somewhat more (34% vs 22%) — at higher identity than either, but still well short of a pure protein search on the most diverged genomes. That's the honest shape of it: on completeness, LiftOn is as good as the DNA lifter it builds on or better, broader in the *kinds* of features it carries, and — uniquely among the four — able to finish a whole genome; out-recovering miniprot's raw protein search on the hardest pairs was never the goal. Where the two LiftOn versions differ, 2.0 is even or slightly ahead; the headline completeness win over v1.0.8 is the full-genome robustness in Figure 3, not the per-chromosome coding counts.

**It stays valid as it scales.** More output is only better if the output is well-formed, so I check every annotation with an independent GFF3 validator and count the errors (Figure 5). LiftOn 2.0 is at or below v1.0.8 on 19 of the 20 datasets, and markedly cleaner on the full genomes — on full *Arabidopsis* its error count falls from 149 to 107, on rice from 174 to 129, on the *Drosophila* genome from 150 to 100 — while emitting far *more* annotation. There is exactly one dataset where 2.0 is worse: on the *S. cerevisiae*-to-*paradoxus* pair it produces 11 validator errors against v1.0.8's 10. I'm showing it rather than hiding it; one extra error on one yeast chromosome is the honest cost of a release that is cleaner everywhere else and on every whole genome.

**It's faster and far lighter.** On identical inputs, LiftOn 2.0 is faster end to end — modestly on small genomes, more than twice as fast on others (honeybee **2.15×**, mouse-to-rat **2.03×**) — and the memory difference is dramatic (Figure 6): the windowed aligner cuts peak RAM by up to **23.9×** on the very same runs, from nearly 11 GB down to under half a gigabyte on the honeybee benchmark. The speedups and the memory savings come from different changes — concurrency and threading buy the time, the windowed aligner buys the memory — and they compound. Taken together they're the difference between a tool you schedule on a cluster and one you run on the machine in front of you.

**A walk down the divergence ladder.** It's worth tracing the whole ladder once, because the shape of the result is the argument for combining DNA and protein in the first place. At the easy end — same-species lift-overs between independent assemblies of human, mouse, rice, *Arabidopsis*, and honeybee — every tool is near identity and the differences are in the third decimal; LiftOn 2.0 leads, but only just, because there's almost nothing to fix. One rung out, at modest cross-species distance, the gap opens: on fruit fly LiftOn 2.0 reaches 0.926 protein identity where Liftoff alone manages 0.908 and miniprot 0.921, and on mouse-to-rat it reaches 0.909 where *both* baselines sit around 0.85 — the merge is now routinely rescuing genes that one tool got wrong and the other got right. Out at the far end the tools diverge sharply: on *C. elegans* to *C. briggsae*, DNA alignment has nearly given up (Liftoff 0.41) while protein alignment is still informative (miniprot 0.65), and LiftOn 2.0, combining them, reaches 0.72 — its single largest margin over the best baseline. Human-to-mouse is the lone case where the DNA signal has degraded so far (Liftoff 0.50) that protein alone (miniprot 0.79) is the better single bet, and LiftOn, anchored partly to that weak DNA evidence, comes in just behind. The lesson isn't that combining always wins; it's that combining wins whenever *both* signals still carry information, and the window where that's true covers almost everything you'd actually want to annotate.

## What's new if you use LiftOn 2.0

If you already run LiftOn, the command line hasn't changed — but several defaults have, all in the direction of more complete, more accurate output, and every one of them is reversible if you need the old behavior.

By default, LiftOn 2.0 now lifts **every gene-like feature type** it finds in your reference, not just protein-coding genes; it applies the **best-of-outcome merge** that keeps a combined gene model only when it scores strictly higher than the plain DNA lift; and it aligns proteins with the memory-bounded **windowed aligner**. If you are reproducing the published LiftOn paper and need byte-for-byte the old results, three flags turn those back off individually — `--gene-only` (lift only protein-coding genes), `--legacy-merge` (always apply the merged model), and `--full-dp-align` (restore the original exact alignment for all but the largest giant genes, which stay memory-bounded) — and `--serial-aligners` restores the old sequential aligner scheduling.

In the other direction, several flags make a run faster or lighter *without changing the output at all* — this is the byte-identity guarantee at work. `--threads N` together with `--locus-pipeline` spreads the per-gene work across cores; `--stream` pipes one aligner's output into the next stage without a disk round-trip; and `--inmemory-liftoff` hands Liftoff's result off in memory. Because the test matrix pins all of these to the default output, you can reach for any of them purely to fit your time and memory budget, and trust that the annotation you get is exactly the one you'd have gotten without them. The windowed aligner and the concurrent aligners are already on by default, so on a typical run you get most of the speed and nearly all of the memory savings for free.

In practice the basic command is unchanged from before — point LiftOn at a reference annotation, a reference genome, and your new assembly:

`lifton -g reference.gff3 reference.fa target.fa -o lifton.gff3`

To reproduce the published paper exactly, add `--gene-only --legacy-merge --full-dp-align`. To go fast on a multi-core machine, add `--threads 8 --locus-pipeline` — and because that path is pinned to the default output, the annotation you get is byte-for-byte the one you'd have gotten on a single thread. A small, separate utility ships alongside: `gff3-validate`, the same checker I used to produce Figure 5, which you can point at any GFF3 file to get the error and warning counts directly.

## The bigger picture

When I wrote about the first LiftOn, I called it plumbing — not a flashy model, but the layer everything downstream stands on. Two years of watching people use it taught me that good plumbing isn't only about being *right*; it's about being *dependable* — finishing the job, fitting in the memory you actually have, not quietly dropping a quarter of the annotation. Most of LiftOn 2.0 is that kind of unglamorous work, and I've come to think it matters at least as much as the idea it serves. The most useful single thing I did this round wasn't an algorithm at all; it was the byte-identity test matrix that let me rebuild the engine — windowed alignment, concurrency, threading, streaming — with the confidence that the answer hadn't moved.

The other lesson is about restraint. A real part of this work was chasing *more* accuracy: smarter ways to pick reading frames, to rescue marginal genes, to fold still more of the protein evidence into the merge. I built and measured each idea, and most of them didn't pay — the gains were lost in the noise, or came with regressions that outweighed them. Reporting those honestly and *not* shipping them is as much a part of the work as the wins. A tool that only ever grows is a tool quietly accumulating risk; knowing when an idea is good enough to leave out is its own kind of progress.

LiftOn 2.0 is free and open source, and it still stands on Liftoff and miniprot — two tools each doing what it does best, with LiftOn refereeing between them. If you have a freshly assembled genome and an annotation you trust, point 2.0 at it: it will carry more of the map across, faster and on a fraction of the memory than before — and it will finish.
