## 1. Introduction

### 1.1 The annotation-transfer problem

A genome assembly without an annotation is a sequence without a map. Long-read sequencing and the T2T/pangenome era now produce finished assemblies far faster than experimental annotation can characterize them, which makes **annotation lift-over** — projecting a curated reference annotation $R_A$ from genome $R$ onto a new target genome $T$ to produce $T_A$ — a foundational step in modern genomics. Two families of evidence dominate lift-over, and each fails in a characteristic way:

- **DNA-level mapping** (Liftoff [2], driven by minimap2 [4]) aligns reference gene loci, with their exon structure, directly to the target DNA. It is fast and exact when $R$ and $T$ are close, but as the sequences diverge it accumulates substitutions and small indels that manifest as frameshifts, premature stops, and incorrect splice boundaries.
- **Protein-level mapping** (miniprot [3]) aligns reference *proteins* to the target with a splice-aware, frame-aware model. Protein sequence is conserved well past the point where DNA alignment degrades, so miniprot survives divergence — but its models are fragmentary and its identifiers do not correspond to reference transcript IDs.

LiftOn [1] resolves this by *combining* both: it obtains a DNA model (Liftoff) and a protein model (miniprot) per gene, aligns each candidate's translated protein back to the reference, and runs a protein-maximization CDS-chaining merge followed by an ORF-rescue pass.

### 1.2 What lifton2 is

lifton2 is a research re-implementation of the lift-over engine built to explore two questions LiftOn's architecture does not isolate. First, **can the vendored Liftoff dependency be replaced by a first-party engine without changing its output?** LiftOn ships a fork of Liftoff as a vendored subtree; lifton2 de-vendors the minimap2+Liftoff DNA-lift into first-party code and pins it **feature-identical to a freshly regenerated vendored-Liftoff golden** with a byte-exact regression gate (§2.1). Second, **how far does a refinement-first posture go?** Rather than a post-hoc merge of two finished models, lifton2 uses miniprot evidence to *rewrite the splice structure* of the genes the DNA lift already places, only ever adopting the rewrite when it scores strictly higher against the reference protein.

### 1.3 Contributions

This report documents lifton2's engine and validates it on 23 representative-chromosome lift-overs across a four-tier divergence ladder (§2.7), each tool re-scored by one tool-neutral parasail kernel (§2.8). The principal pieces are:

- A **de-vendored minimap2+Liftoff DNA-lift core** held feature-identical to vendored Liftoff by a byte-exact `--dna-only` regression gate under a pinned interpreter and `PYTHONHASHSEED=0` (§2.1).
- **Strategy-B miniprot-informed refinement** that replaces a transcript's exon/CDS structure with miniprot's splice-aware structure only when the swap scores strictly higher against the reference protein — a guaranteed per-transcript non-regression on the genes the DNA lift places (§2.2).
- A **post-lift miniprot-only rescue pass** (default-ON) that adds reference coding genes the DNA lift misses entirely, with a dedup guard so it only ever *adds* genes, never displaces a lifted one (§2.3).
- A **divergence-adaptive rescue floor** that lowers the rescue's protein-identity threshold from 0.5 toward 0.3 as DNA-lift recall drops, recovering divergent recall at **zero redundancy** (§2.3, §3.3).
- **Gene-like feature autodetection** that lifts every reference parent type bearing a transcript/exon hierarchy — pseudogenes, ncRNA/lnc_RNA genes — in an additive pass that leaves the coding output byte-identical, bringing lifton2's feature breadth up to LiftOn's (§2.5, §3.5).
- **Write-time GFF3 containment normalization** (and unique multi-copy identifiers) that makes lifton2 the only tool in the comparison to emit standards-clean GFF3, **without touching any coding sequence**, so protein identity is provably unchanged (§2.6, §3.6).
- Engine robustness and concurrency work — bounded giant-gene polish memory, a recursion-limit guard, concurrent aligners — that let lifton2 **complete whole genomes that crash standalone Liftoff and LiftOn v1.0.8** (§2.4, §3.7), all under a byte-exact gate that keeps the DNA core output-neutral.

## 2. Methods

### 2.1 The de-vendored DNA-lift core and its byte-exact contract

lifton2's lift core is a first-party port of the minimap2-driven Liftoff DNA-lift: it aligns reference gene loci to the target, resolves overlaps, and emits a lifted GFF3. The load-bearing correctness contract is that this de-vendoring **changes nothing**. A regression gate (`benchmarks/dna_equiv_ab.py --dna-only`) regenerates a vendored-Liftoff golden and compares lifton2's lift to it under a pinned `lifton_devel` interpreter with `PYTHONHASHSEED=0` and single-threaded minimap2 (to remove the only sources of run-to-run nondeterminism, repeat-copy ordering and hash seeding). The verdict is feature-equivalence of the primary lift: identical feature counts, coordinates, and CDS phases, with zero missing, extra, or coordinate-mismatched features. The gate is **green** on drosophila, mouse_to_rat, and rice, verified both before and after the engine work in this report landed — so every result below sits on a DNA core proven equal to the established Liftoff.

### 2.2 Strategy-B: miniprot-informed splice refinement

For each transcript the DNA lift places, lifton2 also has miniprot's protein-to-genome alignment of the same reference protein. Strategy-B (`miniprot_chain_override`) translates the DNA-lift CDS and the miniprot CDS, aligns each back to the reference protein with parasail [5], and replaces the DNA-lift exon/CDS structure with miniprot's splice structure **only when the latter's protein identity is strictly higher**. Because the swap is gated on a strict improvement against a fixed reference, it is a per-transcript non-regression on the recovered set: it can only raise, never lower, the emitted protein identity of a transcript the DNA lift already placed. This is the mechanism behind lifton2's accuracy lead on divergent pairs (§3.1) — divergence is exactly where the DNA model frameshifts and the protein model does not.

### 2.3 miniprot-only rescue and the divergence-adaptive floor

Strategy-B refines genes the DNA lift *places*; it does nothing for genes the DNA lift *misses*. On divergent pairs the DNA lift misses many. lifton2 therefore runs a **miniprot-only rescue** as a separate pass *after* the main lift: for each reference coding gene with no overlapping lifted locus, it builds a gene model from miniprot's alignment and emits it if its protein identity clears a floor. The pass carries a dedup guard keyed on reference gene id, so it only ever **adds** a gene at an empty locus — it never displaces or duplicates a lifted model (this is what keeps the rescue zero-redundant).

A fixed identity floor is a blunt instrument across the divergence range: 0.5 is appropriate on same-species pairs but discards genuine, lower-identity orthologs on very-distant pairs where 0.5-identity *is* the conserved core. lifton2 makes the floor **divergence-adaptive**: it estimates DNA-lift recall (the fraction of miniprot-found genes the DNA lift already placed) and lowers the floor from 0.5 toward 0.3 as that recall drops, leaving it at 0.5 on high-recall pairs where it is inert. Across the very-distant tier this recovers thousands of genuine coding genes at zero redundancy while leaving same/close-species output unchanged (§3.3).

Throughout this report, **lifton2 (refine)** is the engine with the rescue pass off — the pure DNA-lift + Strategy-B refinement — and **lifton2 (complete)** is the engine with the rescue pass and adaptive floor on (the default). Reporting both isolates the precision contribution (refine) from the recall contribution (complete).

### 2.4 Engine robustness and the output-neutral contract

Two robustness guards let lifton2 finish large genomes without changing output on normal ones. A giant-gene polish guard bounds the post-lift polish alignment: an unbounded full-DP parasail on a $\sim$100 kb single-exon interval requires $O(L^2)$ memory and exhausted RAM on plant genomes; lifton2 skips polishing an interval whose alignment matrix would exceed a size bound (default $2\times10^9$ cells), and guards a null-traceback crash mode. These are **byte-neutral on normal genes** — the `--polish` byte-exact gate is identical on drosophila/mouse_to_rat/rice with zero guard fires — and only affect true giants. Concurrent execution (overlapping the miniprot subprocess with the DNA lift) and the de-vendored core are likewise governed by the byte-exact gate, so the engine optimizations leave the lifted output bytes unchanged.

### 2.5 Gene-like feature lift

lifton2's DNA lift, like the Liftoff it de-vendors, lifts only the protein-coding `gene` hierarchy by default. lifton2 adds **gene-like feature autodetection**: a bounded scan classifies a reference parent type as gene-like iff at least one *top-level* instance carries a transcript/exon child — capturing pseudogenes, ncRNA/lnc_RNA genes, tRNA/rRNA, and structured mobile elements while excluding childless meta/regulatory types. The detected non-coding types are lifted in a **separate additive pass that runs after the coding lift completes**, into a disjoint set of reference-id keys. The coding-`gene` output is therefore byte-identical to the gene-only baseline (an `off ⊆ on` guarantee *by construction*), and the gene-like records are pure additions: this brings lifton2's feature breadth up to LiftOn's without perturbing the coding result (§3.5). The opt-out flag `--gene-only` restores the coding-only lift; the same byte-exact gate that pins the DNA core (§2.1) holds the coding lineage byte-neutral under the two-pass design.

### 2.6 GFF3 containment normalization

Lift-over chaining can introduce GFF3 the reference itself does not have: a frameshift-corrected CDS that extends a few base pairs past its exon (so the CDS falls outside the mRNA span), a rebuilt exon structure that reuses stale reference exon IDs (duplicate identifiers), or a multi-copy gene whose copies share one identifier. lifton2 runs a **write-time normalization** that, for each transcript, widens each exon to contain its CDS, sorts the exons, renumbers them only on a *real* ID collision, sets the transcript/gene span to the child envelope, and assigns a unique identifier per copy. Because the **CDS coordinates are never touched**, protein identity is provably unchanged — the normalization moves only exon/mRNA/gene boundary and identifier lines. It is a no-op on already-valid transcripts and is gated so the pre-normalization bytes can be reproduced. With it, lifton2 is the only tool in the comparison whose output validates clean against the [GFF3 specification](https://github.com/The-Sequence-Ontology/Specifications/blob/master/gff3.md) (§3.6).

### 2.7 Benchmark design

We evaluate six tools — **Liftoff** (the DNA baseline lifton2 refines), **LiftOn v1.0.8** (the published release), **LiftOn devel** (the development line), **lifton2 (refine)**, and **lifton2 (complete)**, with **miniprot** shown as the *internal protein evidence* lifton2 fuses (the splice structure Strategy-B adopts and the source of the rescue), not a rival lift-over tool — on **23 representative-chromosome lift-overs** organized as a four-tier divergence ladder: **same-species** (7 cells; *Arabidopsis*, honeybee, mouse, rice, and three human annotation sets), **close cross-species** (6; *Drosophila* D. mel→D. erecta, mouse→rat, human→chimp, *Arabidopsis*→*A. lyrata*, two yeasts), **distant cross-species** (6; *C. elegans*→*C. briggsae*, chicken→quail, *D. mel*→*D. pseudoobscura*, human→mouse, rice→sorghum, zebrafish→medaka), and **very-distant cross-species** (4; *Arabidopsis*→rice eudicot→monocot, *S. cerevisiae*→*S. pombe*, *Drosophila*→*Anopheles*, human→zebrafish). The tier of each cell is taken from the LiftOn benchmark registry. The subset cells lift a single representative chromosome per genome so that all six tools complete every cell; in the data used here, **all 23 cells are scored by all six tools with no crashes**, so every tier average is a true matched-cell mean (no tool is averaged over a different cell set than another — the crash-asymmetry confound that distorts naïve cross-tool averages does not arise here).

### 2.8 The tool-neutral evaluator

Every metric is computed by a single version-agnostic re-scorer (`benchmarks/compare/evaluator.py`) that re-aligns each tool's emitted CDS-translated protein back to the reference protein with LiftOn's parasail kernel — **no tool's self-reported numbers are used**. **Coding completeness** (recall) is the fraction of reference coding transcripts a tool recovers; **mean protein identity** is the mean, over the transcripts a tool recovers, of its emitted protein's identity to the reference protein; **feature-total completeness** is the fraction of *all* reference features (coding plus gene-like) a tool recovers, the axis on which the gene-like lift of §2.5 registers; and **validity** is the `gff3-validate` error count under one consistent validator applied to all six outputs. Mean protein identity is a **set-mean** — each tool is averaged over its *own* recovered set — so it must be read together with completeness: a higher mean identity over a smaller, easier recovered set is not a like-for-like win. We make this explicit throughout, and a per-transcript common-set (apples-to-apples) comparison is part of the whole-genome follow-up (§5).

## 3. Results

### 3.1 Accuracy: lifton2 refines the DNA baseline, the gain growing with divergence

Figure 1 and Table 2 give the set-mean protein identity by tier. The cleanest reading is against **Liftoff — the DNA baseline lifton2 extends, scored on the same metric over the same tool family**: lifton2 is more accurate on **every tier**, by a margin that is a fraction of a percent on the easy tiers ($+0.001$ same-species, $+0.016$ close) and **large on both divergent tiers** ($+0.214$ distant, $+0.212$ very-distant), where DNA-only frameshifts are most frequent and the protein evidence most often supplies a strictly better splice structure (§2.2; Figure 1B). Because Strategy-B adopts the miniprot structure only when it scores strictly higher than the DNA model against a fixed reference, this gain is a **guaranteed per-transcript non-regression** — lifton2 can only sit at or above Liftoff on every recovered transcript.

Against the dual-evidence **LiftOn** — the pipeline lifton2 re-implements — lifton2 **tracks closely but sits marginally behind on every tier** (same-species $-0.003$, close $-0.017$, distant $-0.026$, very-distant $-0.038$ vs LiftOn devel). A first-party engine thus captures most of the dual-evidence gain over the DNA baseline, but does not surpass the mature LiftOn merge on raw per-gene identity; lifton2's advantages over LiftOn lie elsewhere — feature breadth (§3.5), standards-clean output (§3.6), and whole-genome robustness (§3.7). miniprot is shown for context as the protein evidence lifton2 fuses, not a rival lift-over column.

### 3.2 Recall: a deliberate precision/recall tradeoff

The accuracy lead over the DNA baseline does not come for free, and reading the identity ladder without the recall axis would misrepresent it. lifton2's **refine** arm recovers, by construction, exactly the set of genes the DNA lift places — its coding completeness equals Liftoff's to within rounding on every tier (very-distant 0.031, vs Liftoff 0.031; distant 0.601 vs Liftoff 0.599). On divergent pairs that set is small: where the protein evidence (miniprot) recovers 0.687 of reference coding transcripts on the very-distant tier and 0.907 on the distant tier, lifton2 (refine) recovers 0.031 and 0.601. The miniprot-only rescue plus adaptive floor (the **complete** arm) closes part of this — very-distant completeness rises to **0.197** and distant to 0.637 (Figure 5A) — but lifton2 remains **far below the protein evidence** on recall ($-0.490$ on very-distant, $-0.271$ on distant). Figure 2 shows this per cell: lifton2's points sit high on identity and low on recall, the protein evidence's the reverse.

This is the honest summary, and it is the same shape LiftOn reports for its own merge at extreme divergence: **lifton2 is a high-precision refinement of the DNA-anchored core, not a recall-maximizing tool.** Its identity advantage over Liftoff is real and is measured over the genes it recovers; it does not recover as many genes as the protein evidence on divergent pairs. (The set-mean identity of the **complete** arm on the very-distant tier, 0.482, is *lower* than the **refine** arm's 0.503 — because the rescue deliberately adds genuine but lower-identity genes, trading mean identity for recall; §3.3.)

### 3.3 The adaptive floor recovers divergent recall at zero redundancy

The recall the **complete** arm adds over **refine** is entirely the miniprot-only rescue, and the divergence-adaptive floor is what makes it substantial on the hardest pairs. Figure 3 shows the per-cell completeness gained going from refine to complete: it is essentially zero on same/close cells (the floor stays at 0.5 and the DNA lift already recovers everything) and large on the divergent cells where the adaptive floor lowers toward 0.3 — **_Arabidopsis_→rice +0.245, cerevisiae→pombe +0.231, _Drosophila_→anopheles +0.139, zebrafish→medaka +0.095, _D. mel_→_D. pseudoobscura_ +0.069**. Because the rescue only ever adds a gene at a locus the DNA lift left empty, with a reference-id dedup guard, these gains carry **zero redundant gene models** — they are genuine recoveries, not duplicate or paralog-confused emissions. This is the recall lever that distinguishes lifton2 (complete) from a pure DNA-lift refinement; it is also the source of the mean-identity dip noted in §3.2, since the recovered genes are real but lower-identity.

### 3.4 The refinement margin, by tier and by cell

Figure 4 isolates lifton2's set-mean identity margin — primarily over **Liftoff**, the DNA baseline it refines, and secondarily over LiftOn devel, the strongest dual-evidence baseline. Against Liftoff the per-tier margin is positive on every tier and large on the divergent ones ($+0.001$, $+0.016$, $+0.214$, $+0.212$); against LiftOn devel it is slightly negative on every tier ($-0.003$, $-0.017$, $-0.026$, $-0.038$), confirming that lifton2 tracks but does not surpass the mature dual-evidence merge on raw identity. The per-cell view shows the divergent cross-species pairs (e.g. *Arabidopsis*→rice, cerevisiae→pombe, human→mouse) driving the lead over the DNA baseline. This is the precision side of the tradeoff stated precisely: on the genes it recovers, lifton2's miniprot-informed splice refinement yields higher protein identity than the DNA lift exactly where DNA divergence makes the difference, and the effect is absent where it cannot help.

### 3.5 Feature breadth: lifting the gene-like features the DNA lift drops

A first-party DNA lift that stops at the protein-coding `gene` hierarchy (§2.5) inherits a narrower genome map than the reference provides. lifton2's additive gene-like pass closes that gap: it lifts the pseudogenes, ncRNA/lnc_RNA genes, and tRNA/rRNA the coding-only baseline drops — for example, hundreds of pseudogenes on a representative *Arabidopsis* chromosome and over a thousand additional features on a representative human chromosome — **with zero coding genes lost** and the coding output byte-identical to the gene-only baseline (§2.5). (A companion 34-dataset validation sweep adds **+6,870 gene-like features** in total.) Measured as **feature-total completeness** (the fraction of *all* reference features recovered, Figure 5B), lifton2 rises **above the coding-only Liftoff on every tier** (from +0.022 on same-species to +0.0004 very-distant) and **matches or exceeds LiftOn's feature breadth on the same- and close-species tiers**; on the divergent tiers its feature-total sits below LiftOn's, but that gap is *recall* — the §3.2 coding-recall difference, where the rescue-limited coding count dominates the feature total — not gene-like breadth. Because the gene-like records are pure additions at disjoint reference IDs, they do not affect the coding accuracy or recall of §3.1–§3.2.

### 3.6 Output validity: the only tool that emits standards-clean GFF3

Re-running every tool's output through one GFF3-specification validator (§2.8), lifton2 is the **only tool in the comparison with zero validation errors**, on every cell. The DNA-only Liftoff, the protein evidence miniprot, and both LiftOn releases each carry tens of specification errors per cell — duplicate exon identifiers, out-of-bounds CDS, and related boundary problems that lift-over chaining introduces and that some references themselves carry (Figure 6). lifton2's write-time containment normalization (§2.6) repairs exactly these, and because it **never touches a CDS coordinate**, the protein-identity numbers of §3.1 are provably unchanged by it: it moves only exon/mRNA/gene boundary and identifier lines. On a representative *Drosophila* chromosome, for example, the validator reports 85 errors for Liftoff, 50 for miniprot, and ~51 for the LiftOn releases, against **0 for lifton2**.

### 3.7 Whole-genome robustness

Although this report's quantitative results are on representative chromosomes (§2.7), the engine work that lets lifton2 run them at scale is most visible on whole genomes. Running its own de-vendored Liftoff under a recursion-limit guard and a bounded giant-gene polish (§2.4), **lifton2 produced scorable output on every whole genome tested**, including the extreme-divergence pairs where standalone Liftoff aborts (full *Arabidopsis*, and the eudicot→monocot *Arabidopsis*→rice) and where LiftOn v1.0.8 aborts mid-run (full rice and *Arabidopsis*) (Figure 7). The very-distant *Arabidopsis*→rice whole genome also reproduces the divergence pattern of §3.1 at full scale — lifton2's mean protein identity there exceeds both the DNA baseline and the prior LiftOn — but the precise whole-genome accuracy table is part of the planned follow-up (§5). Robustness — completing the run at all — is the precondition for every accuracy number, and it is where a carefully re-engineered first-party engine pays off.

## 4. Discussion

lifton2 demonstrates three things. First, the lift core can be **de-vendored** from Liftoff into first-party code without changing its output — the byte-exact gate makes this a verified equality, not an approximation, and it frees the engine to evolve independently of the vendored fork. Second, a **refinement-first** posture — rewrite splice structure when (and only when) the protein evidence strictly improves it — produces gene models that exceed the DNA baseline on every tier, by a margin that grows with divergence, on the genes the DNA lift recovers. Third, the same engine that refines also **broadens and cleans** the output: it lifts the gene-like features the coding-only baseline drops (§3.5), it is the only tool in the comparison that emits standards-clean GFF3 without touching a coding base (§3.6), and it completes whole genomes that crash the tools it descends from (§3.7) — the engineering that makes a research re-implementation usable rather than merely correct.

The cost is recall. Because lifton2 anchors on the DNA lift and refines it, its recovered set on divergent pairs is the DNA lift's set; the miniprot-only rescue with the adaptive floor recovers a substantial, zero-redundant slice of the divergent gap, but the protein-only miniprot baseline still recovers far more genes there. Which posture is preferable depends on the use case: an analysis that needs every recoverable ortholog at divergent range should weight recall (and miniprot, or LiftOn's full merge, is the better tool); an analysis that needs the highest-fidelity models for the genes it does recover is served by lifton2's refinement. The two are not interchangeable, and the value of reporting both lifton2 arms is that it makes the posture explicit rather than hiding it inside a single aggregate.

## 5. Limitations

- **Recall on divergent pairs.** This is the central tradeoff, not an incidental caveat. lifton2's recall is the DNA lift's recall plus a bounded rescue; on very-distant pairs it recovers 0.197 of reference coding transcripts where miniprot recovers 0.687. lifton2 is precision-focused by design.
- **Identity vs LiftOn.** lifton2 exceeds the DNA baseline (Liftoff) on every tier but **does not surpass the mature dual-evidence LiftOn merge** on raw per-gene identity — it trails LiftOn devel by 0.003–0.038 mean identity across the tiers (§3.1, §3.4). Its advantages over LiftOn are first-party de-vendoring, feature breadth, output validity, and robustness, not a higher identity score.
- **Set-mean identity.** The mean-identity numbers are each tool over its own recovered set. They are reported alongside recall throughout, but a per-transcript **common-set (apples-to-apples)** comparison — restricting to the transcripts a pair of tools both recover — is a sharper instrument and is part of the whole-genome follow-up below.
- **Representative-chromosome scope.** The 23 cells lift a single representative chromosome per genome. This is what makes the matched-cell, crash-free comparison clean, but it is not a whole-genome accuracy claim. The **whole-genome accuracy benchmark is in progress** and will be reported as a second part, with the same verification protocol.
- **Evaluator determinism.** On datasets carrying duplicate CDS IDs in the reference, the evaluator's `gffutils` de-duplication can shift per-transcript pairing; this adds noise to per-transcript A/B comparisons but does not affect per-cell corpus means or completeness, which is what is reported here.
- **Paralog copies.** Like the underlying DNA lift, lifton2 is conservative about emitting extra paralog copies under `-copies`; recovering them is future work.

## 6. Availability and reproducibility

lifton2 is open-source at [github.com/Kuanhao-Chao/lifton2](https://github.com/Kuanhao-Chao/lifton2). It is a Python engine over minimap2, miniprot, and the parasail [5] kernel. The benchmark harness, the tool-neutral evaluator (`benchmarks/compare/evaluator.py`), the per-cell scores, and the verified-numbers source for every figure and table in this report (`benchmarks/compare/report_verified_numbers.json`) are archived in the repository, as is the byte-exact DNA-equivalence gate (`benchmarks/dna_equiv_ab.py`). Runs are pinned to a fixed interpreter with `PYTHONHASHSEED=0`; the DNA-only gate is re-run single-threaded so that its feature-equivalence verdict against the vendored-Liftoff golden is deterministic. All numbers in this report are re-derived from the raw per-cell re-scores by the verification script that emits `report_verified_numbers.json`, and the figures are generated from that file alone.
