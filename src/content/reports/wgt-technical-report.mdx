## 1. Introduction

### 1.1 The Wheeler-graph recognition problem

The Burrows–Wheeler Transform and the FM-index let a string be indexed once and then queried in time that scales with the query rather than the text — the machinery underneath read aligners such as Bowtie and BWA. **Wheeler graphs** [2] generalize that idea from a string to a labeled graph. An edge-labeled directed graph $G=(V,E)$ is a Wheeler graph if there is a total order $\pi$ on $V$ such that the following hold for all nodes $u,v\in V$ and all pairs of edges $e_1=(u_1,v_1,a_1)$, $e_2=(u_2,v_2,a_2)$:

$$
\textbf{(A1)}\quad \mathrm{indeg}(u)=0 \,\wedge\, \mathrm{indeg}(v)>0 \;\Rightarrow\; \pi(u) < \pi(v),
$$
$$
\textbf{(A2)}\quad a_1 < a_2 \;\Rightarrow\; \pi(v_1) < \pi(v_2),
$$
$$
\textbf{(A3)}\quad a_1 = a_2 \,\wedge\, \pi(u_1) < \pi(u_2) \;\Rightarrow\; \pi(v_1) \leq \pi(v_2).
$$

When such an order exists, the FM-index query machinery carries over to the graph, which is why Wheeler graphs underpin graph-based pangenome indexes. The difficulty is that deciding whether a valid $\pi$ exists is **NP-complete**, and hard even to approximate [3]. Every decision procedure must therefore search the space of orderings; they differ only in how cleverly they prune it.

### 1.2 WGT and the heuristic-plus-solver approach

The Wheeler Graph Toolkit (WGT) [1] makes recognition practical with its recognizer, **Wheelie**, which runs a three-step pipeline (§2.1). The decisive idea is a **renaming heuristic**: because a node's position in any valid order is strongly constrained by the labels on its incoming edges, the heuristic groups nodes by incoming label and repeatedly sorts and relabels them, propagating constraints until they converge. Convergence either exposes a contradiction (the graph is not a Wheeler graph) or narrows each node to a small window of possible positions, which is handed to one of two interchangeable backends: a **permutation** search that enumerates the remaining orderings with pruning, or an **SMT** backend that encodes the residual constraints for the Z3 solver [4]. WGT also ships generators (De Bruijn graphs, tries, reverse-deterministic graphs, and random graphs), a visualizer, and a benchmark harness that includes a reference implementation of the only previously proposed recognition algorithm, due to Gibney and Thankachan [3].

### 1.3 What motivated this study

WGT's published recognition framework [1] — the three axioms and the renaming heuristic — is unchanged; what changed between v1.0.0 and the current version is the *solver* the heuristic feeds (a lazy/CEGAR encoding, §2.6, §4) and the addition of an independent ground truth to validate against. Three goals motivated that work, since a recognizer is an oracle other research builds on and must be **demonstrably** correct:

1. **An independent ground truth.** Verdicts were trusted on the strength of the algorithm and internal checks; a recognizer is far stronger if its verdicts are cross-validated against a separate, obviously-correct implementation, because anything that rests on "this is a Wheeler graph" inherits that verdict. Establishing this is the first thing the study does.
2. **A reference that can refute.** The bundled exponential reference was a timing baseline rather than a decision procedure — it did not check the axioms against the graph's edges — so it could not serve as an independent second opinion. Making it an honest decision procedure gives a second, independent check on small graphs.
3. **Two capabilities to add.** The full-range SMT path emitted a quadratic number of constraints and became the bottleneck on dense, few-label graphs; and there was no way to turn a "not a Wheeler graph" verdict into a constructive fix.

### 1.4 Contributions

This report studies the current WGT recognizer along three deliberately separated axes (§2.8), using the last stable GitHub release (v1.0.0, July 2023) as the speed/memory/scale baseline, with all intermediate development versions omitted. The principal results are:

- A **lazy / CEGAR SMT solver** (§2.6, §4) — now the default — that generates the within-label axiom on demand from counterexamples rather than materializing it in full. It moves the default-path size ceiling to at least 32,768 nodes, about **554× faster and 133× lighter** at matched size than the prior release, verified sound and never regressing a verdict (§3.2). This is the headline result.
- **Independently validated soundness**: the recognizer is checked against an **independent brute-force oracle** (§2.2) and a **differential-testing harness** (§2.3) across ~18,000 graphs — zero false-accepts and zero false-rejects on every backend, every emitted order re-validated, and 0 disagreements among three independent decision procedures (§3.1, §2.4).
- A **sparser full-range encoding** (§2.6, §4) that reduces the A2 cross-label constraint from $O(E^2)$ to $O(E+L)$ and the A3 within-label constraint from $O(E_\ell^2)$ to $O(D_\ell^2+E_\ell)$, ≈14× faster encoding setup and ≈2× faster full-range time on real graphs, with no per-type regression (§3.3).
- An **honest exponential reference** (§2.5) rebuilt into a real decision procedure (used here as auxiliary ground truth), and a **string-preserving repair module** (§2.7) that unfolds any non-Wheeler DAG into a Wheeler graph preserving its path-string set exactly (§3.4, §5).
- A **scalability characterization** (§2.8, §3.2) of every path's size limit under a fixed budget, and an in-depth account of **why the new solver is faster** (§4).

## 2. Methods

### 2.1 The recognizer pipeline

Wheelie parses a DOT graph into an internal digraph and runs the three-step pipeline of Table 1 (`recognizer/src/`). Steps 1–2 are the heuristic; step 3 is the solver, which is interchangeable between the SMT and permutation backends and, in full-range mode (`-f`), always uses SMT with the step-2 range narrowing disabled.

**Table 1.** The Wheelie recognition pipeline.

| Step | Stage | Purpose |
|---|---|---|
| 0 | Parse | Read the `strict digraph` DOT, build the internal digraph from `A -> B [label=x]` edges. |
| 1 | `relabel_initialization` | Impose the axiom-implied initial constraints (in-degree-0 prefix; per-label head grouping). If they are already violated, reject — the graph is not a Wheeler graph. |
| 2 | `innodelist_sort_relabel` | The renaming heuristic: iteratively sort and relabel nodes by incoming-edge structure, narrowing each node to an order window $[\mathrm{lb},\mathrm{ub}]$. Under `-f`, the range-narrowing branches are disabled. |
| 3 | Solve | Search within the windows: **SMT** (`solve_smt`, Z3 `QF_IDL`; the default, now lazy/CEGAR) or **permutation** (`permutation_start`, pruned enumeration with `WG_checker`). |
| 4 | Verdict | Exit code $1$ = Wheeler, $-1$ = not a Wheeler graph, $0$ = undecided. Benchmark mode (`-b`) prints `verdict, nodes, cpu, path`. |

### 2.2 The brute-force oracle

Ground truth is established by a deliberately trivial procedure (`verify/brute_oracle.py`), written independently of the recognizer so that the two cannot share a bug. For a graph with $n$ nodes it enumerates all $n!$ node orderings; for each, it checks A1–A3 directly against the edge list and accepts iff some ordering satisfies all three. Its cost,

$$
T_\text{oracle}(n,e) = O(n!\cdot e^2),
$$

confines it to $n \leq 9$, but on those graphs it is an incontrovertible decision: a completely separate implementation computing the same predicate from first principles. (One subtlety the oracle itself had to get right: a single node with two differently-labeled self-loops violates A2, so only the empty graph, not every one-node graph, is a trivial accept.)

### 2.3 Differential-testing harness and corpora

The harness (`verify/difftest.py`, `verify/edgecases.py`, `verify/check_order.py`) draws graphs from many families, runs each through both the oracle and every recognizer backend, flags any verdict that differs, and re-validates every accepted order against the axioms directly. Table 2 summarizes the corpora used across this report.

**Table 2.** Evaluation corpora.

| Corpus | Size | Role |
|---|---:|---|
| Random + reject-heavy, oracle-checked (both binaries) | ~18,000 | Validation: 0 false-accepts / 0 false-rejects on every backend (§3.1). |
| Oracle-decidable cross-check (recognizer + exponential ref) | 2,447 | Verdict agreement vs the oracle — 0 disagreements (§3.1). |
| Edge cases (self-loops, cycles, parallel edges, …) | 16 | Hand-checked structural cases, every backend (§3.1). |
| Real DNA/AA De Bruijn / reverse-deterministic | 900 | Full-range `-f` timing, current vs the v1.0.0 baseline (§3.3). |
| Synthetic `complete` / `dnfa` families | ladder | Scalability and the default-path ceiling (§3.2, §4). |
| Random DAGs | 820 | Repair (§3.4). |

Each family is Wheeler or non-Wheeler by construction or by oracle anchoring, so every rung doubles as a correctness test as well as a timing or scaling measurement.

### 2.4 Sound-by-construction accept paths

So that a "Wheeler" verdict is never issued without a satisfied check, the current recognizer's accept paths are gated and validated, and every one is exercised against the oracle (§2.2, §3.1):

- **Every accept is gated by a final axiom check.** No backend returns "Wheeler" on the strength of its internal state alone: the SMT, permutation, and lazy paths each pass their candidate order through the same `WG_checker`, which verifies A1–A3 on the concrete order before the verdict is emitted. The permutation backend rejects explicitly when a non-exhaustive search completes with no valid order, and the step-2 propagation fast path (which can settle every node to a singleton window without invoking a solver) is re-validated by the same checker before it accepts.
- **Degenerate and large inputs are guarded.** The solver-dispatch heuristic uses saturating arithmetic, and the SMT path guards degenerate inputs — an empty or isolated-node graph that would otherwise produce an empty `z3::distinct` — so recognition is robust on the boundary cases the corpora include.

These are design properties of the current recognizer, and §3.1 reports the result of checking them against the oracle: zero disagreements over ~18,000 graphs, with every emitted order independently re-validated.

### 2.5 An honest exponential decision procedure

The bundled reference implementation of the Gibney–Thankachan algorithm [3] was, as originally written, a timing baseline: its checker received only scalar counts rather than the graph's edges, so it could not test the axioms and never emitted a definite non-Wheeler verdict. To make it a trustworthy independent second opinion, its decision core was reimplemented (`benchmark/exponential_recognizer/`) to enumerate node orderings and verify A1–A3 against the real edges — mirroring the oracle of §2.2 — under an $O(n!\cdot e^2)$ work budget, returning a clean three-way verdict:

$$
\text{verdict} = \begin{cases}
\text{Wheeler} & \text{some ordering satisfies A1--A3},\\
\text{not a Wheeler graph} & \text{no ordering does, within budget exhausted},\\
\text{undecided} & \text{work budget exceeded before exhaustion}.
\end{cases}
$$

It is now usable as a decision procedure on small graphs and as a second independent check against the recognizer, not merely a stopwatch — it is the third decision procedure in the agreement check of §3.1.

### 2.6 A sparser full-range encoding and a lazy default solver

In full-range mode the recognizer hands the entire ordering space to Z3: one integer order variable per node, a global `distinct`, and the axiom constraints, solved as a quantifier-free integer-difference-logic (`QF_IDL`) problem [4]. The cost is dominated by the *number of axiom constraints*, and a naive encoding writes them pairwise. Two changes reduce that cost; both are explained in depth, with the algorithm and its soundness, in §4.

**A sparser full-range encoding.** The cross-label monotonicity axiom (A2) is rewritten from an $O(E^2)$ all-pairs form to an $O(E+L)$ form that brackets each label's heads and chains the brackets across labels; the within-label axiom (A3) is rewritten from $O(E_\ell^2)$ to an $O(D_\ell^2+E_\ell)$ endpoint block, applied under a shape-aware guard that keeps the verified pairwise loop wherever the block would not help. Both forms are **equisatisfiable** with the pairwise originals, so correctness is unaffected; only the encoding size changes (§4.5).

**A lazy default solver.** On the default path, where the step-2 heuristic supplies tight order windows, the current version replaces the up-front A3 encoding with **lazy, counterexample-guided generation** (`recognizer/src/lazy_solve.cpp`, the default; `-s smt` selects the prior, non-lazy encoding). The solver asserts the window and `distinct` constraints with no A3, asks Z3 for a model, re-checks A3 against that concrete order, adds only the constraints the model actually violates, and re-solves — keeping Z3's learned clauses across rounds. It is sound by construction and falls back to the full solve on a budget. The full mechanism, the per-round cost, and the soundness argument are in §4.

### 2.7 String-preserving graph repair

Repair (`repair/wheelerize.py`) turns a non-Wheeler **DAG** into a Wheeler graph by unfolding it into the **trie of its path-strings**. A trie is always a Wheeler graph — order its nodes by the co-lexicographic rank of the string spelled along each incoming path — so the construction always succeeds for a DAG, and it preserves the **set of strings** the graph spells, exactly. Each repair self-verifies three independent ways before it is returned: the recognizer accepts the output, the oracle accepts it when it is small enough, and the set of path-strings is confirmed identical to the input's. The node-count ratio (output / input) can fall below one — when distinct input nodes spell the same string they are *merged* — which is a reduction in nodes, not a loss of represented strings; ratios above one are genuine node splitting. This first version is correct but not minimal; minimal node-splitting is left to future work, and cyclic graphs (where a string-preserving repair is not always possible) are out of scope.

### 2.8 Benchmark design and the three-axis methodology

The cardinal rule of this evaluation is that three questions are reported **separately** and never merged into a single "X% better":

- **Correctness** — does the verdict match the independent oracle? Measured as oracle agreement on the oracle-decidable and reject-heavy corpora, and re-validation of every accepted order.
- **Capability** — can a verdict be produced at all? Measured as the largest graph a path decides within its budget, and the fraction of a corpus the rebuilt exponential reference decides, independent of speed.
- **Performance** — how fast, and how large, when correct? Measured as full-range setup/solve time, the per-graph timing distribution, peak memory, and the largest graph each backend decides under a fixed budget, with v1.0.0 as the baseline.

A faster wrong answer is not progress, an undecided result is not a refutation, and additional memory is a cost to be stated, not hidden. The full-range timing uses string labels (matching the biological corpora), the median of replicate runs, and the `-b` CPU-time column, which is robust to machine load; timeouts are recorded as timeouts, never as a verdict.

**The scalability sweep** (`benchmark/limit_test/`, `benchmark/lazy_cegar/`) climbs a geometric size ladder per (backend, family) under a **600 s** wall-clock budget, takes the median of $R=3$ replicate graphs per rung, and bisects the largest size still decided. Limits are **typed**: *threshold* (timeout-bounded — the backend would continue with more time), *capped* (a hard capability ceiling — the exponential reference's $n!$ budget, or Z3 returning `unknown`), or *unreached* (no wall hit up to the ladder top). The two families are `complete` (dense, all-distinct endpoints — the symmetric worst case) and `dnfa` (random $d$-NFA-style), both Wheeler by construction so each rung is also a correctness check.

**Environment and reproducibility.** Comparisons are run on a single Linux host with the recognizer built from source (g++ 17, Z3 linked statically). The recognizer, oracle, harnesses, and all source data and figures are versioned in the repository; every number in this report is recomputed from the committed data by `benchmark/report_figs/verify_report_numbers.py` (an all-PASS check), with figure data under `benchmark/report_figs/data/`, `benchmark/lazy_cegar/`, and `benchmark/limit_test/results/`.

### 3.1 Axis I — Validation against an independent oracle

**Result:** the recognizer agrees with the independent oracle on every decided graph — **zero false-accepts, zero false-rejects** across ~18,000 oracle-checked graphs and on every backend — and every order it emits is re-validated against the axioms directly.

Correctness is established the way it should be: against a separate, obviously-correct decision procedure, not by trusting the recognizer's own internal state. The brute-force oracle (§2.2) enumerates all $n!$ orderings and checks the three axioms directly; the differential harness (§2.3) runs every graph through both the oracle and every recognizer backend and flags any verdict that differs; and the witness gate (`check_order.py`) re-validates every accepted order against A1–A3 independently. Across the random and reject-heavy corpora (~18,000 oracle-checked graphs counting both binaries) the current recognizer records **zero false-accepts and zero false-rejects on every backend** — default, lazy, permutation, full-range, and their exhaustive and full-range-lazy variants — and every Wheeler order it emits passes the independent re-validation (Table 4).

### 3.2 Axis II — Capability and the size ceiling

**Result:** the default-path size ceiling — the path a user gets with no flags — moves to **≥32,768** nodes on both synthetic families, from the prior release's **2,816** `complete` / **2,176** `dnfa`, and at a matched 2,816 nodes the current solver is **≈554× faster and ≈133× lighter**, by generating the within-label axiom on demand (just **3,835 of 402,722,292** A3 constraints at $n=32{,}768$; mechanism in §4).

Under a 600 s budget the prior release's default solver (vanilla Z3 plus the renaming heuristic) decides `complete` graphs of **2,816** nodes and `dnfa` graphs of **2,176**, its limit a timeout — at $n=4{,}096$ it does not finish in ten minutes, and given a full hour reaches only 4,608/3,584. The current version's default is the new lazy/CEGAR solver, and it decides **at least 32,768** nodes on both families — the largest size on our test ladder, not a wall the solver hit — in under a minute (Table 5, Figure 3). That is a **more-than-tenfold** increase in the size of worst-case graph the recognizer decides, on the path almost everyone runs, and it comes entirely from the solving algorithm, not faster hardware. Every rung is Wheeler by construction and was decided correctly: the sweep recorded **zero correctness failures** end to end.

## 4. How the new algorithm achieves its speedup

The capability and speed results of §3.2 are not a constant-factor tune-up; they come from changing *what the solver is asked to prove*. This section explains the mechanism in depth, from the bottleneck to the lazy/CEGAR loop to why it is simultaneously faster and sound (Figure 5).

### 4.1 The bottleneck: a dense, quadratic within-label axiom

On the symmetric worst-case families, profiling Z3's own statistics tells a surprising story: the solver does almost **no search** — tens to hundreds of SAT conflicts even at $n=1{,}000$ — while its memory allocations explode roughly **12,000-fold** over the same range. The wall is therefore not combinatorial search; it is the cost of *building and propagating* an $O(n^2)$ difference-logic formula that grows with the graph. The culprit is the within-label axiom A3: writing it pairwise instantiates one implication for every pair of same-label edges, $\sum_\ell \binom{E_\ell}{2}$ in total — at $n=32{,}768$ that universe is **402,722,292** constraints. The vanilla `-s smt` encoding (the prior default) materializes all of them up front, so Z3 spends its time constructing and propagating a formula whose answer turns out to need almost none of it.

### 4.2 The heuristic's brackets nearly determine the order

The renaming heuristic (`innodelist_sort_relabel`, Table 1) is what makes the bottleneck removable. By iteratively sorting and relabeling nodes by their incoming-edge structure, it narrows each node to a tight order window $[\mathrm{lb},\mathrm{ub}]$ (`_node_ranges`) — and on these families it nearly determines the order outright, leaving the solver only a small residue to settle. That is the leverage the lazy solver exploits: when the candidate order is already almost fixed, almost none of the quadratic A3 universe can actually be violated, so almost none of it needs to exist.

### 4.3 Lazy, counterexample-guided generation (the CEGAR loop)

A3 is a **chain** constraint: within a label group, the heads must be monotone non-decreasing in tail order. It is $O(E_\ell^2)$ only *statically*, because the tail order is unknown a priori — but given a concrete candidate order the tail order is *known*, and any violation is then forbidden by just the $O(E)$ constraints on consecutive elements. The lazy default solver (`recognizer/src/lazy_solve.cpp`) turns this into a counterexample-guided loop:

```
assert the heuristic's brackets + all-different (+ sparse A2 under -f);  // NO A3
loop:
    if Z3 finds no model      -> NOT a Wheeler graph     (the partial set is already unsat)
    if Z3 returns 'unknown'   -> fall back to the full solve
    otherwise, read the model's order and, per label group:
        sort the group's edges by (tail position, head position)   // O(E log E)
        scan consecutive pairs; a head that drops is an A3 violation
    if no violation in any group -> run WG_checker on the order;
                                    accept iff it passes            // the soundness gate
    else add only the violated consecutive-pair A3 lemmas (deduplicated) and re-solve
```

Three properties make this fast. (i) **The recheck is cheap:** detecting A3 violations is a sort and a linear scan per group, $O(E\log E)$, not a quadratic comparison. (ii) **Only binding constraints are ever built:** each round adds just the handful of lemmas the current model actually violates, so Z3 never sees the quadratic formula. (iii) **Learning is preserved:** lemmas are added *monotonically* with `s.add` and the solver is never reset (no `push`/`pop`), so Z3's conflict-driven learned clauses from the cheap early rounds carry forward and prune the later ones. On these families the loop converges in **≈5–8 rounds** at every size, and the number of A3 constraints it ever materializes stays in the low thousands while the full universe grows into the hundreds of millions (Figure 5A): deciding a 32,768-node `complete` graph builds just **3,835 of 402,722,292** pairs — the dense formula is **~99.999% redundant**.

### 4.4 Why it is both faster and sound

Speed without soundness would be worthless, so the loop is sound by construction. At every `s.check()` the constraint set is a **subset** of the full encoding's — the same brackets, `distinct`, and (under `-f`) sparse A2, plus a subset of the *identical* canonical pairwise A3 lemmas — so its model set is a **superset** of the full encoding's. That gives both directions:

- **No false reject.** The loop returns "not a Wheeler graph" only when Z3 reports `unsat`; unsatisfiability of a constraint subset implies unsatisfiability of the full set, so a rejection is always genuine.
- **No false accept.** The loop returns "Wheeler" only when a model has **zero** A3 violations **and** that order passes the same final `WG_checker` (A1–A3 on the concrete order) the other backends use. An accepted order is therefore independently re-validated, exactly as in §2.4 and §3.1.

When the instance is genuinely dense — so that lazy generation would have to materialize most of the universe anyway — the loop detects this (it aborts once it has built more than 60% of the pair universe, or after a round budget) and **falls back to the full Z3 solve**, so a verdict can never regress relative to the prior encoding. `-s smt` still selects that prior, non-lazy encoding for direct comparison.

### 4.5 The sparser full-range encoding

The `-f` path keeps a single up-front encoding (the heuristic's narrowing is deliberately disabled there), so it cannot generate A3 lazily; instead it makes the up-front encoding *sparser* (`recognizer/src/smt.cpp`), which is the mechanism behind the §3.3 timings and the atom counts in Figure 5B–C:

- **Cross-label A2, $O(E^2)\to O(E+L)$.** Instead of ordering every head of a smaller label before every head of a larger one (all pairs), the encoding introduces per-label head-boundary variables $(\#\mathrm{lo}_\ell,\#\mathrm{hi}_\ell)$ that bracket that label's heads into a window, and chains the windows in label order ($\#\mathrm{hi}_{\ell-1} < \#\mathrm{lo}_\ell$). One bracket per edge plus one link per label reproduces the all-pairs relation in $O(E+L)$, and the two forms are equisatisfiable.
- **Within-label A3, $O(E_\ell^2)\to O(D_\ell^2+E_\ell)$.** With $D_\ell$ the smaller of the distinct-tail and distinct-head counts, the block brackets the opposite endpoint per distinct key node $(\#\mathrm{mn}_k,\#\mathrm{mx}_k)$ and relates the $D_\ell$ keys pairwise, trading $E_\ell^2$ pairwise atoms for $2D_\ell$ auxiliaries plus $D_\ell(D_\ell-1)$ key relations.
- **A shape-aware guard.** The block's auxiliaries can enlarge Z3's search on dense instances even when the atom count drops, so it is applied to a label only when $2D_\ell < E_\ell \wedge D_\ell(D_\ell-1)+2E_\ell < E_\ell(E_\ell-1)$ — i.e. only when the endpoints are genuinely sparse *and* the block emits fewer atoms; otherwise the verified pairwise loop is used. Because both forms are equisatisfiable, the guard changes only *which* encoding is built, never the verdict — which is why §3.3 shows speedups with no regression.

Measured directly, this makes the full-range formula a median of **13.4× smaller** in atoms (an analytical count validated to the atom against Z3's own `s.assertions().size()`), genuinely sub-quadratic ($\propto E^{1.6}$) where the within-label block fires and a large constant-factor reduction elsewhere (Figure 5B–C). The decisive caveat from §3.3 holds: atoms are not wall time — a smaller formula makes *building* the problem nearly free, but Z3's solve is the bottleneck and falls only ≈2×.

## 5. Usefulness and applications

### 5.1 Why recognition is the gatekeeper for graph indexing

A Wheeler graph is exactly the class of edge-labeled graph that BWT/FM-index machinery can index and query in time scaling with the query rather than the data [2] — the property that makes such an index compact and fast. For a pangenome, where the "text" is a graph of many genomes rather than one string, that is the difference between an index that answers pattern queries directly on the graph and one that cannot. Recognition is the **gatekeeper**: only a graph that *is* a Wheeler graph can be given this representation, so before any graph is committed to an FM-index pipeline, it must be decided — and, given the NP-completeness of the problem, decided by a procedure that is both fast enough and trustworthy. That is the niche this recognizer fills.

### 5.2 A validated recognizer as a trustworthy building block

Because the recognizer is now validated against an independent oracle and sound on every backend (§3.1), its verdicts can be relied on as ground truth by the tools built on top of it: a downstream pipeline that assumes "this graph is a Wheeler graph" no longer inherits an unchecked assumption. Just as valuable, the verification apparatus itself — the brute-force oracle, the differential harness, and the order-witness checker — is reusable infrastructure: any new backend, encoding, or construction can be differential-tested against the same oracle before it is trusted, and the discipline (build a slow, obviously-correct reference, then differential-test the fast implementation across many generated inputs) transfers directly to other combinatorial recognizers.

### 5.3 Real-time on the graphs biology actually produces

The hour-long, multi-thousand-node runs of §3.2 are a property of adversarial synthetic graphs. On real data the recognizer is effectively instantaneous: all **500** graphs built from Ensembl gene alignments were decided in **under a second** (§3.4), the largest only ~2,400 nodes. For practitioners this means recognition is not the bottleneck in a graph-indexing workflow; the cost lives upstream, in how the graph is constructed and how large it grows. The most actionable finding is that **the construction determines the verdict**: De Bruijn $k$-mer graphs and tries were Wheeler on every real alignment we tried, while the reverse-deterministic automaton almost never was — so a practitioner who needs an indexable graph should reach for the constructions that produce one.

### 5.4 Turning a negative verdict into an indexable graph

Recognition answers a yes/no question; the repair module (§2.7, §3.4) makes "no" actionable. Any non-Wheeler DAG is unfolded into a Wheeler graph that **spells exactly the same set of path-strings** (316 of 316, zero failures), so a graph that cannot be indexed as a Wheeler graph can be replaced by one that can, representing the same sequences. That is precisely the primitive an index-construction pipeline needs: when a chosen construction yields a non-Wheeler graph — as the reverse-deterministic automaton usually does — repair converts it into an indexable form without changing the strings it encodes. (The first version is correct but not minimal, and is DAG-only; both are clear directions for future work.)

### 5.5 Headroom as pangenome graphs grow

Finally, the capability gain is not a one-time convenience but headroom for the future. Pangenome graphs grow as more genomes are added, and the default-path ceiling moving from a few thousand to **≥32,768** worst-case nodes — with near-flat time and memory in between (§3.2) — means recognition keeps pace on the path users actually run as inputs scale, rather than becoming the step that fails first. Where the prior release would time out, the lazy default returns a verified answer in seconds.

## 6. Discussion and conclusions

**The recognizer is validated, far faster on the default path, and verifiable — and we state precisely what it costs.** That is the whole result, and the three-axis discipline (§2.8) is what makes it defensible: a real per-graph speedup (§3.3) reported alongside, not instead of, the facts that the full-range size ceiling does not move (§3.2) and that the block costs memory on tractable graphs (§3.3).

**Validation is the load-bearing result.** The most consequential change in this pass is not the faster encoding or the new repair capability — it is that the recognizer's verdicts are now checked against an independent, obviously-correct oracle and a second decision procedure, with every accepted order re-validated. A recognizer is an oracle other research builds on; establishing 0 disagreements over ~18k graphs (§3.1) is the prerequisite that makes the speed and capability results worth reporting at all.

**Trades, stated plainly.** Three costs are volunteered, not extracted. (i) The full-range `-f` size ceiling is **unchanged** — both encodings stop at $n=832$ where Z3 returns `unknown` (§3.2); the lazy win is confined to the default path, exactly where the heuristic supplies the brackets the lazy loop exploits (§4.2). (ii) The sparse `-f` encoding costs **~1.8× peak RAM** on the low-$D/E$ graphs where the within-label block fires (§3.3) — a space-for-time trade, not a uniform improvement. (iii) Recognition **remains NP-complete**, hard already at a two-letter alphabet even for DAGs; the lazy default wins by exploiting the heuristic's brackets, not by beating the hardness.

**The heuristic is the engine.** The scalability separation (§3.2, §4.2) is a quantitative restatement of WGT's core design insight: the renaming heuristic, by narrowing each node's window before the solver runs, is what lets the default backend scale orders of magnitude past the no-heuristic full-range path. The improvements here sharpen the periphery — a lazy default solver, a leaner full-range encoding, a trustworthy reference, repair — around a method whose central idea is unchanged.

**Recognition is not the bottleneck in practice — so the lever is upstream.** Every Wheeler graph built from a real gene alignment was recognized in under a second (§3.4, §5.3), and parallelism does not move the synthetic ceiling because the cost is a single `QF_IDL` solve (§3.2). The practical lever is therefore which *construction* one chooses, which determines whether the graph is Wheeler at all; and when it is not, repair (§5.4) turns that "no" into an indexable graph spelling the same strings. For realistic pangenome-scale inputs the open problem is graph *construction and size*, not recognition speed — and the recognizer now has the headroom (§5.5) to keep pace.

In short: the WGT recognizer is now **validated** against an independent oracle, **≈554× faster and ≈133× lighter** on the default path with a **>10×** higher size ceiling, **≈14× cheaper** to build and **1.3–1.9×** faster on the full-range path with one honest memory trade, **real-time** on biological graphs, and paired with a **string-preserving repair** that makes a negative verdict actionable — every claim traceable to a figure or table and recomputed by `verify_report_numbers.py`.

## 7. Limitations

The study and tool are presented with the following honest caveats, the first two of which bound the headline result and are stated in §3 and §6 as well:

1. **The full-range ceiling is unchanged.** The sparse encoding lowers per-graph time but not the size at which the full-range path can decide: both the new and prior encodings stop at $n=832$ where Z3 returns `unknown` (§3.2). Raising that ceiling is a solver-encoding problem left open; the default-path lazy win does not extend to `-f`, where the heuristic brackets are absent (§4.2).
2. **A memory cost on tractable graphs.** The sparse encoding's auxiliary variables raise full-range peak memory by roughly 1.8× on the low-$D/E$ graphs where the within-label block fires (§3.3); it is a space-for-time trade, not a uniform improvement. (On the default path the lazy solver is the opposite — far lighter.)
3. **Recognition remains NP-complete.** None of this changes the worst case. The dense, all-distinct-endpoint `complete` family is where every backend is slowest, and where the full-range block is correctly disabled. The lazy default wins by exploiting the heuristic's brackets (§4), not by beating the hardness.
4. **Oracle reach.** The brute-force oracle is exact but $O(n!\cdot e^2)$, so ground truth is direct only for $n\leq9$. Validation on larger graphs rests on cross-backend agreement and construction-by-design families, not on the oracle.
5. **Repair is not minimal and is DAG-only.** The trie unfolding always succeeds for a DAG and preserves the string set, but the result is not a minimum-node Wheeler graph; cyclic graphs, where a string-preserving repair need not exist, are out of scope.
6. **The per-type speedup covers paired jobs only.** The §3.3 breakdown uses the jobs with a comparable finite time from *both* versions; the excluded rows are the largest amino-acid instances where the dense encoding exceeds the timeout, so the table is a conservative per-type summary rather than an optimistic subset.
