## 1. Introduction

### 1.1 The Wheeler-graph recognition problem

The Burrows‚ÄìWheeler Transform and the FM-index let a string be indexed once and then queried in time that scales with the query rather than the text ‚Äî the machinery underneath read aligners such as Bowtie and BWA. **Wheeler graphs** [2] generalize that idea from a string to a labeled graph. An edge-labeled directed graph $G=(V,E)$ is a Wheeler graph if there is a total order $\pi$ on $V$ such that the following hold for all nodes $u,v\in V$ and all pairs of edges $e_1=(u_1,v_1,a_1)$, $e_2=(u_2,v_2,a_2)$:

$$
\textbf{(A1)}\quad \mathrm{indeg}(u)=0 \,\wedge\, \mathrm{indeg}(v)>0 \;\Rightarrow\; \pi(u) < \pi(v),
$$
$$
\textbf{(A2)}\quad a_1 < a_2 \;\Rightarrow\; \pi(v_1) < \pi(v_2),
$$
$$
\textbf{(A3)}\quad a_1 = a_2 \,\wedge\, \pi(u_1) < \pi(u_2) \;\Rightarrow\; \pi(v_1) \leq \pi(v_2).
$$

When such an order exists, the FM-index query machinery carries over to the graph, which is why Wheeler graphs underpin graph-based pangenome indexes. The difficulty is that deciding whether a valid $\pi$ exists is **NP-complete**, and hard even to approximate [3]. Every decision procedure must therefore search the space of orderings; they differ only in how cleverly they prune it.

### 1.2 WGT and the heuristic-plus-solver approach

The Wheeler Graph Toolkit (WGT) [1] makes recognition practical with its recognizer, **Wheelie**, which runs a three-step pipeline (¬ß2.1). The decisive idea is a **renaming heuristic**: because a node's position in any valid order is strongly constrained by the labels on its incoming edges, the heuristic groups nodes by incoming label and repeatedly sorts and relabels them, propagating constraints until they converge. Convergence either exposes a contradiction (the graph is not a Wheeler graph) or narrows each node to a small window of possible positions, which is handed to one of two interchangeable backends: a **permutation** search that enumerates the remaining orderings with pruning, or an **SMT** backend that encodes the residual constraints for the Z3 solver [4]. WGT also ships generators (De Bruijn graphs, tries, reverse-deterministic graphs, and random graphs), a visualizer, and a benchmark harness that includes a reference implementation of the only previously proposed recognition algorithm, due to Gibney and Thankachan [3].

### 1.3 What motivated this study

WGT's published recognition framework [1] ‚Äî the three axioms and the renaming heuristic ‚Äî is unchanged; what changed between v1.0.0 and the current version is the *solver* the heuristic feeds (a lazy/CEGAR encoding, ¬ß2.6) and the soundness of the accept paths. Three concerns motivated that work, since a recognizer is an oracle other research builds on and must be **demonstrably** correct:

1. **No independent ground truth.** The recognizer's verdicts were trusted on the strength of the algorithm and internal checks, but not cross-validated against a separate, obviously-correct implementation. A recognizer that occasionally accepts a non-Wheeler graph silently invalidates everything downstream of that verdict, so this is the first thing that must be established, not assumed.
2. **A reference that could not refute.** The bundled exponential reference was a timing baseline, not a decision procedure: it did not check the axioms against the graph's edges and so could not return a definite "not a Wheeler graph," which makes it unusable as an independent second opinion.
3. **Two missing capabilities.** The full-range SMT path emitted a quadratic number of constraints and became the bottleneck on dense, few-label graphs; and there was no way to turn a "not a Wheeler graph" verdict into a constructive fix.

### 1.4 Contributions

This report compares the WGT recognizer's last stable GitHub release (v1.0.0, July 2023) against the current version along three deliberately separated axes (¬ß2.8), with all intermediate development versions omitted. The principal differences are:

- A **lazy / CEGAR SMT solver** (¬ß2.6) ‚Äî now the default ‚Äî that generates the within-label axiom on demand from counterexamples rather than materializing it in full. It moves the default-path size ceiling from v1.0.0's ~2,800 nodes to at least 32,768, about 540√ó faster and 137√ó lighter at matched size, verified sound and never regressing a verdict (¬ß3.5, ¬ß3.9). This is the headline difference.
- **Soundness fixes**: v1.0.0's permutation backends false-accept non-Wheeler graphs and its full-range path can crash on degenerate input; the current version is correct on every backend, validated against an **independent brute-force oracle** (¬ß2.2) and a **differential-testing harness** (¬ß2.3) ‚Äî zero false-accepts and zero false-rejects (¬ß3.1, ¬ß2.4).
- A **sparser full-range encoding** (¬ß2.6) that reduces the A2 cross-label constraint from $O(E^2)$ to $O(E+L)$ and the A3 within-label constraint from $O(E_\ell^2)$ to $O(D_\ell^2+E_\ell)$, ‚âà14√ó faster encoding setup and ‚âà2√ó faster full-range time on real graphs, with no per-type regression (¬ß3.3, ¬ß3.8).
- An **honest exponential reference** (¬ß2.5) rebuilt into a real decision procedure (used here as auxiliary ground truth), and a **string-preserving repair module** (¬ß2.7) that unfolds any non-Wheeler DAG into a Wheeler graph preserving its path-string set exactly (¬ß3.7).
- A **scalability characterization** (¬ß2.8, ¬ß3.5) of every path's size limit under a fixed budget, v1.0.0 versus current.

## 2. Methods

### 2.1 The recognizer pipeline

Wheelie parses a DOT graph into an internal digraph and runs the three-step pipeline of Table 1 (`recognizer/src/`). Steps 1‚Äì2 are the heuristic; step 3 is the solver, which is interchangeable between the SMT and permutation backends and, in full-range mode (`-f`), always uses SMT with the step-2 range narrowing disabled.

**Table 1.** The Wheelie recognition pipeline.

| Step | Stage | Purpose |
|---|---|---|
| 0 | Parse | Read the `strict digraph` DOT, build the internal digraph from `A -> B [label=x]` edges. |
| 1 | `relabel_initialization` | Impose the axiom-implied initial constraints (in-degree-0 prefix; per-label head grouping). If they are already violated, reject ‚Äî the graph is not a Wheeler graph. |
| 2 | `innodelist_sort_relabel` | The renaming heuristic: iteratively sort and relabel nodes by incoming-edge structure, narrowing each node to an order window $[\mathrm{lb},\mathrm{ub}]$. Under `-f`, the range-narrowing branches are disabled. |
| 3 | Solve | Search within the windows: **SMT** (`solve_smt`, Z3 `QF_IDL`; the default) or **permutation** (`permutation_start`, pruned enumeration with `WG_checker`). |
| 4 | Verdict | Exit code $1$ = Wheeler, $-1$ = not a Wheeler graph, $0$ = undecided. Benchmark mode (`-b`) prints `verdict, nodes, cpu, path`. |

### 2.2 The brute-force oracle

Ground truth is established by a deliberately trivial procedure (`verify/brute_oracle.py`), written independently of the recognizer so that the two cannot share a bug. For a graph with $n$ nodes it enumerates all $n!$ node orderings; for each, it checks A1‚ÄìA3 directly against the edge list and accepts iff some ordering satisfies all three. Its cost,

$$
T_\text{oracle}(n,e) = O(n!\cdot e^2),
$$

confines it to $n \leq 9$, but on those graphs it is an incontrovertible decision: a completely separate implementation computing the same predicate from first principles. (One subtlety the oracle itself had to get right: a single node with two differently-labeled self-loops violates A2, so only the empty graph, not every one-node graph, is a trivial accept.)

### 2.3 Differential-testing harness and corpora

The harness (`verify/difftest.py`, `verify/edgecases.py`) draws graphs from many families, runs each through both the oracle and every recognizer backend, and flags any verdict that differs. Table 2 summarizes the corpora used across this report.

**Table 2.** Evaluation corpora.

| Corpus | Size | Role |
|---|---:|---|
| Reject-heavy random (non-Wheeler), oracle-checked | 3,839 + 2,626 (dense) | Stress the accept paths: count false-accepts, v1.0.0 vs current (¬ß3.1). |
| Random + positives, both binaries vs oracle | ~18,000 | Verdict agreement / zero false-accepts & false-rejects (¬ß3.1). |
| Edge cases (self-loops, cycles, parallel edges, ‚Ä¶) | 16 | Hand-checked structural cases, both binaries (¬ß3.1). |
| Real DNA/AA De Bruijn / reverse-deterministic | 900 | Full-range `-f` timing, v1.0.0 vs current (¬ß3.3, ¬ß3.8). |
| Synthetic `complete` / `dnfa` families | ladder | Scalability limits (¬ß3.5). |
| Random DAGs | 820 | Repair (¬ß3.7). |

Each family is Wheeler or non-Wheeler by construction or by oracle anchoring, so every rung doubles as a correctness test as well as a timing or scaling measurement.

### 2.4 Soundness fixes (v1.0.0 ‚Üí current)

Differential testing against the oracle surfaced several defects in **v1.0.0**'s *accept* paths ‚Äî the places where a verdict of "Wheeler" is issued ‚Äî each fixed in the current version so that an accept is never issued without a satisfied check (these are the source of v1.0.0's false-accepts in ¬ß3.1):

- **Reject on permutation exhaustion.** The permutation backend (`permutation_start`) issued its verdict only on the accept branch; on a fully pruned, exhausted search it returned without a decision and the caller defaulted to "Wheeler." It now rejects explicitly when a non-exhaustive search completes with no valid order.
- **Re-check the propagation fast path.** When step-2 propagation alone settled every node to a singleton window, the result was accepted without the final `WG_checker` the SMT and permutation paths run. The fast path now verifies A1‚ÄìA3 before accepting.
- **Overflow and degeneracy guards.** An integer overflow in the solver-dispatch heuristic (a product of per-window factorials) was made saturating; and the SMT path now guards degenerate inputs ‚Äî an empty or isolated-node graph that produced an empty `z3::distinct` ‚Äî that previously aborted.

These are corrective, not algorithmic: they change a verdict only on inputs the un-hardened code would have decided incorrectly or crashed on, and they leave every already-correct verdict unchanged. The exponential reference rewrite (¬ß2.5) also corrected the oracle's own one-node shortcut noted in ¬ß2.2.

### 2.5 An honest exponential decision procedure

The bundled reference implementation of the Gibney‚ÄìThankachan algorithm [3] was, as originally written, a timing baseline: its checker received only scalar counts rather than the graph's edges, so it could not test the axioms and never emitted a definite non-Wheeler verdict. To make it a trustworthy independent second opinion, its decision core was reimplemented (`benchmark/exponential_recognizer/`) to enumerate node orderings and verify A1‚ÄìA3 against the real edges ‚Äî mirroring the oracle of ¬ß2.2 ‚Äî under an $O(n!\cdot e^2)$ work budget, returning a clean three-way verdict:

$$
\text{verdict} = \begin{cases}
\text{Wheeler} & \text{some ordering satisfies A1--A3},\\
\text{not a Wheeler graph} & \text{no ordering does, within budget exhausted},\\
\text{undecided} & \text{work budget exceeded before exhaustion}.
\end{cases}
$$

It is now usable as a decision procedure on small graphs and as a second independent check against the recognizer, not merely a stopwatch.

### 2.6 A sparser full-range SMT encoding

In full-range mode the recognizer hands the entire ordering space to Z3: one integer order variable per node, a global `distinct`, and the axiom constraints, solved as a quantifier-free integer-difference-logic (`QF_IDL`) problem [4]. The cost is dominated by the *number of axiom constraints*, and the original encoding wrote them pairwise.

**Cross-label monotonicity (A2).** Writing A2 as an all-pairs constraint over edges costs $O(E^2)$. The sparse form introduces, per label, head-boundary variables ($\#\mathrm{lo}_\ell,\#\mathrm{hi}_\ell$) that bracket the order positions of that label's heads, and chains the brackets across labels in label order. This expresses the same "smaller label ‚áí earlier head" relation in

$$
O(E^2) \;\longrightarrow\; O(E+L),
$$

where $L$ is the number of distinct labels.

**Within-label ordering (A3).** For a single label $\ell$ with $E_\ell$ edges, the pairwise A3 constraint costs $O(E_\ell^2)$. The sparse form encodes A3 as an endpoint block: with $D_\ell = \min(\#\text{distinct tails},\,\#\text{distinct heads})$ for that label, per-endpoint min/max position variables ($\#\mathrm{mn},\#\mathrm{mx}$) replace the all-pairs comparison, giving

$$
O(E_\ell^2) \;\longrightarrow\; O(D_\ell^2 + E_\ell).
$$

Both forms are **equisatisfiable** with the pairwise originals ‚Äî they accept and reject exactly the same graphs ‚Äî so correctness is unaffected; only the encoding size changes.

**A shape-aware guard.** The block form trades pairwise difference atoms for $2D_\ell$ auxiliary integer variables, and those auxiliaries enlarge Z3's search. Gating the block on atom count alone (fewer atoms) is insufficient: on dense, few-label instances the auxiliaries can make the *solve* slower even when the encoding is smaller. The block is therefore applied to label $\ell$ only when

$$
2D_\ell < E_\ell \quad\wedge\quad D_\ell(D_\ell-1)+2E_\ell < E_\ell(E_\ell-1),
$$

i.e. only when the label's endpoints are genuinely sparse (the first clause; the De Bruijn regime, $D_\ell/E_\ell \approx 0.25$) *and* the block emits fewer atoms (the second). Otherwise the verified pairwise loop is used. Because the two encodings are equisatisfiable, the guard changes only *which* encoding is used, never the verdict.

**Lazy generation of the within-label axiom (the default solver).** The encoding changes above shrink the full-range formula but do not move its size ceiling (¬ß3.6). The decisive change is on the *default* path, where the step-2 heuristic supplies tight order windows: there the current version replaces the up-front A3 encoding with **lazy, counterexample-guided generation** (`recognizer/src/lazy_solve.cpp`, the default; `-s smt` selects v1.0.0's vanilla encoding). A3 is a *chain* constraint ‚Äî within a label group, heads are monotone non-decreasing in tail order ‚Äî and is $O(E_\ell^2)$ only because the tail order is unknown a priori. So the solver asserts the window and `distinct` constraints with **no A3**, asks Z3 for a model, re-checks A3 against that concrete order in $O(E\log E)$, adds only the $O(E)$ consecutive-pair constraints the model actually violates, and re-solves ‚Äî monotonically, so Z3 keeps its learned clauses across rounds (a lazy-SMT / CEGAR loop). Z3 never materializes the quadratic formula; it sees only the constraints that bind, which the heuristic's windows make a tiny fraction of the whole (¬ß3.9). The loop is **sound by construction**: it accepts only a model that passes the same final `WG_checker` the other backends use (with the emitted order independently re-validated), it rejects only when the partial constraint set is already unsatisfiable, and on a budget it falls back to the full Z3 solve ‚Äî so a verdict can never regress relative to v1.0.0's. This is the headline of ¬ß3.5 and ¬ß3.9.

### 2.7 String-preserving graph repair

Repair (`repair/wheelerize.py`) turns a non-Wheeler **DAG** into a Wheeler graph by unfolding it into the **trie of its path-strings**. A trie is always a Wheeler graph ‚Äî order its nodes by the co-lexicographic rank of the string spelled along each incoming path ‚Äî so the construction always succeeds for a DAG, and it preserves the **set of strings** the graph spells, exactly. Each repair self-verifies three independent ways before it is returned: the recognizer accepts the output, the oracle accepts it when it is small enough, and the set of path-strings is confirmed identical to the input's. The node-count ratio (output / input) can fall below one ‚Äî when distinct input nodes spell the same string they are *merged* ‚Äî which is a reduction in nodes, not a loss of represented strings; ratios above one are genuine node splitting. This first version is correct but not minimal; minimal node-splitting is left to future work, and cyclic graphs (where a string-preserving repair is not always possible) are out of scope.

### 2.8 Benchmark design and the three-axis methodology

The cardinal rule of this evaluation is that three questions are reported **separately** and never merged into a single "X% better":

- **Correctness** ‚Äî does the verdict match the independent oracle? Measured as false-accepts on the reject-heavy corpus and verdict agreement on the oracle-decidable and synthetic corpora.
- **Capability** ‚Äî can a verdict be produced at all? Measured as the fraction of a corpus a procedure decides within its budget (the rebuilt exponential reference), independent of speed.
- **Performance** ‚Äî how fast, and how large, when correct? Measured as full-range setup/solve time, the per-graph timing distribution, peak memory, and the largest graph each backend decides under a fixed budget.

A faster wrong answer is not progress, an undecided result is not a refutation, and additional memory is a cost to be stated, not hidden. The full-range timing uses string labels (matching the biological corpora), the median of replicate runs, and the `-b` CPU-time column, which is robust to machine load; timeouts are recorded as timeouts, never as a verdict.

**The scalability sweep** (`benchmark/limit_test/`) climbs a geometric size ladder per (backend, family) under a **600 s** wall-clock budget, takes the median of $R=3$ replicate graphs per rung, and bisects the largest size still decided. Limits are **typed**: *threshold* (timeout-bounded ‚Äî the backend would continue with more time), *capped* (a hard capability ceiling ‚Äî the exponential reference's $n!$ budget, or Z3 returning `unknown`), or *unreached* (no wall hit up to the ladder top). The two families are `complete` (dense, all-distinct endpoints ‚Äî the symmetric worst case) and `dnfa` (random $d$-NFA-style), both Wheeler by construction so each rung is also a correctness check.

**Environment and reproducibility.** Comparisons are run on a single Linux host with the recognizer built from source (g++ 17, Z3 linked statically). The recognizer, oracle, harnesses, and all source data and figures are versioned in the repository (`benchmark/REPORT.md` is the consolidated source for every number here; figure data lives under `benchmark/report_figs/data/` and `benchmark/limit_test/results/`).

## 3. Results

### 3.1 Correctness: agreement with an independent oracle

Against the brute-force oracle, the current recognizer is sound on every graph both can decide; the v1.0.0 release is not. On a reject-heavy corpus of **3,839 non-Wheeler graphs** ‚Äî generated to exercise the accept paths and oracle-checked at $n\leq7$ ‚Äî v1.0.0's permutation backend declares **770** of them Wheeler, and its exhaustive permutation mode the same 770, while its default-SMT and full-range paths are correct (Table 3). A second, denser corpus (2,626 non-Wheeler graphs with self-loops and parallel edges) widens the gap to **1,147** v1.0.0 false-accepts. The current version produces **zero** false-accepts on every backend, on both corpora.

### 3.2 Capability: graphs the release version cannot decide

Beyond correctness, the current version decides graphs that v1.0.0 cannot at any practical budget. On the synthetic worst-case families (Wheeler graphs built to be hard), v1.0.0's default solver reaches **2,816** `complete` / **2,176** `dnfa` nodes under a 600 s budget and then times out ‚Äî at $n=4{,}096$ it does not finish in ten minutes, and given an hour reaches only 4,608/3,584. The current version's lazy default decides **at least 32,768** nodes in under a minute (¬ß3.5, ¬ß3.9). So an entire range of graphs ‚Äî every worst-case instance between roughly 4,000 and 32,768 nodes ‚Äî is decidable now and was not before, independent of how long one is willing to wait. This capability gain comes entirely from the new solving algorithm, not from faster hardware; the full mechanism and numbers are in ¬ß3.9.

As a verification aside, the toolkit's exponential reference implementation ‚Äî used here only as an auxiliary ground truth, not as a recognizer ‚Äî was also rebuilt into an honest axiom-checking decision procedure: on the oracle-decidable subset (931 graphs of at most nine nodes) it now returns a verdict on all 931 (the original implementation exceeded its budget on 159, and never produced a non-Wheeler verdict at all), including seven non-Wheeler graphs built from real human gene orthologs (BTBD17, FAM53A, LCP1, TRAM1, TRPC1) it could not previously classify.

### 3.3 Performance: the encoding measured in atoms, then in time

**The result, then the evidence.** The new full-range (`-f`) SMT encoding is faster, validated to the atom, and honestly bounded:

- **A median ‚âà13√ó smaller formula.** The encoding emits about **13.4√ó fewer** SMT assertions (per-type 3.8‚Äì15.6√ó; up to ~300√ó on the largest many-label graphs) ‚Äî genuinely sub-quadratic ($\propto E^{1.57}$) where the within-label block fires, a large constant factor elsewhere ‚Äî a count *validated exactly* against Z3's own `s.assertions().size()` from an instrumented build (below).
- **Nearly-free construction, ~2√ó end-to-end.** Building the problem becomes **‚âà14√ó cheaper** on the headline graph, and total full-range time falls by **1.3√ó‚Äì1.9√ó** across the real biological corpora (by type), **with zero regressions** ‚Äî the current encoding is ‚â• v1.0.0 on every graph measured (¬ß3.8).
- **Complementary sparsifications.** The cross-group A2 form carries the many-label amino-acid regime (RevDet AA 1.82√ó, De Bruijn AA 1.87√ó); the within-group A3 block carries the few-label DNA regime (¬ß3.8).
- **Honestly bounded.** A smaller formula speeds the solver but does **not move the size ceiling** ‚Äî identical old and new, set by Z3 returning `unknown` at $n=832$ (¬ß3.6) ‚Äî and the block costs ~2√ó peak RAM only when it fires (¬ß3.4). The graph that *scales* is the default recognizer (¬ß3.5), which never builds this formula at all.

The rest of this section, and ¬ß3.4‚Äì¬ß3.8, are the evidence for each point.

Before any wall-clock number, the sparsification can be measured *directly* ‚Äî as the size of the SMT problem it builds. `atom_count.py` computes, analytically from each graph's label and endpoint structure, exactly how many `s.add(...)` assertions each generation emits under `-f`, mirroring the encoder line for line. The count is **exact**, not a model: on a calibration set it equals Z3's own `s.assertions().size()` from an instrumented build to the atom (the only residual is a shared `n+1`-assertion range/`distinct` baseline identical across generations). Fitting the counts over all 900 biological graphs against the edge count $E$ gives clean power laws (Fig. 2): v1.0.0's encoding is **$\propto E^{2.00}$ on every graph type** ‚Äî the dense all-pairs baseline ‚Äî while the current encoding is genuinely **sub-quadratic only where the within-label block fires** (few-label De Bruijn DNA, $\propto E^{1.57}$, a 24√ó atom reduction at the largest graph). On twenty-letter amino-acid alignments the block stays off and the gain is instead a large **constant factor** from the cross-group sparsification (‚âà17√ó), with the exponent near $E^{1.8}$ ‚Äî an honest distinction the fit makes visible.

The decisive caveat is that **atoms are not wall time**. The formula shrinks by a median of about thirteen-fold (and up to several-hundred-fold on the largest many-label graphs), and the *encoding setup* shrinks in step (‚âà14√ó on the headline graph, below). But the **total** full-range time falls only ‚âà2√ó, because Z3's *solve* is the bottleneck and its cost is not proportional to the atom count ‚Äî a smaller formula helps the solver but does not shrink the NP-hard search by the same factor. The sparsification's first-order effect is to make building the problem nearly free; the solve speedup is a real but second-order benefit.

### 3.4 Memory: an honest space-for-time trade

On the *full-range* path the current encoding makes one honest space-for-time trade, confined to a single regime. **Where the within-label block fires** ‚Äî low-$D/E$ graphs like *DOCK4* and *TRPC1* at $k=5$ ($D/E\approx0.27$) ‚Äî the current encoding pays for the block's auxiliary integer variables with about **1.8√ó the peak RAM** (662‚Üí1,210 MB and 670‚Üí1,289 MB): the genuine cost of the ‚âà14√ó setup speedup. **Where the block is off** ‚Äî higher-$D/E$ graphs such as the $k=4$ graphs ($D/E$ high enough that the guard keeps the block off) ‚Äî there are no such aux variables and peak memory is **essentially equal** to v1.0.0 (‚âà58 MB either way). This is the opposite of the *default*-path memory story, where the new lazy solver uses **far less** memory than v1.0.0 (¬ß3.9) ‚Äî there the win is in both time and space.

### 3.5 Scalability: how large a graph each version decides

The starkest difference is on the default path ‚Äî the one a user gets with no flags. ### 3.6 The full-range ceiling is a solver limit, the same in both versions

The full-range (`-f`) path's size limit deserves a precise reading, because it bears on what ¬ß3.3 does and does not claim. Both v1.0.0's dense `-f` encoding and the current sparse one stop at **832** nodes on both synthetic families ‚Äî and not by timeout but because Z3 returns `unknown` (undecided) on the `QF_IDL` problem at that size. That ceiling is a property of the solver on this encoding, the **same** for both versions. The sparser encoding therefore **lowers the time curve, not the size wall**: on the real biological corpora the current `-f` encoding runs well below v1.0.0's (¬ß3.3, ¬ß3.8) and finishes graphs v1.0.0 times out on, but on the dense synthetic families neither sparsification has much to remove and both stop at the same wall. (At large sizes v1.0.0's dense $O(E^2)$ `-f` encoding is also far heavier and can exhaust memory at lower $n$ than the current sparse form ‚Äî another reason the current `-f` path is preferable even though the asymptotic wall is shared.) The unflagged default path, where the lazy solver applies, is where the size wall actually moves (¬ß3.5, ¬ß3.9).

### 3.9 The headline: lazy A3 generation moves the default ceiling

The default-path ceiling of ¬ß3.5 ‚Äî v1.0.0's 2,816/2,176 nodes ‚Äî is not a hardware limit or a search explosion. Profiling Z3's own statistics on these symmetric worst cases shows it solves with **near-zero search** (tens to hundreds of SAT conflicts even at $n=1{,}000$) while its memory allocations explode roughly **12,000-fold** over the same range. The wall is the cost of building and propagating an $O(n^2)$ difference-logic formula that grows with the graph ‚Äî specifically the within-label A3 axiom, which v1.0.0 (and any vanilla encoding) materializes in full as one implication per pair of same-label edges.

The current version's default solver removes that wall by *not building the formula*. A3 ‚Äî within a label group, heads are monotone non-decreasing in tail order ‚Äî is a **chain** constraint, and it is $O(n^2)$ only *statically*, because the tail order is unknown a priori. Given a concrete candidate order the tail order is *known*, so a violation is forbidden by just $O(E)$ consecutive-pair constraints. So the solver runs a counterexample-guided loop (lazy SMT / CEGAR): assert the heuristic's bracket and all-different constraints but **no A3**, ask Z3 for a model, re-check A3 in $O(E\log E)$, add only the few consecutive-pair constraints the model actually violates, and re-solve ‚Äî monotonically, so Z3 keeps its learned clauses across rounds. Z3 never materializes the $O(n^2)$ formula; it only ever sees the handful of A3 constraints that bind. Because the renaming heuristic's brackets already nearly determine the order, that handful is tiny: deciding a 32,768-node `complete` graph builds just **3,835 of its 402,722,292** A3 pairs ‚Äî the full encoding is ~99.99% redundant.

The effect on the default path is dramatic (Table 9, Fig. 8). Where v1.0.0's vanilla solver needs **488 s and 7.7 GB** to decide a 2,816-node `complete` graph and then times out at 4,096, the current lazy default decides 2,816 in **0.9 s and 56 MB** and climbs to **‚â•32,768 nodes in under a minute** ‚Äî roughly **540√ó the speed at 1/137th the memory**, and a >10√ó higher size ceiling bounded by our test ladder, not by the solver. It is verified sound by the same oracle differential as every other result (zero disagreements over ~13,000 random and constructed graphs and every edge case, each emitted order independently re-validated), and whenever it cannot decide within budget it falls back to the full Z3 solve, so a verdict can never regress. `-s smt` still selects the original vanilla encoding, which is exactly v1.0.0's default.

Two contrasts sharpen the result. First, *time alone is no substitute for the algorithm*: given a full hour rather than 600 s, v1.0.0's vanilla default reaches only **4,608** `complete` / **3,584** `dnfa` nodes ‚Äî a 1.64√ó return for 6√ó the budget (the sub-linear scaling expected when solver cost grows steeply with $n$), still an order of magnitude below what the lazy default does in under a minute. Second, *parallelism is no substitute either*: at scale the recognizer's wall time is almost entirely a single Z3 `QF_IDL` solve, and a prototype with Z3's parallel mode showed no measurable gain on these families ‚Äî "parallelize the recognizer" means "parallelize one `QF_IDL` query," which it does not.

Two honest caveats keep the broader picture intact. The worst-case asymptotics are unchanged ‚Äî recognition is NP-complete already at a two-letter alphabet, even for DAGs ‚Äî so no encoding makes the *general* problem polynomial; the lazy solver wins by exploiting the heuristic's brackets, not by beating the hardness. And that reliance on the brackets is exactly why the gain is confined to the default path: in the full-range `-f` regime, where the heuristic is deliberately bypassed and the brackets are absent, lazy generation needs many rounds and does not help (a separate Z3 `arith.solver` sweep moves nothing either ‚Äî the lever is the encoding, not the engine). But on the path the recognizer runs by default, the ceiling moved by more than an order of magnitude, while real biological graphs remain decided in well under a second (¬ß3.10).

## 4. Discussion

**Correctness is the load-bearing result.** The most consequential change in this pass is not the faster encoding or the new repair capability ‚Äî it is that the recognizer's verdicts are now validated against an independent, obviously-correct oracle and a trustworthy second decision procedure. A recognizer is an oracle other research builds on; if it can silently accept a non-Wheeler graph, every downstream claim that rests on "this is a Wheeler graph" is unsupported. Establishing zero false-accepts and full oracle agreement (¬ß3.1) is therefore the prerequisite that makes the speed and capability results worth reporting at all.

**Why the three axes must stay separate.** Collapsing correctness, capability, and performance into one number is how tools come to look better than they are. A backend that returns answers quickly but occasionally returns the wrong one has not improved; an `unknown` from a solver is not a refutation; and a faster encoding that doubles memory is a trade, not a free win. Keeping the axes apart is what let this study report a real per-graph speedup (¬ß3.3) *and*, honestly, that it does not raise the full-range size ceiling (¬ß3.6) and costs memory on tractable graphs (¬ß3.4) ‚Äî three true statements that a single headline figure would have blurred.

**The heuristic is the engine.** The scalability separation (¬ß3.5) is a quantitative restatement of WGT's core design insight: the renaming heuristic, by narrowing each node's window before the solver runs, is what lets the default backend scale orders of magnitude past the no-heuristic full-range path and the exponential reference. The improvements here sharpen the periphery ‚Äî a leaner full-range encoding, a trustworthy reference, repair ‚Äî around a method whose central idea is unchanged and remains the reason recognition is practical at all.

**Repair changes what a negative verdict means.** Recognition answers a yes/no question; repair (¬ß3.7) makes "no" actionable. Because the trie unfolding preserves the path-string set exactly, a graph that cannot be indexed as-is can be replaced by one that can, representing the same strings ‚Äî a useful primitive for building indexable pangenome structures, and a direction (minimal node-splitting, cyclic graphs) with clear future work.

**Verification as software practice.** The brute-force oracle and differential harness are cheap to run and decisive on small graphs, and they are the reason every claim in ¬ß3.1 is checkable rather than asserted. The same discipline that surfaced the accept-path edge cases (¬ß2.4) is directly transferable to other combinatorial recognizers: build a slow, obviously-correct reference, then differential-test the fast implementation against it across many generated inputs.

**Recognition is not the bottleneck in practice.** The scalability study (¬ß3.5, ¬ß3.9) measures adversarial synthetic families precisely because they are the worst case; the practicality study (¬ß3.10) shows where real data actually sits. Every Wheeler graph built from a real gene alignment was recognized in under a second, three orders of magnitude inside the one-hour ceiling, and parallelism does not move that ceiling because the cost is a single `QF_IDL` solve (¬ß3.9). The practical lever is therefore *upstream*: which construction one chooses determines whether the graph is Wheeler at all (¬ß3.10), and when it is not, repair (¬ß3.7) turns that "no" into an indexable graph spelling the same strings. For realistic pangenome-scale inputs the open problem is graph *construction and size*, not recognition speed.

## 5. Limitations

The study and tool are presented with the following honest caveats:

1. **Oracle reach.** The brute-force oracle is exact but $O(n!\cdot e^2)$, so ground truth is direct only for $n\leq9$. Correctness on larger graphs rests on cross-backend agreement and construction-by-design families, not on the oracle.
2. **The full-range ceiling is unchanged.** The sparse encoding lowers per-graph time but not the size at which the full-range path can decide: both the new and prior encodings stop at $n=832$ where Z3 returns `unknown` (¬ß3.6). Raising that ceiling is a solver-encoding problem left open.
3. **A memory cost on tractable graphs.** The sparse encoding's auxiliary variables raise peak memory by roughly 2√ó on graphs whose old encoding never exploded (¬ß3.4); it is a space-for-time trade, not a uniform improvement.
4. **Recognition remains NP-complete.** None of this changes the worst case. The dense, all-distinct-endpoint `complete` family is where every backend is slowest, and where the full-range block is correctly disabled.
5. **Repair is not minimal and is DAG-only.** The trie unfolding always succeeds for a DAG and preserves the string set, but the result is not a minimum-node Wheeler graph; cyclic graphs, where a string-preserving repair need not exist, are out of scope.
6. **The per-type speedup covers paired jobs only.** The ¬ß3.8 breakdown uses the jobs with a comparable finite time from *both* versions; the excluded rows are the largest amino-acid instances where v1.0.0's dense encoding exceeds the timeout (so no finite ratio exists). Those omissions are overwhelmingly cases where v1.0.0 itself times out, so the table is a conservative per-type summary rather than an optimistic subset.

## 6. Availability and reproducibility

WGT is open source ([github.com/Kuanhao-Chao/Wheeler_Graph_Toolkit](https://github.com/Kuanhao-Chao/Wheeler_Graph_Toolkit)); the verification and improvement work described here is on the `devel` branch. The recognizer builds from source against Z3 (`recognizer/`); on Linux it is built directly with g++ linking Z3 statically (the committed `recognizer/bin/recognizer` is a macOS binary and must be rebuilt on other platforms). A typical recognition run is:

```
./bin/recognizer graph.dot          # default: heuristic + SMT (Z3)
./bin/recognizer graph.dot -s p     # permutation backend
./bin/recognizer graph.dot -f       # full-range SMT (no heuristic narrowing)
```

The brute-force oracle and differential harness live under `verify/` (`brute_oracle.py`, `difftest.py`, `difftest_exp.py`, `edgecases.py`); the repair module under `repair/` (`wheelerize.py`); the rebuilt exponential decision procedure under `benchmark/exponential_recognizer/`; and the full benchmark ‚Äî the three-axis sweeps, the scalability limit test, and the figure-generation scripts ‚Äî under `benchmark/` (`REPORT.md` is the consolidated record of every number in this report, with per-figure data under `benchmark/report_figs/data/` and `benchmark/limit_test/results/`). Comparisons were run on a single Linux host with the recognizer built from source and Z3 linked statically; the scalability sweep used a 600 s per-graph budget with three replicates per size.
