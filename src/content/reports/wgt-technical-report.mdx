## 1. Introduction

### 1.1 The Wheeler-graph recognition problem

The Burrows–Wheeler Transform and the FM-index let a string be indexed once and then queried in time that scales with the query rather than the text — the machinery underneath read aligners such as Bowtie and BWA. **Wheeler graphs** [2] generalize that idea from a string to a labeled graph. An edge-labeled directed graph $G=(V,E)$ is a Wheeler graph if there is a total order $\pi$ on $V$ such that the following hold for all nodes $u,v\in V$ and all pairs of edges $e_1=(u_1,v_1,a_1)$, $e_2=(u_2,v_2,a_2)$:

$$
\textbf{(A1)}\quad \mathrm{indeg}(u)=0 \,\wedge\, \mathrm{indeg}(v)>0 \;\Rightarrow\; \pi(u) < \pi(v),
$$
$$
\textbf{(A2)}\quad a_1 < a_2 \;\Rightarrow\; \pi(v_1) < \pi(v_2),
$$
$$
\textbf{(A3)}\quad a_1 = a_2 \,\wedge\, \pi(u_1) < \pi(u_2) \;\Rightarrow\; \pi(v_1) \leq \pi(v_2).
$$

When such an order exists, the FM-index query machinery carries over to the graph, which is why Wheeler graphs underpin graph-based pangenome indexes. The difficulty is that deciding whether a valid $\pi$ exists is **NP-complete**, and hard even to approximate [3]. Every decision procedure must therefore search the space of orderings; they differ only in how cleverly they prune it.

### 1.2 WGT and the heuristic-plus-solver approach

The Wheeler Graph Toolkit (WGT) [1] makes recognition practical with its recognizer, **Wheelie**, which runs a three-step pipeline (§2.1). The decisive idea is a **renaming heuristic**: because a node's position in any valid order is strongly constrained by the labels on its incoming edges, the heuristic groups nodes by incoming label and repeatedly sorts and relabels them, propagating constraints until they converge. Convergence either exposes a contradiction (the graph is not a Wheeler graph) or narrows each node to a small window of possible positions, which is handed to one of two interchangeable backends: a **permutation** search that enumerates the remaining orderings with pruning, or an **SMT** backend that encodes the residual constraints for the Z3 solver [4]. WGT also ships generators (De Bruijn graphs, tries, reverse-deterministic graphs, and random graphs), a visualizer, and a benchmark harness that includes a reference implementation of the only previously proposed recognition algorithm, due to Gibney and Thankachan [3].

### 1.3 What motivated this study

The published WGT method [1] is unchanged here. What motivated this pass is that a recognizer is an oracle other research builds on, so it must be **demonstrably** correct, and two adjacent capabilities were missing. Three concerns drove the work:

1. **No independent ground truth.** The recognizer's verdicts were trusted on the strength of the algorithm and internal checks, but not cross-validated against a separate, obviously-correct implementation. A recognizer that occasionally accepts a non-Wheeler graph silently invalidates everything downstream of that verdict, so this is the first thing that must be established, not assumed.
2. **A reference that could not refute.** The bundled exponential reference was a timing baseline, not a decision procedure: it did not check the axioms against the graph's edges and so could not return a definite "not a Wheeler graph," which makes it unusable as an independent second opinion.
3. **Two missing capabilities.** The full-range SMT path emitted a quadratic number of constraints and became the bottleneck on dense, few-label graphs; and there was no way to turn a "not a Wheeler graph" verdict into a constructive fix.

### 1.4 Contributions

This report documents the verification framework and the improvements it enabled, and evaluates them along three deliberately separated axes (§2.8). The principal contributions are:

- An **independent brute-force oracle** (§2.2) that enumerates all $n!$ node orderings and checks A1–A3 directly against the graph's edges, plus a **differential-testing harness** (§2.3) that establishes recognizer correctness against it and motivated targeted **hardening** of the accept paths (§2.4): zero false-accepts and full oracle agreement (§3.1).
- An **honest exponential decision procedure** (§2.5) — the reference implementation rebuilt to enumerate orderings and verify the axioms, returning a three-way verdict — giving a trustworthy second opinion and new verdicts on graphs it previously left undecided (§3.2).
- A **sparser full-range SMT encoding** (§2.6) that reduces the A2 cross-label constraint from $O(E^2)$ to $O(E+L)$ and the A3 within-label constraint from $O(E_\ell^2)$ to $O(D_\ell^2+E_\ell)$, with a guard that applies the new within-label form only where it pays (§3.3, §3.4).
- A **string-preserving repair module** (§2.7) that unfolds any non-Wheeler DAG into a Wheeler graph preserving its path-string set exactly (§3.7).
- A **scalability characterization** (§2.8, §3.5) of every backend's size limit under a fixed time budget.

## 2. Methods

### 2.1 The recognizer pipeline

Wheelie parses a DOT graph into an internal digraph and runs the three-step pipeline of Table 1 (`recognizer/src/`). Steps 1–2 are the heuristic; step 3 is the solver, which is interchangeable between the SMT and permutation backends and, in full-range mode (`-f`), always uses SMT with the step-2 range narrowing disabled.

**Table 1.** The Wheelie recognition pipeline.

| Step | Stage | Purpose |
|---|---|---|
| 0 | Parse | Read the `strict digraph` DOT, build the internal digraph from `A -> B [label=x]` edges. |
| 1 | `relabel_initialization` | Impose the axiom-implied initial constraints (in-degree-0 prefix; per-label head grouping). If they are already violated, reject — the graph is not a Wheeler graph. |
| 2 | `innodelist_sort_relabel` | The renaming heuristic: iteratively sort and relabel nodes by incoming-edge structure, narrowing each node to an order window $[\mathrm{lb},\mathrm{ub}]$. Under `-f`, the range-narrowing branches are disabled. |
| 3 | Solve | Search within the windows: **SMT** (`solve_smt`, Z3 `QF_IDL`; the default) or **permutation** (`permutation_start`, pruned enumeration with `WG_checker`). |
| 4 | Verdict | Exit code $1$ = Wheeler, $-1$ = not a Wheeler graph, $0$ = undecided. Benchmark mode (`-b`) prints `verdict, nodes, cpu, path`. |

### 2.2 The brute-force oracle

Ground truth is established by a deliberately trivial procedure (`verify/brute_oracle.py`), written independently of the recognizer so that the two cannot share a bug. For a graph with $n$ nodes it enumerates all $n!$ node orderings; for each, it checks A1–A3 directly against the edge list and accepts iff some ordering satisfies all three. Its cost,

$$
T_\text{oracle}(n,e) = O(n!\cdot e^2),
$$

confines it to $n \leq 9$, but on those graphs it is an incontrovertible decision: a completely separate implementation computing the same predicate from first principles. (One subtlety the oracle itself had to get right: a single node with two differently-labeled self-loops violates A2, so only the empty graph, not every one-node graph, is a trivial accept.)

### 2.3 Differential-testing harness and corpora

The harness (`verify/difftest.py`, `verify/edgecases.py`) draws graphs from many families, runs each through both the oracle and every recognizer backend, and flags any verdict that differs. Table 2 summarizes the corpora used across this report.

**Table 2.** Evaluation corpora.

| Corpus | Size | Role |
|---|---:|---|
| Reject-heavy random (non-Wheeler) | 4,559 | Stress the accept paths: count false-accepts (§3.1). |
| Oracle-decidable real subset ($n\leq9$) | 931 | Verdict agreement + capability vs the oracle (§3.1, §3.2). |
| Synthetic random + edge cases | 1,516 | Verdict agreement across backends (§3.1). |
| Real DNA De Bruijn / reverse-deterministic | 289 | Full-range `-f` timing (§3.3). |
| Synthetic `complete` / `dnfa` families | ladder | Scalability limits (§3.5). |
| Random DAGs | 820 | Repair (§3.7). |

Each family is Wheeler or non-Wheeler by construction or by oracle anchoring, so every rung doubles as a correctness test as well as a timing or scaling measurement.

### 2.4 Correctness hardening

Differential testing surfaced a small number of edge cases in the recognizer's *accept* paths — the places where a verdict of "Wheeler" is issued — each of which was tightened so that an accept is never issued without a satisfied check:

- **Reject on permutation exhaustion.** The permutation backend (`permutation_start`) issued its verdict only on the accept branch; on a fully pruned, exhausted search it returned without a decision and the caller defaulted to "Wheeler." It now rejects explicitly when a non-exhaustive search completes with no valid order.
- **Re-check the propagation fast path.** When step-2 propagation alone settled every node to a singleton window, the result was accepted without the final `WG_checker` the SMT and permutation paths run. The fast path now verifies A1–A3 before accepting.
- **Overflow and degeneracy guards.** An integer overflow in the solver-dispatch heuristic (a product of per-window factorials) was made saturating; and the SMT path now guards degenerate inputs — an empty or isolated-node graph that produced an empty `z3::distinct` — that previously aborted.

These are corrective, not algorithmic: they change a verdict only on inputs the un-hardened code would have decided incorrectly or crashed on, and they leave every already-correct verdict unchanged. The exponential reference rewrite (§2.5) also corrected the oracle's own one-node shortcut noted in §2.2.

### 2.5 An honest exponential decision procedure

The bundled reference implementation of the Gibney–Thankachan algorithm [3] was, as originally written, a timing baseline: its checker received only scalar counts rather than the graph's edges, so it could not test the axioms and never emitted a definite non-Wheeler verdict. To make it a trustworthy independent second opinion, its decision core was reimplemented (`benchmark/exponential_recognizer/`) to enumerate node orderings and verify A1–A3 against the real edges — mirroring the oracle of §2.2 — under an $O(n!\cdot e^2)$ work budget, returning a clean three-way verdict:

$$
\text{verdict} = \begin{cases}
\text{Wheeler} & \text{some ordering satisfies A1--A3},\\
\text{not a Wheeler graph} & \text{no ordering does, within budget exhausted},\\
\text{undecided} & \text{work budget exceeded before exhaustion}.
\end{cases}
$$

It is now usable as a decision procedure on small graphs and as a second independent check against the recognizer, not merely a stopwatch.

### 2.6 A sparser full-range SMT encoding

In full-range mode the recognizer hands the entire ordering space to Z3: one integer order variable per node, a global `distinct`, and the axiom constraints, solved as a quantifier-free integer-difference-logic (`QF_IDL`) problem [4]. The cost is dominated by the *number of axiom constraints*, and the original encoding wrote them pairwise.

**Cross-label monotonicity (A2).** Writing A2 as an all-pairs constraint over edges costs $O(E^2)$. The sparse form introduces, per label, head-boundary variables ($\#\mathrm{lo}_\ell,\#\mathrm{hi}_\ell$) that bracket the order positions of that label's heads, and chains the brackets across labels in label order. This expresses the same "smaller label ⇒ earlier head" relation in

$$
O(E^2) \;\longrightarrow\; O(E+L),
$$

where $L$ is the number of distinct labels.

**Within-label ordering (A3).** For a single label $\ell$ with $E_\ell$ edges, the pairwise A3 constraint costs $O(E_\ell^2)$. The sparse form encodes A3 as an endpoint block: with $D_\ell = \min(\#\text{distinct tails},\,\#\text{distinct heads})$ for that label, per-endpoint min/max position variables ($\#\mathrm{mn},\#\mathrm{mx}$) replace the all-pairs comparison, giving

$$
O(E_\ell^2) \;\longrightarrow\; O(D_\ell^2 + E_\ell).
$$

Both forms are **equisatisfiable** with the pairwise originals — they accept and reject exactly the same graphs — so correctness is unaffected; only the encoding size changes.

**A shape-aware guard.** The block form trades pairwise difference atoms for $2D_\ell$ auxiliary integer variables, and those auxiliaries enlarge Z3's search. Gating the block on atom count alone (fewer atoms) is insufficient: on dense, few-label instances the auxiliaries can make the *solve* slower even when the encoding is smaller. The block is therefore applied to label $\ell$ only when

$$
2D_\ell < E_\ell \quad\wedge\quad D_\ell(D_\ell-1)+2E_\ell < E_\ell(E_\ell-1),
$$

i.e. only when the label's endpoints are genuinely sparse (the first clause; the De Bruijn regime, $D_\ell/E_\ell \approx 0.25$) *and* the block emits fewer atoms (the second). Otherwise the verified pairwise loop is used. Because the two encodings are equisatisfiable, the guard changes only *which* encoding is used, never the verdict.

### 2.7 String-preserving graph repair

Repair (`repair/wheelerize.py`) turns a non-Wheeler **DAG** into a Wheeler graph by unfolding it into the **trie of its path-strings**. A trie is always a Wheeler graph — order its nodes by the co-lexicographic rank of the string spelled along each incoming path — so the construction always succeeds for a DAG, and it preserves the **set of strings** the graph spells, exactly. Each repair self-verifies three independent ways before it is returned: the recognizer accepts the output, the oracle accepts it when it is small enough, and the set of path-strings is confirmed identical to the input's. The node-count ratio (output / input) can fall below one — when distinct input nodes spell the same string they are *merged* — which is a reduction in nodes, not a loss of represented strings; ratios above one are genuine node splitting. This first version is correct but not minimal; minimal node-splitting is left to future work, and cyclic graphs (where a string-preserving repair is not always possible) are out of scope.

### 2.8 Benchmark design and the three-axis methodology

The cardinal rule of this evaluation is that three questions are reported **separately** and never merged into a single "X% better":

- **Correctness** — does the verdict match the independent oracle? Measured as false-accepts on the reject-heavy corpus and verdict agreement on the oracle-decidable and synthetic corpora.
- **Capability** — can a verdict be produced at all? Measured as the fraction of a corpus a procedure decides within its budget (the rebuilt exponential reference), independent of speed.
- **Performance** — how fast, and how large, when correct? Measured as full-range setup/solve time, the per-graph timing distribution, peak memory, and the largest graph each backend decides under a fixed budget.

A faster wrong answer is not progress, an undecided result is not a refutation, and additional memory is a cost to be stated, not hidden. The full-range timing uses string labels (matching the biological corpora), the median of replicate runs, and the `-b` CPU-time column, which is robust to machine load; timeouts are recorded as timeouts, never as a verdict.

**The scalability sweep** (`benchmark/limit_test/`) climbs a geometric size ladder per (backend, family) under a **600 s** wall-clock budget, takes the median of $R=3$ replicate graphs per rung, and bisects the largest size still decided. Limits are **typed**: *threshold* (timeout-bounded — the backend would continue with more time), *capped* (a hard capability ceiling — the exponential reference's $n!$ budget, or Z3 returning `unknown`), or *unreached* (no wall hit up to the ladder top). The two families are `complete` (dense, all-distinct endpoints — the symmetric worst case) and `dnfa` (random $d$-NFA-style), both Wheeler by construction so each rung is also a correctness check.

**Environment and reproducibility.** Comparisons are run on a single Linux host with the recognizer built from source (g++ 17, Z3 linked statically). The recognizer, oracle, harnesses, and all source data and figures are versioned in the repository (`benchmark/REPORT.md` is the consolidated source for every number here; figure data lives under `benchmark/report_figs/data/` and `benchmark/limit_test/results/`).

## 3. Results

### 3.1 Correctness: agreement with an independent oracle

Against the brute-force oracle, the hardened recognizer is correct on every graph both can decide. On the reject-heavy corpus of 4,559 non-Wheeler graphs — built specifically to exercise the accept paths — it produces **zero** false-accepts on every backend (Table 3). Before hardening, the same corpus exposed a substantial false-accept rate on the permutation paths; after the §2.4 fixes the rate is zero throughout, which is the property a recognizer must have to be usable as an oracle for other work.

### 3.2 Capability: verdicts that did not exist before

The rebuilt exponential decision procedure (§2.5) can now refute, not only confirm. On the oracle-decidable subset of the WGT benchmark corpora (931 graphs of at most nine nodes), it returns a verdict on **all 931** — where the original reference exceeded its budget on **159** of them — and every verdict matches the oracle (Table 5, Fig. 2). Among the newly decided graphs are **seven non-Wheeler graphs built from real human gene orthologs** (BTBD17, FAM53A, LCP1, TRAM1, and TRPC1 De Bruijn graphs) that the original reference could never classify. This is a pure capability gain: these are decisions that did not previously exist at any time budget, distinct from any question of speed.

### 3.3 Performance: the encoding measured in atoms, then in time

**The result, then the evidence.** The new full-range (`-f`) SMT encoding is faster, validated to the atom, and honestly bounded:

- **A median ≈13× smaller formula.** The encoding emits about **13.4× fewer** SMT assertions (per-type 3.8–15.6×; up to ~300× on the largest many-label graphs) — genuinely sub-quadratic ($\propto E^{1.57}$) where the within-label block fires, a large constant factor elsewhere — a count *validated exactly* against Z3's own `s.assertions().size()` from an instrumented build (below).
- **Nearly-free construction, ~2× end-to-end.** Building the problem becomes **≈14.5× cheaper** on the headline graph, and total full-range time falls by a median **1.79×** across a 682-graph cross-type grid (geomean 1.77×, up to 3.5×; 1.53× on the 289-graph DNA corpus), **with zero regressions** — the new encoding is ≥ the old on every graph measured (§3.8).
- **Complementary sparsifications.** The cross-group A2 form carries the many-label amino-acid regime; the within-group A3 block adds a further ≈1.8× where it fires (few-label DNA) (§3.8).
- **Honestly bounded.** A smaller formula speeds the solver but does **not move the size ceiling** — identical old and new, set by Z3 returning `unknown` at $n=832$ (§3.6) — and the block costs ~2× peak RAM only when it fires (§3.4). The graph that *scales* is the default recognizer (§3.5), which never builds this formula at all.

The rest of this section, and §3.4–§3.8, are the evidence for each point.

Before any wall-clock number, the sparsification can be measured *directly* — as the size of the SMT problem it builds. `atom_count.py` computes, analytically from each graph's label and endpoint structure, exactly how many `s.add(...)` assertions each generation emits under `-f`, mirroring the encoder line for line. The count is **exact**, not a model: on a calibration set it equals Z3's own `s.assertions().size()` from an instrumented build to the atom (the only residual is a shared `n+1`-assertion range/`distinct` baseline identical across generations). Fitting the counts over all 900 biological graphs against the edge count $E$ gives clean power laws (Fig. 3): the prior encoding is **$\propto E^{2.00}$ on every graph type** — the dense all-pairs baseline — while this work is genuinely **sub-quadratic only where the within-label block fires** (few-label De Bruijn DNA, $\propto E^{1.57}$, a 24× atom reduction at the largest graph). On twenty-letter amino-acid alignments the block stays off and the gain is instead a large **constant factor** from the cross-group sparsification (≈17×), with the exponent near $E^{1.8}$ — an honest distinction the fit makes visible.

The decisive caveat is that **atoms are not wall time**. The formula shrinks by a median of about thirteen-fold (and up to several-hundred-fold on the largest many-label graphs), and the *encoding setup* shrinks in step (≈14× on the headline graph, below). But the **total** full-range time falls only ≈2×, because Z3's *solve* is the bottleneck and its cost is not proportional to the atom count — a smaller formula helps the solver but does not shrink the NP-hard search by the same factor. The sparsification's first-order effect is to make building the problem nearly free; the solve speedup is a real but second-order benefit.

### 3.4 Memory: an honest space-for-time trade

Memory depends on *which* sparsification is active, and splits into two honest regimes. **Where the within-label A3 block fires** — low-$D/E$ graphs like *DOCK4* and *TRPC1* at $k=5$ ($D/E\approx0.27$) — the new encoding pays for the block's auxiliary integer variables with about twice the peak RAM (0.66→1.21 GB and 0.67→1.29 GB): the genuine space-for-time trade. **Where the block is off** — higher-$D/E$ graphs, the $k=6$ De Bruijn DNA ladder of Fig. 5B ($D/E\approx0.6$–$0.85$) — there are no such aux variables, and the sparse cross-group A2 form *replaces* the prior encoding's dense quadratic constraint set with a linear one, so the new encoding is **no heavier than pre-4.1** (comparable, up to ~15–20% lighter on some rungs) and matches pre-4.2. Memory is therefore a ~2× cost only when the block engages and no cost at all when it does not — and the time wins (≈14× setup, ≈2× total) hold in both regimes. The real limit is solve time on the all-distinct dense cases where the full-range path does not finish at all (§3.5).

### 3.6 The full-range ceiling is a solver limit, identical old and new

The full-range path's size limit deserves a precise reading, because it bears on what §3.3 does and does not claim. Both the new and the prior `-f` encodings stop at exactly **832** nodes on both synthetic families — and not by timeout but because Z3 returns `unknown` (undecided) on the `QF_IDL` problem at that size. That ceiling is therefore a property of the solver on this encoding, the **same** for both generations (Fig. 7). The consequence is the honest scope of the §3.3 result: the sparser encoding **lowers the time curve, not the size wall.** On the dense `complete` and `dnfa` families the two curves essentially coincide (panels A, B) — those are few-label and dense, so neither sparsification has much to remove. Either way the point is the same — the encoding **lowers the time curve, not the size wall**; the gap below the wall is the speedup, and the wall itself does not move.

### 3.9 The one-hour ceiling — parallelism, a native solver, and how lazy A3 generation finally raises it

§3.5 measured sizes under a 600 s budget; the natural follow-up is how large a graph the production default decides given a full **hour**. A fresh 3,600 s sweep (default SMT, $R=2$, same ladder-then-bisect, in a clean output directory so it cannot inherit the 600 s timeouts as false caps) answers it, and every rung again served as a large-scale differential correctness test on Wheeler-by-construction families: **zero correctness failures** over the whole one-hour sweep. The default backend decides `complete` graphs of **4,608** nodes (median 52 min at the limit) and `dnfa` graphs of **3,584** (median 59 min), both still threshold-limited — Z3 is making progress at the wall, not returning `unknown` (Table 11, Fig. 12).

|---|---:|---:|

Six times the budget buys only about **1.64×** the size (2,816 → 4,608; 2,176 → 3,584), the sub-linear return expected when the solver's cost grows steeply with $n$ on these symmetric worst cases. But the operative point is the green band in Fig. 12: **real biology sits an order of magnitude below this ceiling.** The largest graph any real Ensembl gene alignment produced (§3.10) is 2,399 nodes, and all of them are decided in under a second — so the hour-scale runs are a synthetic stress test, not a practical limit.

A reasonable question is whether parallelism would push the ceiling higher; it would not, for a structural reason. At scale the recognizer's wall time is almost entirely the Z3 `QF_IDL` solve — on the headline full-range graph the solve is ~98% of total and encoding/setup ~2% (Fig. 5), and the gap only widens with size — so "parallelize the recognizer" means "parallelize a single `QF_IDL` query," which Z3's parallel mode does not reliably speed up (a prototype with `parallel.enable` showed no measurable wall-time gain on the worst-case families; it was not retained, so this is reported qualitatively). The only other component, the Step-2 range-narrowing heuristic, is a sequential prefix-sum relabel and a negligible fraction of runtime. The honest answer is therefore that the parallel size ceiling is essentially the serial one, and the numbers here are reported as serial.

A deeper question is whether the *solver itself* could be made faster, or replaced, to push the ceiling. We profiled the wall directly and then built the most direct attack on it. Instrumentation of Z3's own statistics shows the surprise: on these symmetric worst-case families Z3 solves with **near-zero search** — only 55 to 510 SAT conflicts as $n$ grows from 400 to 1,000 — while its memory allocations explode roughly **12,000-fold** over the same range. The ceiling is therefore not a backtracking wall but the cost of *propagating* an $O(n^2)$ difference-logic encoding that grows with the graph; a literature and lower-bound assessment frames why no encoding removes it (recognition is NP-complete already at a two-letter alphabet, even for DAGs, with no known fixed-parameter-tractable handle on the general decision).

That diagnosis — memory and propagation over a materialized $O(n^2)$ formula, not search — points to an obvious idea: a native solver that decides the post-heuristic residual *without ever building the formula*, in $O(V+E)$ space. We implemented one (a difference-logic bound-consistency propagator with a trailed, randomized-restart search) and verified it sound — zero disagreements with the brute-force oracle across roughly twelve thousand graphs and every hand-built edge case. But it does **not** move the ceiling: it decides only to a few hundred nodes on the synthetic families, three-to-nine times *below* Z3, before its search blows up. The reason closes the loop with the profiling — Z3's `QF_IDL` **theory propagation** is exactly what makes its search trivial, and hand-rolled bound consistency is far weaker, so the native search explodes where Z3's does not. The lesson was subtle but decisive: the *diagnosis* (the encoding, not the search) was right, but *replacing* Z3's search was the wrong fix. The right move is to keep Z3's search and change only **what it is fed**.

That is exactly what a lazy encoding does, and it moves the ceiling by more than an order of magnitude. The A3 axiom — within a label group, heads are monotone non-decreasing in tail order — is a **chain** constraint, and it is $O(n^2)$ only *statically*, because the tail order is unknown a priori (forcing an implication for every pair of same-label edges). Given a concrete candidate order the tail order is *known*, so a violation is forbidden by just $O(E)$ consecutive-pair constraints. This invites a counterexample-guided loop (lazy SMT / CEGAR): assert the bracket and all-different constraints but **no A3**, ask Z3 for a model, re-check A3 in $O(E\log E)$, add only the few consecutive-pair constraints the model actually violates, and re-solve — monotonically, so Z3 keeps its learned clauses across rounds. Z3 never materializes the $O(n^2)$ formula; it only ever sees the handful of A3 constraints that bind. The effect is dramatic, because the Step-2 heuristic's brackets already nearly determine the order, leaving the full A3 encoding **~99.99% redundant**: deciding a 32,768-node `complete` graph builds just **3,835 of its 402,722,292** A3 pairs. Where the vanilla backend needs **539 s and 7.5 GB** to decide 2,816 nodes and then times out at 4,096, the lazy backend decides 2,816 in **0.9 s and 56 MB** and climbs to **≥ 32,768 nodes in under a minute** — roughly **600× the speed at 1/130th the memory**, and a **>10× higher ceiling** that is bounded by our test ladder, not by the solver (Table 12). It is verified sound by the same oracle differential as everything else (zero disagreements over 13,438 random and constructed graphs plus every edge case, with an independent re-check of each emitted ordering), and whenever it cannot decide within budget it falls back to the full Z3 solve, so a verdict can never regress. It is now the **production default**; `-s smt` selects the original backend, and the one-hour figures of Table 11 describe that original backend.

The earlier "practical limit" reading held only because two levers were spent — a native *search*, and *static* encoding compression (the within-label block of §3.3, which *regresses* on these dense families). Lazy *dynamic* generation is a third lever, and on the production path the recognizer was not at the limit at all. Two honest caveats keep the broader picture intact. The worst-case asymptotics are unchanged — recognition is NP-complete already at a two-letter alphabet, even for DAGs, so no encoding makes the *general* problem polynomial. And the gain is tied to the heuristic's brackets: in the full-range `-f` regime, where Step 2 is deliberately bypassed and those brackets are absent, lazy generation needs many rounds and does not help (a separate Z3 `arith.solver` sweep likewise moves nothing — the lever is the encoding, not the engine). But for the graphs the recognizer actually runs on, the productive ceiling moved by more than an order of magnitude, and the heuristic fast-path still decides every real biological graph in well under a second (§3.10).

We ran all three over 50 human gene orthologue alignments retrieved from Ensembl (DNA and amino-acid, sequence length capped for the De Bruijn ladder), 500 graphs in all, measuring graph size, recognition wall time, and Wheeler rate by construction (Table 12, Fig. 13).

## 4. Discussion

**Correctness is the load-bearing result.** The most consequential change in this pass is not the faster encoding or the new repair capability — it is that the recognizer's verdicts are now validated against an independent, obviously-correct oracle and a trustworthy second decision procedure. A recognizer is an oracle other research builds on; if it can silently accept a non-Wheeler graph, every downstream claim that rests on "this is a Wheeler graph" is unsupported. Establishing zero false-accepts and full oracle agreement (§3.1) is therefore the prerequisite that makes the speed and capability results worth reporting at all.

**Why the three axes must stay separate.** Collapsing correctness, capability, and performance into one number is how tools come to look better than they are. A backend that returns answers quickly but occasionally returns the wrong one has not improved; an `unknown` from a solver is not a refutation; and a faster encoding that doubles memory is a trade, not a free win. Keeping the axes apart is what let this study report a real per-graph speedup (§3.3) *and*, honestly, that it does not raise the full-range size ceiling (§3.6) and costs memory on tractable graphs (§3.4) — three true statements that a single headline figure would have blurred.

**The heuristic is the engine.** The scalability separation (§3.5) is a quantitative restatement of WGT's core design insight: the renaming heuristic, by narrowing each node's window before the solver runs, is what lets the default backend scale orders of magnitude past the no-heuristic full-range path and the exponential reference. The improvements here sharpen the periphery — a leaner full-range encoding, a trustworthy reference, repair — around a method whose central idea is unchanged and remains the reason recognition is practical at all.

**Repair changes what a negative verdict means.** Recognition answers a yes/no question; repair (§3.7) makes "no" actionable. Because the trie unfolding preserves the path-string set exactly, a graph that cannot be indexed as-is can be replaced by one that can, representing the same strings — a useful primitive for building indexable pangenome structures, and a direction (minimal node-splitting, cyclic graphs) with clear future work.

**Verification as software practice.** The brute-force oracle and differential harness are cheap to run and decisive on small graphs, and they are the reason every claim in §3.1 is checkable rather than asserted. The same discipline that surfaced the accept-path edge cases (§2.4) is directly transferable to other combinatorial recognizers: build a slow, obviously-correct reference, then differential-test the fast implementation against it across many generated inputs.

**Recognition is not the bottleneck in practice.** The scalability study (§3.5, §3.9) measures adversarial synthetic families precisely because they are the worst case; the practicality study (§3.10) shows where real data actually sits. Every Wheeler graph built from a real gene alignment was recognized in under a second, three orders of magnitude inside the one-hour ceiling, and parallelism does not move that ceiling because the cost is a single `QF_IDL` solve (§3.9). The practical lever is therefore *upstream*: which construction one chooses determines whether the graph is Wheeler at all (§3.10), and when it is not, repair (§3.7) turns that "no" into an indexable graph spelling the same strings. For realistic pangenome-scale inputs the open problem is graph *construction and size*, not recognition speed.

## 5. Limitations

The study and tool are presented with the following honest caveats:

1. **Oracle reach.** The brute-force oracle is exact but $O(n!\cdot e^2)$, so ground truth is direct only for $n\leq9$. Correctness on larger graphs rests on cross-backend agreement and construction-by-design families, not on the oracle.
2. **The full-range ceiling is unchanged.** The sparse encoding lowers per-graph time but not the size at which the full-range path can decide: both the new and prior encodings stop at $n=832$ where Z3 returns `unknown` (§3.6). Raising that ceiling is a solver-encoding problem left open.
3. **A memory cost on tractable graphs.** The sparse encoding's auxiliary variables raise peak memory by roughly 2× on graphs whose old encoding never exploded (§3.4); it is a space-for-time trade, not a uniform improvement.
4. **Recognition remains NP-complete.** None of this changes the worst case. The dense, all-distinct-endpoint `complete` family is where every backend is slowest, and where the full-range block is correctly disabled.
5. **Repair is not minimal and is DAG-only.** The trie unfolding always succeeds for a DAG and preserves the string set, but the result is not a minimum-node Wheeler graph; cyclic graphs, where a string-preserving repair need not exist, are out of scope.
6. **The per-type speedup covers paired jobs only.** The full 900-job timing grid ran to completion, but the §3.8 breakdown uses the 682 jobs with complete timing across all three binaries; the other 218 are the largest amino-acid instances where one or more binaries exceed the 120 s cap (all-timeout, so no finite ratio exists). Those omissions are overwhelmingly cases where the old pre-4.1 binary itself times out, so the table is a conservative per-type summary of the completed grid rather than an optimistic subset.

## 6. Availability and reproducibility

WGT is open source ([github.com/Kuanhao-Chao/Wheeler_Graph_Toolkit](https://github.com/Kuanhao-Chao/Wheeler_Graph_Toolkit)); the verification and improvement work described here is on the `devel` branch. The recognizer builds from source against Z3 (`recognizer/`); on Linux it is built directly with g++ linking Z3 statically (the committed `recognizer/bin/recognizer` is a macOS binary and must be rebuilt on other platforms). A typical recognition run is:

```
./bin/recognizer graph.dot          # default: heuristic + SMT (Z3)
./bin/recognizer graph.dot -s p     # permutation backend
./bin/recognizer graph.dot -f       # full-range SMT (no heuristic narrowing)
```

The brute-force oracle and differential harness live under `verify/` (`brute_oracle.py`, `difftest.py`, `difftest_exp.py`, `edgecases.py`); the repair module under `repair/` (`wheelerize.py`); the rebuilt exponential decision procedure under `benchmark/exponential_recognizer/`; and the full benchmark — the three-axis sweeps, the scalability limit test, and the figure-generation scripts — under `benchmark/` (`REPORT.md` is the consolidated record of every number in this report, with per-figure data under `benchmark/report_figs/data/` and `benchmark/limit_test/results/`). Comparisons were run on a single Linux host with the recognizer built from source and Z3 linked statically; the scalability sweep used a 600 s per-graph budget with three replicates per size.
