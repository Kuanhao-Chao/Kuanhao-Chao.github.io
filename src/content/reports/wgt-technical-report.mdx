## 1. Introduction

### 1.1 The Wheeler-graph recognition problem

The Burrows–Wheeler Transform and the FM-index let a string be indexed once and then queried in time that scales with the query rather than the text — the machinery underneath read aligners such as Bowtie and BWA. **Wheeler graphs** [2] generalize that idea from a string to a labeled graph. An edge-labeled directed graph $G=(V,E)$ is a Wheeler graph if there is a total order $\pi$ on $V$ such that the following hold for all nodes $u,v\in V$ and all pairs of edges $e_1=(u_1,v_1,a_1)$, $e_2=(u_2,v_2,a_2)$:

$$
\textbf{(A1)}\quad \mathrm{indeg}(u)=0 \,\wedge\, \mathrm{indeg}(v)>0 \;\Rightarrow\; \pi(u) < \pi(v),
$$
$$
\textbf{(A2)}\quad a_1 < a_2 \;\Rightarrow\; \pi(v_1) < \pi(v_2),
$$
$$
\textbf{(A3)}\quad a_1 = a_2 \,\wedge\, \pi(u_1) < \pi(u_2) \;\Rightarrow\; \pi(v_1) \leq \pi(v_2).
$$

When such an order exists, the FM-index query machinery carries over to the graph, which is why Wheeler graphs underpin graph-based pangenome indexes. The difficulty is that deciding whether a valid $\pi$ exists is **NP-complete**, and hard even to approximate [3]. Every decision procedure must therefore search the space of orderings; they differ only in how cleverly they prune it.

### 1.2 WGT and the heuristic-plus-solver approach

The Wheeler Graph Toolkit (WGT) [1] makes recognition practical with its recognizer, **Wheelie**, which runs a three-step pipeline (§2.1). The decisive idea is a **renaming heuristic**: because a node's position in any valid order is strongly constrained by the labels on its incoming edges, the heuristic groups nodes by incoming label and repeatedly sorts and relabels them, propagating constraints until they converge. Convergence either exposes a contradiction (the graph is not a Wheeler graph) or narrows each node to a small window of possible positions, which is handed to one of two interchangeable backends: a **permutation** search that enumerates the remaining orderings with pruning, or an **SMT** backend that encodes the residual constraints for the Z3 solver [4]. WGT also ships generators (De Bruijn graphs, tries, reverse-deterministic graphs, and random graphs), a visualizer, and a benchmark harness that includes a reference implementation of the only previously proposed recognition algorithm, due to Gibney and Thankachan [3].

### 1.3 What motivated this study

WGT's published recognition framework [1] — the three axioms and the renaming heuristic — is unchanged; what changed between v1.0.0 and the current version is the *solver* the heuristic feeds (a lazy/CEGAR encoding, §2.6) and the soundness of the accept paths. Three concerns motivated that work, since a recognizer is an oracle other research builds on and must be **demonstrably** correct:

1. **No independent ground truth.** The recognizer's verdicts were trusted on the strength of the algorithm and internal checks, but not cross-validated against a separate, obviously-correct implementation. A recognizer that occasionally accepts a non-Wheeler graph silently invalidates everything downstream of that verdict, so this is the first thing that must be established, not assumed.
2. **A reference that could not refute.** The bundled exponential reference was a timing baseline, not a decision procedure: it did not check the axioms against the graph's edges and so could not return a definite "not a Wheeler graph," which makes it unusable as an independent second opinion.
3. **Two missing capabilities.** The full-range SMT path emitted a quadratic number of constraints and became the bottleneck on dense, few-label graphs; and there was no way to turn a "not a Wheeler graph" verdict into a constructive fix.

### 1.4 Contributions

This report compares the WGT recognizer's last stable GitHub release (v1.0.0, July 2023) against the current version along three deliberately separated axes (§2.8), with all intermediate development versions omitted. The principal differences are:

- A **lazy / CEGAR SMT solver** (§2.6) — now the default — that generates the within-label axiom on demand from counterexamples rather than materializing it in full. It moves the default-path size ceiling from v1.0.0's 2,816 nodes to at least 32,768, about **554× faster and 133× lighter** at matched size, verified sound and never regressing a verdict (§3.2). This is the headline difference.
- **Soundness fixes**: v1.0.0's permutation backends false-accept non-Wheeler graphs and its full-range path can crash on degenerate input; the current version is correct on every backend, validated against an **independent brute-force oracle** (§2.2) and a **differential-testing harness** (§2.3) — zero false-accepts and zero false-rejects (§3.1, §2.4).
- A **sparser full-range encoding** (§2.6) that reduces the A2 cross-label constraint from $O(E^2)$ to $O(E+L)$ and the A3 within-label constraint from $O(E_\ell^2)$ to $O(D_\ell^2+E_\ell)$, ≈14× faster encoding setup and ≈2× faster full-range time on real graphs, with no per-type regression (§3.3).
- An **honest exponential reference** (§2.5) rebuilt into a real decision procedure (used here as auxiliary ground truth), and a **string-preserving repair module** (§2.7) that unfolds any non-Wheeler DAG into a Wheeler graph preserving its path-string set exactly (§3.4).
- A **scalability characterization** (§2.8, §3.2) of every path's size limit under a fixed budget, v1.0.0 versus current.

## 2. Methods

### 2.1 The recognizer pipeline

Wheelie parses a DOT graph into an internal digraph and runs the three-step pipeline of Table 1 (`recognizer/src/`). Steps 1–2 are the heuristic; step 3 is the solver, which is interchangeable between the SMT and permutation backends and, in full-range mode (`-f`), always uses SMT with the step-2 range narrowing disabled.

**Table 1.** The Wheelie recognition pipeline.

| Step | Stage | Purpose |
|---|---|---|
| 0 | Parse | Read the `strict digraph` DOT, build the internal digraph from `A -> B [label=x]` edges. |
| 1 | `relabel_initialization` | Impose the axiom-implied initial constraints (in-degree-0 prefix; per-label head grouping). If they are already violated, reject — the graph is not a Wheeler graph. |
| 2 | `innodelist_sort_relabel` | The renaming heuristic: iteratively sort and relabel nodes by incoming-edge structure, narrowing each node to an order window $[\mathrm{lb},\mathrm{ub}]$. Under `-f`, the range-narrowing branches are disabled. |
| 3 | Solve | Search within the windows: **SMT** (`solve_smt`, Z3 `QF_IDL`; the default, now lazy/CEGAR) or **permutation** (`permutation_start`, pruned enumeration with `WG_checker`). |
| 4 | Verdict | Exit code $1$ = Wheeler, $-1$ = not a Wheeler graph, $0$ = undecided. Benchmark mode (`-b`) prints `verdict, nodes, cpu, path`. |

### 2.2 The brute-force oracle

Ground truth is established by a deliberately trivial procedure (`verify/brute_oracle.py`), written independently of the recognizer so that the two cannot share a bug. For a graph with $n$ nodes it enumerates all $n!$ node orderings; for each, it checks A1–A3 directly against the edge list and accepts iff some ordering satisfies all three. Its cost,

$$
T_\text{oracle}(n,e) = O(n!\cdot e^2),
$$

confines it to $n \leq 9$, but on those graphs it is an incontrovertible decision: a completely separate implementation computing the same predicate from first principles. (One subtlety the oracle itself had to get right: a single node with two differently-labeled self-loops violates A2, so only the empty graph, not every one-node graph, is a trivial accept.)

### 2.3 Differential-testing harness and corpora

The harness (`verify/difftest.py`, `verify/edgecases.py`) draws graphs from many families, runs each through both the oracle and every recognizer backend, and flags any verdict that differs. Table 2 summarizes the corpora used across this report.

**Table 2.** Evaluation corpora.

| Corpus | Size | Role |
|---|---:|---|
| Reject-heavy random (non-Wheeler), oracle-checked | 3,839 + 2,626 (dense) | Stress the accept paths: count false-accepts, v1.0.0 vs current (§3.1). |
| Random + positives, both binaries vs oracle | ~18,000 | Verdict agreement / zero false-accepts & false-rejects (§3.1). |
| Edge cases (self-loops, cycles, parallel edges, …) | 16 | Hand-checked structural cases, both binaries (§3.1). |
| Real DNA/AA De Bruijn / reverse-deterministic | 900 | Full-range `-f` timing, v1.0.0 vs current (§3.3). |
| Synthetic `complete` / `dnfa` families | ladder | Scalability and the default-path ceiling (§3.2). |
| Random DAGs | 820 | Repair (§3.4). |

Each family is Wheeler or non-Wheeler by construction or by oracle anchoring, so every rung doubles as a correctness test as well as a timing or scaling measurement.

### 2.4 Soundness fixes (v1.0.0 → current)

Differential testing against the oracle surfaced several defects in **v1.0.0**'s *accept* paths — the places where a verdict of "Wheeler" is issued — each fixed in the current version so that an accept is never issued without a satisfied check (these are the source of v1.0.0's false-accepts in §3.1):

- **Reject on permutation exhaustion.** The permutation backend (`permutation_start`) issued its verdict only on the accept branch; on a fully pruned, exhausted search it returned without a decision and the caller defaulted to "Wheeler." It now rejects explicitly when a non-exhaustive search completes with no valid order.
- **Re-check the propagation fast path.** When step-2 propagation alone settled every node to a singleton window, the result was accepted without the final `WG_checker` the SMT and permutation paths run. The fast path now verifies A1–A3 before accepting.
- **Overflow and degeneracy guards.** An integer overflow in the solver-dispatch heuristic (a product of per-window factorials) was made saturating; and the SMT path now guards degenerate inputs — an empty or isolated-node graph that produced an empty `z3::distinct` — that previously aborted.

These are corrective, not algorithmic: they change a verdict only on inputs the un-hardened code would have decided incorrectly or crashed on, and they leave every already-correct verdict unchanged. The exponential reference rewrite (§2.5) also corrected the oracle's own one-node shortcut noted in §2.2.

### 2.5 An honest exponential decision procedure

The bundled reference implementation of the Gibney–Thankachan algorithm [3] was, as originally written, a timing baseline: its checker received only scalar counts rather than the graph's edges, so it could not test the axioms and never emitted a definite non-Wheeler verdict. To make it a trustworthy independent second opinion, its decision core was reimplemented (`benchmark/exponential_recognizer/`) to enumerate node orderings and verify A1–A3 against the real edges — mirroring the oracle of §2.2 — under an $O(n!\cdot e^2)$ work budget, returning a clean three-way verdict:

$$
\text{verdict} = \begin{cases}
\text{Wheeler} & \text{some ordering satisfies A1--A3},\\
\text{not a Wheeler graph} & \text{no ordering does, within budget exhausted},\\
\text{undecided} & \text{work budget exceeded before exhaustion}.
\end{cases}
$$

It is now usable as a decision procedure on small graphs and as a second independent check against the recognizer, not merely a stopwatch.

### 2.6 A sparser full-range SMT encoding and a lazy default solver

In full-range mode the recognizer hands the entire ordering space to Z3: one integer order variable per node, a global `distinct`, and the axiom constraints, solved as a quantifier-free integer-difference-logic (`QF_IDL`) problem [4]. The cost is dominated by the *number of axiom constraints*, and the original encoding wrote them pairwise.

**Cross-label monotonicity (A2).** Writing A2 as an all-pairs constraint over edges costs $O(E^2)$. The sparse form introduces, per label, head-boundary variables ($\#\mathrm{lo}_\ell,\#\mathrm{hi}_\ell$) that bracket the order positions of that label's heads, and chains the brackets across labels in label order. This expresses the same "smaller label ⇒ earlier head" relation in

$$
O(E^2) \;\longrightarrow\; O(E+L),
$$

where $L$ is the number of distinct labels.

**Within-label ordering (A3).** For a single label $\ell$ with $E_\ell$ edges, the pairwise A3 constraint costs $O(E_\ell^2)$. The sparse form encodes A3 as an endpoint block: with $D_\ell = \min(\#\text{distinct tails},\,\#\text{distinct heads})$ for that label, per-endpoint min/max position variables ($\#\mathrm{mn},\#\mathrm{mx}$) replace the all-pairs comparison, giving

$$
O(E_\ell^2) \;\longrightarrow\; O(D_\ell^2 + E_\ell).
$$

Both forms are **equisatisfiable** with the pairwise originals — they accept and reject exactly the same graphs — so correctness is unaffected; only the encoding size changes.

**A shape-aware guard.** The block form trades pairwise difference atoms for $2D_\ell$ auxiliary integer variables, and those auxiliaries enlarge Z3's search. Gating the block on atom count alone (fewer atoms) is insufficient: on dense, few-label instances the auxiliaries can make the *solve* slower even when the encoding is smaller. The block is therefore applied to label $\ell$ only when

$$
2D_\ell < E_\ell \quad\wedge\quad D_\ell(D_\ell-1)+2E_\ell < E_\ell(E_\ell-1),
$$

i.e. only when the label's endpoints are genuinely sparse (the first clause; the De Bruijn regime, $D_\ell/E_\ell \approx 0.25$) *and* the block emits fewer atoms (the second). Otherwise the verified pairwise loop is used. Because the two encodings are equisatisfiable, the guard changes only *which* encoding is used, never the verdict.

**Lazy generation of the within-label axiom (the default solver).** The encoding changes above shrink the full-range formula but do not move its size ceiling (§3.2). The decisive change is on the *default* path, where the step-2 heuristic supplies tight order windows: there the current version replaces the up-front A3 encoding with **lazy, counterexample-guided generation** (`recognizer/src/lazy_solve.cpp`, the default; `-s smt` selects v1.0.0's vanilla encoding). A3 is a *chain* constraint — within a label group, heads are monotone non-decreasing in tail order — and is $O(E_\ell^2)$ only because the tail order is unknown a priori. So the solver asserts the window and `distinct` constraints with **no A3**, asks Z3 for a model, re-checks A3 against that concrete order in $O(E\log E)$, adds only the $O(E)$ consecutive-pair constraints the model actually violates, and re-solves — monotonically, so Z3 keeps its learned clauses across rounds (a lazy-SMT / CEGAR loop). Z3 never materializes the quadratic formula; it sees only the constraints that bind, which the heuristic's windows make a tiny fraction of the whole (§3.2). The loop is **sound by construction**: it accepts only a model that passes the same final `WG_checker` the other backends use (with the emitted order independently re-validated), it rejects only when the partial constraint set is already unsatisfiable, and on a budget it falls back to the full Z3 solve — so a verdict can never regress relative to v1.0.0's. This is the headline of §3.2.

### 2.7 String-preserving graph repair

Repair (`repair/wheelerize.py`) turns a non-Wheeler **DAG** into a Wheeler graph by unfolding it into the **trie of its path-strings**. A trie is always a Wheeler graph — order its nodes by the co-lexicographic rank of the string spelled along each incoming path — so the construction always succeeds for a DAG, and it preserves the **set of strings** the graph spells, exactly. Each repair self-verifies three independent ways before it is returned: the recognizer accepts the output, the oracle accepts it when it is small enough, and the set of path-strings is confirmed identical to the input's. The node-count ratio (output / input) can fall below one — when distinct input nodes spell the same string they are *merged* — which is a reduction in nodes, not a loss of represented strings; ratios above one are genuine node splitting. This first version is correct but not minimal; minimal node-splitting is left to future work, and cyclic graphs (where a string-preserving repair is not always possible) are out of scope.

### 2.8 Benchmark design and the three-axis methodology

The cardinal rule of this evaluation is that three questions are reported **separately** and never merged into a single "X% better":

- **Correctness** — does the verdict match the independent oracle? Measured as false-accepts on the reject-heavy corpus and verdict agreement on the oracle-decidable and synthetic corpora.
- **Capability** — can a verdict be produced at all? Measured as the largest graph a path decides within its budget, and the fraction of a corpus the rebuilt exponential reference decides, independent of speed.
- **Performance** — how fast, and how large, when correct? Measured as full-range setup/solve time, the per-graph timing distribution, peak memory, and the largest graph each backend decides under a fixed budget.

A faster wrong answer is not progress, an undecided result is not a refutation, and additional memory is a cost to be stated, not hidden. The full-range timing uses string labels (matching the biological corpora), the median of replicate runs, and the `-b` CPU-time column, which is robust to machine load; timeouts are recorded as timeouts, never as a verdict.

**The scalability sweep** (`benchmark/limit_test/`, `benchmark/lazy_cegar/`) climbs a geometric size ladder per (backend, family) under a **600 s** wall-clock budget, takes the median of $R=3$ replicate graphs per rung, and bisects the largest size still decided. Limits are **typed**: *threshold* (timeout-bounded — the backend would continue with more time), *capped* (a hard capability ceiling — the exponential reference's $n!$ budget, or Z3 returning `unknown`), or *unreached* (no wall hit up to the ladder top). The two families are `complete` (dense, all-distinct endpoints — the symmetric worst case) and `dnfa` (random $d$-NFA-style), both Wheeler by construction so each rung is also a correctness check.

**Environment and reproducibility.** Comparisons are run on a single Linux host with the recognizer built from source (g++ 17, Z3 linked statically). The recognizer, oracle, harnesses, and all source data and figures are versioned in the repository; every number in this report is recomputed from the committed data by `benchmark/report_figs/verify_report_numbers.py` (an all-PASS check), with figure data under `benchmark/report_figs/data/`, `benchmark/lazy_cegar/`, and `benchmark/limit_test/results/`.

### 3.1 Axis I — Correctness: agreement with an independent oracle

**Result:** v1.0.0's permutation backends accept **770** (simple corpus) and **1,147** (dense corpus) non-Wheeler graphs as Wheeler; the current version accepts **none**, on every backend, with **zero false-rejects** over ~18,000 oracle-checked graphs.

Against the brute-force oracle, the current recognizer is sound on every graph both can decide; the v1.0.0 release is not. On a reject-heavy corpus of **3,839 non-Wheeler graphs** — generated to exercise the accept paths and oracle-checked at $n\leq7$ — v1.0.0's permutation backend declares **770** of them Wheeler, and its exhaustive permutation mode the same 770, while its default-SMT and full-range paths are correct (Table 4). A second, denser corpus (2,626 non-Wheeler graphs with self-loops and parallel edges) widens the gap to **1,147** v1.0.0 false-accepts. The current version produces **zero** false-accepts on every backend, on both corpora.

### 3.2 Axis II — Capability and the size ceiling

**Result:** the default-path size ceiling — the path a user gets with no flags — moves from v1.0.0's **2,816** `complete` / **2,176** `dnfa` nodes to **≥32,768** on both families, **≈554× faster and ≈133× lighter** at a matched 2,816 nodes, by generating the within-label axiom on demand (just **3,835 of 402,722,292** A3 constraints at $n=32{,}768$). The mechanism is a lazy/CEGAR SMT solver, now the default.

Under a 600 s budget v1.0.0's default solver (vanilla Z3 plus the renaming heuristic) decides `complete` graphs of **2,816** nodes and `dnfa` graphs of **2,176**, its limit a timeout — at $n=4{,}096$ it does not finish in ten minutes, and given a full hour reaches only 4,608/3,584. The current version's default is the new lazy/CEGAR solver, and it decides **at least 32,768** nodes on both families — the largest size on our test ladder, not a wall the solver hit — in under a minute (Table 5, Fig. 3). That is a **more-than-tenfold** increase in the size of worst-case graph the recognizer decides, on the path almost everyone runs, and it comes entirely from the solving algorithm, not faster hardware. Every rung is Wheeler by construction and was decided correctly by both versions where they finished: the sweep recorded **zero correctness failures** end to end.

**Why the wall was there, and how it moves.** The default-path ceiling is not a hardware limit or a search explosion. Profiling Z3's own statistics on these symmetric worst cases shows it solves with **near-zero search** (tens to hundreds of SAT conflicts even at $n=1{,}000$) while its memory allocations explode roughly **12,000-fold** over the same range. The wall is the cost of building and propagating an $O(n^2)$ difference-logic formula that grows with the graph — specifically the within-label A3 axiom, which v1.0.0 (and any vanilla encoding) materializes in full as one implication per pair of same-label edges.

The current version's default solver removes that wall by *not building the formula*. A3 — within a label group, heads are monotone non-decreasing in tail order — is a **chain** constraint, and it is $O(n^2)$ only *statically*, because the tail order is unknown a priori. Given a concrete candidate order the tail order is *known*, so a violation is forbidden by just $O(E)$ consecutive-pair constraints. So the solver runs a counterexample-guided loop (lazy SMT / CEGAR): assert the heuristic's bracket and all-different constraints but **no A3**, ask Z3 for a model, re-check A3 in $O(E\log E)$, add only the few consecutive-pair constraints the model actually violates, and re-solve — monotonically, so Z3 keeps its learned clauses across rounds. Z3 never materializes the $O(n^2)$ formula; it only ever sees the handful of A3 constraints that bind. Because the renaming heuristic's brackets already nearly determine the order, that handful is tiny: deciding a 32,768-node `complete` graph builds just **3,835 of its 402,722,292** A3 pairs — the full encoding is ~99.99% redundant.

The effect on the default path is dramatic (Table 6). Where v1.0.0's vanilla solver needs **488 s and 7.3 GB** to decide a 2,816-node `complete` graph and then times out at 4,096, the current lazy default decides 2,816 in **0.9 s and 56 MB** and climbs to **≥32,768 nodes in under a minute** — roughly **554× the speed at 1/133rd the memory**, and a >10× higher size ceiling bounded by our test ladder, not by the solver. It is verified sound by the same oracle differential as every other result (zero disagreements over ~13,000 random and constructed graphs and every edge case, each emitted order independently re-validated), and whenever it cannot decide within budget it falls back to the full Z3 solve, so a verdict can never regress. `-s smt` still selects the original vanilla encoding, which is exactly v1.0.0's default.

Two contrasts sharpen the result. First, *time alone is no substitute for the algorithm*: given a full hour rather than 600 s, v1.0.0's vanilla default reaches only **4,608** `complete` / **3,584** `dnfa` nodes — a 1.64× return for 6× the budget (the sub-linear scaling expected when solver cost grows steeply with $n$), still an order of magnitude below what the lazy default does in under a minute. Second, *parallelism is no substitute either*: at scale the recognizer's wall time is almost entirely a single Z3 `QF_IDL` solve, and a prototype with Z3's parallel mode showed no measurable gain on these families — "parallelize the recognizer" means "parallelize one `QF_IDL` query," which it does not.

**The full-range ceiling, by contrast, does not move — and that is stated plainly.** The full-range (`-f`) path's limit deserves a precise reading, because it bounds what §3.3 claims. Both v1.0.0's dense `-f` encoding and the current sparse one stop at **832** nodes on both synthetic families — and not by timeout but because Z3 returns `unknown` on the `QF_IDL` problem at that size. That ceiling is a property of the solver on this encoding, the **same** for both versions; the sparser encoding **lowers the time curve, not the size wall** (§3.3). It is the *unflagged default* path, where the lazy solver applies and the heuristic supplies the brackets, where the wall actually moves.

**A verification aside on capability.** The toolkit's exponential reference — used here only as an auxiliary ground truth, not as a recognizer — was also rebuilt into an honest axiom-checking decision procedure (§2.5): on the oracle-decidable subset (931 graphs of at most nine nodes) it now returns a verdict on all 931 (the original implementation exceeded its budget on 159, and never produced a non-Wheeler verdict at all), including seven non-Wheeler graphs built from real human gene orthologs (BTBD17, FAM53A, LCP1, TRAM1, TRPC1) it could not previously classify.

### 3.3 Axis III — Performance: the full-range encoding, in atoms then in time

**Result:** the sparser full-range (`-f`) encoding is **≈14× cheaper to build** and **1.3–1.9× faster end-to-end** on the real biological corpora, with **no per-type regression**, at an honest **~1.8× peak-RAM** cost only where the within-label block fires.

Before any wall-clock number, the sparsification can be measured *directly* — as the size of the SMT problem it builds. `atom_count.py` computes, analytically from each graph's label and endpoint structure, exactly how many `s.add(...)` assertions each generation emits under `-f`, mirroring the encoder line for line. The count is **exact**, not a model: on a calibration set it equals Z3's own `s.assertions().size()` from an instrumented build to the atom. Fitting the counts over all 900 biological graphs against the edge count $E$ gives clean power laws (Fig. 4): v1.0.0's encoding is **$\propto E^{2.0}$ on every graph type** — the dense all-pairs baseline — while the current encoding is genuinely **sub-quadratic only where the within-label block fires** (few-label De Bruijn DNA, $\propto E^{1.6}$). On twenty-letter amino-acid alignments the block stays off and the gain is instead a large **constant factor** from the cross-group sparsification, with the exponent near $E^{1.8}$ — an honest distinction the fit makes visible. Across all 900 graphs the formula is a median of **13.4× smaller** (per-type 3.8–15.6×; up to ~300× on the largest many-label graphs).

The decisive caveat is that **atoms are not wall time**. The formula shrinks by a median of about thirteen-fold, and the *encoding setup* shrinks in step (≈14×, below). But the **total** full-range time falls only ≈2×, because Z3's *solve* is the bottleneck and its cost is not proportional to the atom count — a smaller formula helps the solver but does not shrink the NP-hard search by the same factor. The sparsification's first-order effect is to make building the problem nearly free; the solve speedup is a real but second-order benefit.

### 3.4 Practicality and repair on real data

**Result:** every one of 500 Wheeler graphs built from real Ensembl gene alignments is decided in **under a second**, and any non-Wheeler DAG repairs into a Wheeler graph that spells the same strings (**316 of 316**, zero failures).

The questions above use synthetic worst-case families; this section closes the loop to biology. WGT ships three MSA-to-graph constructions — a De Bruijn $k$-mer graph, a reverse-deterministic column automaton, and a trie — wrapped in a single turnkey CLI (`pipeline/fasta_to_wg.py`: FASTA alignment in; graph built, recognized, and timed). We ran all three over 50 human gene orthologue alignments retrieved from Ensembl (DNA and amino-acid, sequence length capped for the De Bruijn ladder), 500 graphs in all (Table 8, Fig. 6A–B).

Two results stand out. First, **construction choice all but determines the Wheeler verdict**: De Bruijn $k$-mer graphs and tries are Wheeler on every real alignment we tried (tries provably so — a prefix tree is a canonical Wheeler graph), whereas the reverse-deterministic column automaton is Wheeler only ~1% of the time, so it is the natural candidate for the repair path below. Second, **recognition is instant in practice**: every one of the 500 real graphs was decided in well under a second, and the largest (2,399 nodes) sits far below even the 600 s ceiling. The multi-thousand-node hour-long runs are a property of adversarial synthetic graphs; for the graphs molecular biology actually produces, the heuristic-plus-SMT recognizer is effectively real-time.

**Repair turns a "no" into an indexable graph.** Across 820 random DAGs, the repair module (§2.7) succeeded on every graph that needed it: **316** required repair and all 316 produced a verified Wheeler graph, **504** were already Wheeler, and there were **zero** verification failures (Fig. 6C). Because every output passes all three self-checks — recognizer, oracle (when small), and exact path-string-set equality — repair is a sound constructive complement to recognition: a "not a Wheeler graph" verdict becomes a concrete Wheeler graph that spells the same strings.

## 4. Discussion

**More correct, more capable, and faster on the path users run — and we state precisely what it costs.** That is the whole result, and the three-axis discipline (§2.8) is what makes it defensible rather than a slogan: a real per-graph speedup (§3.3) reported alongside, not instead of, the facts that the full-range size ceiling does not move (§3.2) and that the block costs memory on tractable graphs (§3.3). Each of those is a true statement a single headline figure would have blurred.

**Correctness is the load-bearing result.** The most consequential change in this pass is not the faster encoding or the new repair capability — it is that the recognizer's verdicts are now validated against an independent, obviously-correct oracle and a trustworthy second decision procedure. A recognizer is an oracle other research builds on; if it can silently accept a non-Wheeler graph, every downstream claim that rests on "this is a Wheeler graph" is unsupported. Establishing zero false-accepts and full oracle agreement (§3.1) is therefore the prerequisite that makes the speed and capability results worth reporting at all.

**Trades, stated plainly.** Three costs are volunteered, not extracted, because a sophisticated reader will look for them. (i) The full-range `-f` size ceiling is **unchanged** — both encodings stop at $n=832$ where Z3 returns `unknown` (§3.2); the lazy win is confined to the default path, which is exactly where the heuristic supplies the brackets the lazy loop exploits. (ii) The sparse `-f` encoding costs **~1.8× peak RAM** on the low-$D/E$ graphs where the within-label block fires (§3.3) — a space-for-time trade, not a uniform improvement. (iii) Recognition **remains NP-complete**, hard already at a two-letter alphabet even for DAGs; no encoding makes the general problem polynomial, and the `complete` family is where every backend is slowest. Naming these is what lets the wins be read at face value.

**The heuristic is the engine.** The scalability separation (§3.2) is a quantitative restatement of WGT's core design insight: the renaming heuristic, by narrowing each node's window before the solver runs, is what lets the default backend scale orders of magnitude past the no-heuristic full-range path and the exponential reference. The improvements here sharpen the periphery — a leaner full-range encoding, a lazy default solver, a trustworthy reference, repair — around a method whose central idea is unchanged and remains the reason recognition is practical at all.

**Verification as software practice.** The brute-force oracle and differential harness are cheap to run and decisive on small graphs, and they are the reason every claim in §3.1 is checkable rather than asserted. The same discipline that surfaced the accept-path edge cases (§2.4) is directly transferable to other combinatorial recognizers: build a slow, obviously-correct reference, then differential-test the fast implementation against it across many generated inputs.

**Recognition is not the bottleneck in practice — so the lever is upstream.** Every Wheeler graph built from a real gene alignment was recognized in under a second, far inside even the 600 s ceiling, and parallelism does not move that ceiling because the cost is a single `QF_IDL` solve (§3.2). The practical lever is therefore which *construction* one chooses, which determines whether the graph is Wheeler at all (§3.4); and when it is not, repair (§3.4) turns that "no" into an indexable graph spelling the same strings. For realistic pangenome-scale inputs the open problem is graph *construction and size*, not recognition speed.

## 5. Limitations

The study and tool are presented with the following honest caveats, the first two of which bound the headline result and are stated in §3 and §4 as well:

1. **The full-range ceiling is unchanged.** The sparse encoding lowers per-graph time but not the size at which the full-range path can decide: both the new and prior encodings stop at $n=832$ where Z3 returns `unknown` (§3.2). Raising that ceiling is a solver-encoding problem left open; the default-path lazy win does not extend to `-f`, where the heuristic brackets are absent.
2. **A memory cost on tractable graphs.** The sparse encoding's auxiliary variables raise full-range peak memory by roughly 1.8× on the low-$D/E$ graphs where the within-label block fires (§3.3); it is a space-for-time trade, not a uniform improvement. (On the default path the lazy solver is the opposite — far lighter than v1.0.0.)
3. **Recognition remains NP-complete.** None of this changes the worst case. The dense, all-distinct-endpoint `complete` family is where every backend is slowest, and where the full-range block is correctly disabled. The lazy default wins by exploiting the heuristic's brackets, not by beating the hardness.
4. **Oracle reach.** The brute-force oracle is exact but $O(n!\cdot e^2)$, so ground truth is direct only for $n\leq9$. Correctness on larger graphs rests on cross-backend agreement and construction-by-design families, not on the oracle.
5. **Repair is not minimal and is DAG-only.** The trie unfolding always succeeds for a DAG and preserves the string set, but the result is not a minimum-node Wheeler graph; cyclic graphs, where a string-preserving repair need not exist, are out of scope.
6. **The per-type speedup covers paired jobs only.** The §3.3 breakdown uses the jobs with a comparable finite time from *both* versions; the excluded rows are the largest amino-acid instances where v1.0.0's dense encoding exceeds the timeout (so no finite ratio exists). Those omissions are overwhelmingly cases where v1.0.0 itself times out, so the table is a conservative per-type summary rather than an optimistic subset.

## 6. Availability and reproducibility

WGT is open source ([github.com/Kuanhao-Chao/Wheeler_Graph_Toolkit](https://github.com/Kuanhao-Chao/Wheeler_Graph_Toolkit)); the verification and improvement work described here is on the `devel` branch. The recognizer builds from source against Z3 (`recognizer/`); on Linux it is built directly with g++ linking Z3 statically (the committed `recognizer/bin/recognizer` is a macOS binary and must be rebuilt on other platforms). A typical recognition run is:

```
./bin/recognizer graph.dot          # default: heuristic + lazy/CEGAR SMT (Z3)
./bin/recognizer graph.dot -s p     # permutation backend
./bin/recognizer graph.dot -s smt   # v1.0.0's vanilla (non-lazy) SMT encoding
./bin/recognizer graph.dot -f       # full-range SMT (no heuristic narrowing)
```

The brute-force oracle and differential harness live under `verify/` (`brute_oracle.py`, `difftest.py`, `difftest_exp.py`, `edgecases.py`, `check_order.py`); the repair module under `repair/` (`wheelerize.py`); the rebuilt exponential decision procedure under `benchmark/exponential_recognizer/`; and the full benchmark — the three-axis sweeps, the scalability limit test, the lazy/CEGAR ceiling, and the figure-generation scripts — under `benchmark/`. Every number in this report is recomputed from the committed data by `benchmark/report_figs/verify_report_numbers.py` (an all-PASS check), with per-figure data under `benchmark/report_figs/data/`, `benchmark/lazy_cegar/`, and `benchmark/limit_test/results/`. Comparisons were run on a single Linux host with the recognizer built from source and Z3 linked statically; the scalability sweep used a 600 s per-graph budget with three replicates per size.
