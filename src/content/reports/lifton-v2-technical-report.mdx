## 1. Introduction

### 1.1 The annotation-transfer problem

A genome assembly without an annotation is a sequence without a map. Knowing where genes, transcripts, and coding sequences (CDS) lie is the prerequisite for variant interpretation, differential expression, comparative genomics, and essentially every downstream genomic analysis. Yet de-novo curation does not scale: long-read sequencing and the T2T/pangenome era produce finished assemblies far faster than experimental annotation pipelines can characterize them. The pragmatic alternative is **annotation lift-over** — projecting a well-curated reference annotation $R_A$ from a reference genome $R$ onto a newly assembled target genome $T$ to produce $T_A$.

Two families of evidence dominate lift-over, and each fails in a characteristic way:

- **DNA-level mapping** (Liftoff [2], driven by minimap2 [4]) aligns the reference gene loci, with their exon structure, directly to the target DNA. It is fast and exact when $R$ and $T$ are close, but as the sequences diverge it accumulates substitutions, small indels, and assembly-boundary artifacts that manifest as frameshifts, premature stop codons, and incorrect splice boundaries in the projected CDS.
- **Protein-level mapping** (miniprot [3]) aligns the reference *proteins* to the target genome with a splice-aware, frame-aware model. Because protein sequence is conserved well past the point where DNA alignment degrades, miniprot survives divergence; but it produces fragmentary and sometimes mutually inconsistent gene models, and its identifiers do not correspond to reference transcript IDs.

Neither signal alone is trustworthy across the full divergence range encountered in practice.

### 1.2 LiftOn and the dual-evidence approach

LiftOn [1] resolves this by *combining* both signals. For each candidate gene it obtains a DNA-based model (from Liftoff) and a protein-based model (from miniprot), aligns each candidate's translated protein back to the reference protein, scores the alignments, and runs a **protein-maximization CDS chaining** algorithm that stitches the best-supported coding segments into a single coherent gene model, followed by an ORF-rescue pass that repairs CDS boundaries. The published method [1] demonstrated that this fusion recovers more complete, in-frame protein-coding annotations than either Liftoff or miniprot alone, with the advantage greatest where the target diverges from the reference.

### 1.3 What motivated v2.0.0

The published LiftOn implementation (v1.0.8) was correct but was engineered for representative-chromosome-scale inputs. Applying it at whole-genome scale exposed four limitations that v2.0.0 addresses:

1. **Unbounded alignment memory.** The two parasail [5] base aligners call full Needleman–Wunsch with traceback (`nw_trace_scan_sat`), which is $O(L^2)$ in *both* time and memory. A single titin-scale alignment (transcript ≈ 106 kb) requires a traceback matrix of $\approx 1.1\times10^{10}$ cells — measured at ~48 GB peak resident memory and ~1264 s of wall time on a mammalian genome — and out-of-memory crashes on typical hardware.
2. **Full-genome crashes.** On full RefSeq *Arabidopsis* and rice, v1.0.8 terminates with an unhandled `gffutils.FeatureNotFoundError` whose error handler itself raises `TypeError: __str__ returned non-string`, aborting the run with a partial annotation.
3. **Single-threaded execution and disk round-trips.** v1.0.8 ran the per-locus merge loop serially, ran the two external aligners sequentially, and round-tripped large intermediate GFF3/FASTA files through disk and a re-ingested SQLite database.
4. **Coding-only lift.** v1.0.8 lifted only the protein-coding `gene` hierarchy, silently dropping pseudogenes, non-coding RNA genes, and other structured features present in the reference.

### 1.4 Contributions

This report documents the algorithmic and architectural advances in LiftOn v2.0.0 and validates them on seventeen whole-genome, end-to-end lift-overs spanning a four-tier divergence ladder (§2.8). The principal contributions are:

- An **anchor-windowed alignment kernel** that reduces alignment memory from $O(L^2)$ to a bounded $O(b^2)$ and is exact (identical to full DP) wherever unique-$k$-mer anchors exist or a divergent region is below the giant boundary (§2.2).
- A **best-of-outcome protein-maximization merge** that adopts the merged dual-evidence model only when it scores strictly higher than the DNA-only model against the reference protein, converting the merge from a net-risky operation into a guaranteed per-transcript non-regression (§2.4).
- **Gene-like feature autodetection** that lifts every reference parent type bearing a transcript/exon hierarchy — pseudogenes, ncRNA/lnc_RNA genes, structured mobile elements — by default (§2.6).
- A re-engineered execution engine (concurrent aligners, a fused per-locus thread pool, streaming extraction, in-process aligner I/O, query collapse) governed by a **24-cell byte-identity regression matrix** that guarantees these optimizations leave the output bytes unchanged (§2.7).
- Robustness fixes that let v2.0.0 complete full RefSeq genomes v1.0.8 abandoned (§2.7, §3.4).

## 2. Methods

### 2.1 Pipeline overview

LiftOn v2.0.0 executes a single, strictly sequential 11-step pipeline (`lifton/lifton.py`, `run_all_lifton_steps`). Table 1 lists the steps in order.

**Table 1.** The LiftOn v2.0.0 pipeline.

| Step | Stage | Purpose |
|---|---|---|
| 0 | Setup | Open target + reference FASTAs (`pyfaidx`); create output directories. |
| — | Input validation | Validate the reference annotation (NCBI GFF3 spec); under `--strict-gff`, exit non-zero on any violation, else write findings to `stats/` and emit a summary. |
| 1 | Reference DB | Build the reference annotation database (`gffutils` [7] SQLite by default; DuckDB-backed `gffbase` under `LIFTON_USE_GFFBASE=1`). |
| 2 | Partition | Auto-detect gene-like parent types (unless `--gene-only`) and partition the reference into the features to lift (§2.6). |
| 3 | Sequence extraction | Stream `transcripts.fa` + `proteins.fa` to disk, then re-open via `pyfaidx` for lazy access (§2.7). |
| 4 | External aligners | Run vendored Liftoff (minimap2) and miniprot; concurrent by default, `--serial-aligners` to serialize; skippable via cached `-L`/`-M` (§2.7). |
| 5 | Load + writers | Re-load the Liftoff and miniprot GFF3s as feature DBs; open output writers; emit the GFF3 directive prologue. |
| 6 | Index | Build the reference-id ↔ miniprot-id map; seed per-chromosome interval trees for miniprot suppression. |
| 7 | Per-locus merge | Walk each Liftoff gene → align with parasail → chain with miniprot (§2.3) → ORF-rescue (§2.5) → write. Parallel under `--threads N --locus-pipeline` (§2.7). The wall-clock hot spot. |
| 8 | miniprot-only rescue | Emit miniprot-only mRNAs that pass the overlap/length filters and are not suppressed. |
| 9 | Report | Emit completeness/coverage statistics. |
| 10 | Output validation | Under `--validate-output`, re-validate the emitted GFF3 (hierarchy, CDS phase, containment). |

The two largest external subtrees — the vendored Liftoff fork (the DNA lift-over engine, invoked as a library) and `gffbase` (a DuckDB-backed, drop-in successor to `gffutils`) — are treated as frozen dependencies.

### 2.2 Anchor-windowed alignment

**The giant-gene problem.** LiftOn scores every candidate by globally aligning its translated protein (and, for non-coding features, its spliced transcript) to the reference with parasail's `nw_trace_scan_sat`. Full Needleman–Wunsch with traceback is

$$T_\text{full}(L) = O(L^2), \qquad M_\text{full}(L) = O(L^2),$$

in time and memory respectively, where $L$ is the longer of the two sequences. The quadratic *memory* term is the binding constraint: the traceback matrix for a titin-scale transcript ($L \approx 1.06\times10^5$ nt) holds $\approx 1.1\times10^{10}$ cells, measured at ~48 GB peak and ~1264 s on a mammalian genome — almost all of it concentrated in a handful of giant genes — which out-of-memory-crashes any machine under ~48 GB.

**The windowed kernel** (`lifton/windowed_align.py`). Above a length gate, alignment is routed to an anchor-windowed aligner that computes a near-diagonal global alignment in bounded memory:

1. **Anchor seeding** ($O(L)$). Enumerate $k$-mers that are *unique in both* sequences; these are reliable diagonal anchors. The protein alphabet (20 letters) makes $k_{\text{AA}}=8$ unique within any realistic protein; DNA (4 letters) requires a longer seed, $k_{\text{NT}}=15$ (since $4^{15}\approx10^9$ exceeds any transcript length). When a window is anchor-sparse, $k$ is shrunk toward a floor ($k_{\text{AA}}\!\geq\!4$, $k_{\text{NT}}\!\geq\!8$) to recover finer anchors.
2. **Co-linear chaining** ($O(a\log a)$, $a\leq L$ anchors). Chain the shared anchors into one strictly co-linear, non-overlapping diagonal via a longest-increasing-subsequence (patience-sorting) pass.
3. **Per-window DP and concatenation.** Align only the small inter-anchor windows with full parasail and concatenate; the anchors themselves contribute exact-match columns. Each window is capped at a band width $b$, so any single DP allocation is at most $b^2$ cells.

The result is a thin object that is drop-in compatible with the parasail result LiftOn consumes (it exposes `.traceback.{query,comp,ref}` and a `.cigar` following parasail's `=`/`X`/`I`/`D` convention), so no downstream consumer changes. The complexity becomes

$$T_\text{win}(L) = O\!\left(L\,b\right), \qquad M_\text{win}(L) = O\!\left(b^2\right),$$

i.e. memory is **independent of $L$**.

**Gating ("band everything").** v2.0.0 makes windowing the *default for all sizes*, not just giants. The gate is low — $2500$ aa / $8000$ nt — with a fine band cap $b = 1500$, and the exact-DP fallback for anchor-less regions is raised to the giant boundary ($8000$ aa / $25000$ nt). The consequence is that windowing is *exact* (bitwise identical to full DP) wherever anchors exist (the common homologous case) or a divergent region is below the giant boundary; only true giant, anchor-less regions take the bounded approximate split. On real titin this reproduces full-DP identity to $\Delta = +0.00000$ at $82\times$ the speed. Tiny genes (below the gate) retain the exact full-DP path. The opt-out flag `--full-dp-align` (or `LIFTON_FULL_DP_ALIGN=1`) restores the exact giant-only path (full DP for every non-giant gene; giants still windowed so it cannot OOM); `LIFTON_ALIGN_WINDOW_{AA,NT}=∞` restores pure full DP for manuscript reproduction.

### 2.3 Protein-maximization CDS chaining

When both a Liftoff model and a miniprot model exist for a transcript, LiftOn chains them (`lifton/protein_maximization.py`, `chaining_algorithm`). Let the Liftoff and miniprot CDS-block lists be $\mathcal{C}_L = (c^L_1,\dots,c^L_{n_L})$ and $\mathcal{C}_M = (c^M_1,\dots,c^M_{n_M})$. The two lists are **synchronized by accumulated reference amino-acid count**: walking both lists, a sync point is a genomic boundary at which both sides have consumed the same cumulative number of reference residues. Between consecutive sync points the two sources cover the *same* reference protein interval, defining a "chunk" $j$. For each chunk the algorithm computes the partial protein identity of each source and keeps the higher:

$$s_j = \operatorname*{arg\,max}_{s\in\{L,M\}} \;\mathrm{id}\!\left(\text{chunk}_j^{\,s}\right),$$

with ties resolved in favor of Liftoff (which preserves the DNA-derived splice structure). The merged CDS list is the concatenation of the chosen chunks. The walk is a single linear pass over both lists with no auxiliary graph, so

$$T_\text{chain} = O(n_L + n_M), \qquad M_\text{chain} = O\!\left(\max(n_L,n_M)\right).$$

Degenerate inputs are handled explicitly: if either side has zero CDS children the Liftoff list is returned as-is; if either side has exactly one CDS block, the synchronization loop is bypassed and the whole alignment is compared as a single chunk.

### 2.4 Best-of-outcome merge

A chained model that improves one chunk can still degrade the *emitted* transcript if the merge perturbs a downstream frame. v2.0.0 therefore adopts the merge **only when it is verified to be at least as good as the DNA-only model** (`lifton/run_liftoff.py`). For each transcript it evaluates two candidates *after* ORF rescue — the same protein the benchmark re-scores — and keeps the better:

$$
\text{emit} = \begin{cases}
\text{(2) chained merge + ORF rescue}, & \text{if } \pi_\text{merge} \geq \pi_\text{liftoff} \\
\text{(1) Liftoff + ORF rescue}, & \text{otherwise}
\end{cases}
$$

where $\pi$ is the emitted protein identity against the reference and ties go to the merge. Because candidate (1) is exactly the result LiftOn would emit with miniprot disabled, enabling miniprot becomes a **strict per-transcript non-regression**: $\pi_\text{emit} = \max(\pi_\text{merge}, \pi_\text{liftoff}) \geq \pi_\text{liftoff}$. The merge candidate is evaluated first so the more expensive Liftoff+ORF candidate can be skipped whenever the merge already wins regardless of it (merge reaches a perfect protein, or the chained CDS is structurally identical to the Liftoff CDS). The opt-out flag `--legacy-merge` restores the pre-v2.0.0 *unconditional* merge (applying the chained CDS regardless of outcome), for exact reproduction of the published algorithm.

### 2.5 ORF rescue and CDS boundary patching

After chaining (or for DNA-only models), an ORF-rescue pass repairs CDS boundaries (`lifton/lifton_class.py`). The spliced transcript is scanned in all three reading frames; per frame the longest ORF is retained, and the resulting candidates (≤3) are each translated and globally aligned to the reference protein. The candidate of **highest identity** is selected, and it is adopted **only if it improves identity by more than 1%** over the current model:

$$\text{adopt } o^\star \iff \mathrm{id}(o^\star) > \pi_\text{current} + 0.01, \qquad o^\star = \operatorname*{arg\,max}_{o}\, \mathrm{id}(o).$$

When adopted, the ORF boundaries are mapped back through the (strand-aware) exon coordinates, adjusting or creating CDS features as needed. The 1% guard prevents cosmetic boundary churn that would not meaningfully improve the model.

### 2.6 Gene-like feature autodetection and lift

v1.0.8 lifted only the protein-coding `gene` hierarchy. v2.0.0 auto-detects and lifts **every** gene-like parent type by default (`lifton/lifton_utils.py`, `get_gene_like_feature_types`). A bounded scan (`sample_cap` = 5000 instances per type) classifies a feature type as gene-like iff at least one **top-level** instance (no `Parent`) carries a child feature — capturing pseudogenes, ncRNA/lnc_RNA genes, tRNA/rRNA/sno/snRNA, and structured mobile elements, while excluding childless meta/regulatory types (region, enhancer, match) and pure sub-feature types (exon, CDS). The detected set is injected into the lift configuration so that *both* LiftOn and the vendored Liftoff lift the same expanded set. Two safeguards preserve coding-gene behavior: (i) gene-like loci are kept *out* of the miniprot-suppression interval tree (seeded with `gene` only), so a pseudogene cannot displace a miniprot coding-gene rescue; and (ii) the output validator accepts exon/CDS nested under a pseudogene. The opt-out flag `--gene-only` restores the coding-only lift for exact reproduction of the published behavior.

The detection predicate is *top-level only*, but the downstream enumeration must apply it per-instance. A gene-like type that also appears as a *child* of a gene (e.g. an `ncRNA` nested under a coding gene) must not be enumerated twice — once via its parent's recursion and once as a top-level locus — or its transcript is double-written, raising a `pyfaidx` duplicate-key error on the full *Arabidopsis* genome. v2.0.0 resolves this by skipping a child instance only when its parent's type is itself in the lift set (`extract_sequence.parent_is_listed_type`), which is byte-identical for any coding-only lift and for every top-level-only annotation (§3.4).

### 2.7 Execution engine and the byte-identity contract

The performance re-engineering is governed by a single discipline: **every optimization that is not intended to change the output must be provably byte-neutral.** The regression oracle is a **24-cell byte-identity matrix** (`tests/test_native_matrix.py`): every combination of `--stream × --inmemory-liftoff × --threads∈{1,2,4} × --native` ($2\cdot2\cdot3\cdot2 = 24$) must reproduce the default output GFF3 byte-for-byte. This decouples the engine from the algorithm — a refactor below the byte-identity line is free; a change above it requires a deliberate, documented output change. The byte-neutral optimizations are:

- **Concurrent external aligners (Step 4).** miniprot (a subprocess that never touches the reference DB) runs on a background worker thread while Liftoff runs on the main thread (it reads the main-thread-bound SQLite reference DB and cannot move off it). Wall time becomes $\max(t_\text{Liftoff}, t_\text{miniprot})$ instead of the sum. Only miniprot's *timing* moves, so the output is unchanged. `--serial-aligners` restores the sequential path.
- **Parallel, fused per-locus merge (Step 7).** Under `--threads N --locus-pipeline`, the per-locus loop is dispatched through a `ThreadPoolExecutor` (threads, not processes: parasail releases the GIL, and feature-DB connections are not picklable). To respect SQLite thread-affinity, each locus is materialized into proxy DBs that worker threads own privately. A *fused* pool has each worker materialize its own locus and then immediately process it, overlapping the SQLite-bound materialize with the GIL-released parasail and avoiding a full-payload barrier. Output is emitted in submission order, so any thread count is byte-identical to `--threads 1`.
- **Streaming extraction and in-process aligner I/O.** `--stream` pipes miniprot's output into an in-memory feature DB and `--inmemory-liftoff` serializes the vendored Liftoff feature list in-process, each skipping a large GFF3 disk round-trip and SQLite re-ingest. `--native` drives minimap2/miniprot through in-process bindings.
- **miniprot thread plumbing.** miniprot's own `-t` now scales with LiftOn's `-t` (gated on `>1` so the default is byte-identical to the prior fixed default).
- **Query collapse (Step 3).** The per-feature extraction issues a single ordered child query plus an in-Python type partition instead of 2–3 separate queries; byte-neutral because the intervals are re-sorted before concatenation.

A counter-example is instructive: an attempt to build the two Step-5 feature databases concurrently was byte-identical but consistently *slower*, because `gffutils.create_db` is GIL-bound pure-Python parsing — two threads time-slice on one core rather than running in parallel. It was reverted. The general rule that emerged: thread-based concurrency helps only GIL-releasing work (subprocesses, parasail), never CPU-bound pure-Python.

Three robustness fixes are *not* byte-neutral but are corrective, and each is byte-identical on every valid annotation (the 24-cell matrix stays green with no golden-file edit). First, v2.0.0 wraps the vendored Liftoff call with a recursion-limit guard and a full-traceback dump, and handles the `gffutils.FeatureNotFoundError` that crashed v1.0.8 mid-genome. Second, the gene-like child double-lift fix (§2.6) changes output only on annotations that previously crashed or double-lifted; it is byte-identical on every top-level-only annotation. Third, the GFF3 writer's malformed-feature guard now catches the project's own `LiftOnError` base, so a single transcript with invalid (e.g. inverted `start > end`) coordinates — which the ORF-rescue path can produce on a pathological reference model — is dropped with a logged warning instead of aborting the whole-genome write phase; this lets v2.0.0 complete a full dog→cat lift (61,276 transcripts) that one malformed feature would otherwise have terminated at 11,048.

### 2.8 Benchmark design and evaluation

This report focuses on the regime that matters for real use: **whole-genome, end-to-end lift-overs**, where each tool runs from scratch on a complete target assembly. We evaluate **seventeen whole genomes** (Table 2) organized as a **four-tier divergence ladder** that spans the full range a lift-over encounters in practice: five **same-species** pairs (independent assemblies or cultivars of *Arabidopsis*, honeybee, rice, maize, and tomato), one congeneric **cross-species** pair (*D. melanogaster* → *D. erecta*), three **close** cross-species pairs (human→gorilla, mouse→*Mus caroli*, tomato→potato), three **distant** cross-species pairs (dog→cat, human→macaque, human→marmoset), and five **very-distant** pairs that probe the regime where genome-wide DNA synteny has essentially disappeared (*Arabidopsis*→rice, eudicot→monocot; human→zebrafish, mammal→fish; human→chicken, mammal→bird; human→frog, mammal→amphibian; *Drosophila*→honeybee). A companion suite of single-chromosome **subset** benchmarks is used only as the *controlled engine comparison* (§3.2, §3.7) — identical cached aligner inputs at `-t1`, which isolates LiftOn's own code — and is archived in full in `MASTER_COMPARISON_REPORT.md`; it is not part of the whole-genome results below.

**Tools.** Four tools are compared: *Liftoff* (DNA lift-over via minimap2), *miniprot* (protein-to-genome), *LiftOn v1.0.8* (the previous stable release; git tag `e503643d`, isolated environment), and *LiftOn v2.0.0* (development HEAD `5a8e382` plus the inverted-coordinate write guard of §2.7, including also the gene-like child-lift fix and the Step-7 error-handling fix). Targets are independent assemblies, so a perfect lift is not guaranteed even within a species.

**Tool-neutral scoring.** Every tool's output GFF3 is re-scored by the *same* parasail kernel (`benchmarks/compare/evaluator.py`), independent of each tool's self-reported numbers. A lifted transcript's CDS is translated and globally aligned to the reference protein with parasail [5] (BLOSUM62 [6]); the per-transcript protein identity is

$$\mathrm{PI} = \frac{\#\{\text{identical aligned residues}\}}{\#\{\text{aligned columns}\}},$$

and **mean protein identity** is the mean of $\mathrm{PI}$ over recovered coding transcripts. Coding completeness is

$$\mathrm{completeness}_\text{coding} = \frac{N_\text{recovered coding}}{N_\text{reference coding}}.$$

Tool transcripts are matched to reference IDs copy-suffix-aware; miniprot via its `Target` attribute. Output validity is the `gff3-validate` error count, applied with one consistent validator on all outputs.

**v1.0.8 crashes on five of the seventeen.** All five fail with the same `FeatureNotFoundError → __str__ TypeError` defect (§1.3, §2.7) that aborts the Liftoff-gene serialization mid-run. On full RefSeq *Arabidopsis* and rice it leaves a *scorable* partial annotation (28% / 77% complete), so its mean identity there is computed over the fraction it completed and reported as the robustness result of §3.4. ## 3. Results

### 3.1 Accuracy: leading the field on every whole genome

On **all seventeen whole-genome lift-overs**, LiftOn v2.0.0 produces a higher mean protein identity than **both single-method baselines** (Liftoff and miniprot) — Table 3, Fig. 1 — and the margin over the better of the two **grows monotonically with divergence**: from +0.00068 on same-species *Arabidopsis* up to **+0.12881** at the mammal→bird extreme (human→chicken; Fig. 1B, positive on all 17 of 17). This is the signature of a correct dual-evidence fusion. On near-identical genomes the DNA alignment is already essentially perfect, so there is little to add; as the target diverges the DNA model frameshifts — Liftoff collapses from ~0.99 to **0.15–0.31** across the very-distant tier (0.31 eudicot→monocot, 0.16 mammal→fish, 0.15 mammal→bird) — while the protein model survives, and the per-chunk maximization of §2.3 keeps whichever signal is locally better.

Against the previous release, v2.0.0 matches or exceeds v1.0.8 on the great majority of the genomes both versions completed — every same-species, cross-species, close, and distant pair, e.g. drosophila +0.00134, human→gorilla +0.00119, mouse→caroli +0.00231, dog→cat +0.00294, human→macaque +0.00178, human→marmoset +0.00127, plus *Arabidopsis*→rice +0.00054. The only dips are at the three most extreme distances and all fractional: human→zebrafish $-0.00199$, human→chicken $-0.00022$, human→xenopus $-0.00027$, each on a transcript set both versions recover at under 25% (§3.3). On *Arabidopsis*, rice, maize, and the two tomato pairs, v1.0.8 **crashed** and never produced a comparable whole-genome number (its *Arabidopsis*/rice values are partial-run artifacts over the 28% / 77% completed before aborting; maize and tomato left no scorable annotation); v2.0.0 completes all five (§3.4).

**The recall/accuracy trade-off at extreme distance.** The five very-distant whole genomes are the most demanding test of the merge, and they expose an honest tension. On per-transcript *accuracy* LiftOn's lead over the best single method is **largest** here — from +0.04069 (eudicot→monocot) up to **+0.12881** (mammal→bird) and +0.11710 (mammal→amphibian) (Table 3). But on *recall* it is far behind: the DNA-anchored lift places only a small fraction of transcripts (coding completeness 4.5–24.4%), whereas miniprot's unconstrained protein search recovers far more (66.7–85.8%) at substantially lower identity (§3.3). Because both LiftOn versions recover essentially the same small set on these genomes, the accuracy comparison there is a clean read on per-transcript quality: where LiftOn places a transcript at extreme distance, it places it markedly more accurately than either single-method baseline.

### 3.2 Where the accuracy gain comes from — controlled-input comparison

> *Engine aside.* This subsection isolates LiftOn's own algorithm on **identical cached aligner inputs** (a single-chromosome subset at `-t1`), not a whole-genome end-to-end run; it explains the source of the v2.0.0 whole-genome lead in §3.1.

On identical inputs, the best-of-outcome merge (§2.4) improves many transcripts and regresses almost none (Table 4). On divergent pairs the effect is substantial — drosophila 115 improved / 1 regressed ($+0.00368$ mean PI), mouse→rat 83 improved / 0 regressed ($+0.00399$) — and on near-identical same-species pairs there is little to improve (rice and bee unchanged, *Arabidopsis* 9/0). This is the empirical signature of the strict-non-regression design of §2.4: the merge replaces a Liftoff CDS only when the merged model scores strictly higher against the reference protein.

### 3.3 Completeness

On coding-transcript completeness the seventeen whole genomes split into two regimes (Table 5; Fig. 2). On the **same-species through distant** genomes every tool that completes recovers most or all of the coding set: the same-species and cross-species pairs sit at ≥ 96.5% for the completing tools, and the close and distant cross-species mammalian pairs at 88–99%. On the **very-distant** pairs an honest trade-off emerges: because LiftOn anchors each model on a Liftoff DNA lift, when genome-wide synteny is gone the anchor almost vanishes and LiftOn recovers only a small fraction (4.5–24.4%), whereas miniprot's unconstrained protein search recovers far more (66.7–85.8%) — but, as §3.1 shows, at substantially lower per-transcript identity. The relationship to Liftoff captures the design exactly: on the **very-distant** tier LiftOn's completeness *exceeds* Liftoff's (e.g. *Arabidopsis*→rice 14.8% vs 1.6%, *Drosophila*→bee 8.5% vs 0.6%), because the protein evidence rescues transcripts the DNA alignment alone could not place in-frame; on the **same-species** genomes the two match; and on the **close and distant mammalian** pairs LiftOn is marginally *below* Liftoff (by ≤ 0.7%), a small recall cost it pays for the per-transcript accuracy gain of §3.1 — it prioritizes consistent, in-frame transcript structure over placing every marginal model.

### 3.4 Full-genome robustness: finishing genomes v1.0.8 could not

The single largest practical advance is robustness on whole genomes. On **five** of the seventeen, v1.0.8 aborts mid-run with the `FeatureNotFoundError → __str__ TypeError` defect (§2.7). On full RefSeq *Arabidopsis* and rice it leaves a scorable partial annotation; v2.0.0 completes both, recovering tens of thousands more coding transcripts (Table 6; Fig. 3): **+34,589** on *Arabidopsis* (13,618 → 48,207; coding completeness 28.21% → 99.88%) and **+9,594** on rice (32,933 → 42,527; 77.34% → 99.88%). On maize and the two tomato pairs v1.0.8 produces no scorable whole-genome annotation at all, while v2.0.0 completes each (maize 96.5%, tomato 99.5%, tomato→potato 93.0%).

A second, distinct robustness fix lives in v2.0.0's own writer (§2.7). On the dog→cat lift the ORF-rescue path produced a single transcript with inverted (`start > end`) coordinates, which the GFF3 writer correctly rejected — but, because the writer's exception guard did not catch the project's own validation error, that one malformed feature propagated out of the parent-thread write phase and aborted the entire run at 11,048 of ~61,000 transcripts. The widened malformed-feature guard now drops that one transcript with a logged warning and completes the genome (61,276 transcripts; coding completeness 98.01%, leading the best single baseline by +0.031 mean identity). It is byte-identical on every valid annotation. This is the same class of "one bad feature must not abort a whole genome" hardening as the §2.6 gene-like fix, and like it was surfaced precisely *because* the benchmark exercised full genomes at cross-species distance.

These crash and robustness fixes are the only places a whole-genome completeness gap reflects a tool failure rather than genuine biology: the five very-distant fulls, where both versions recover only a small fraction, are genuine low-recall-at-extreme-distance runs that both versions complete, and they belong with the accuracy discussion (§3.1, §3.3), not here.

### 3.7 Engine efficiency (controlled comparison)

They are not whole-genome end-to-end wall times; the whole-genome resource picture is stated separately at the end of this section.

**Whole-genome resources.** On a fresh whole-genome run the trade is different and worth stating plainly: v2.0.0 does substantially *more work* than v1.0.8 — it lifts the full gene-like feature set (e.g. bee·full emits 138,297 CDS features vs v1.0.8's 16,432) — so although peak memory still falls (bee·full 23.1 GB → 7.1 GB; full *Drosophila* 7.1 GB for v2.0.0 vs 19.7 GB for v1.0.8), end-to-end wall time can rise on the run that completes the entire expanded annotation (full rice ≈ 2.8 h at 8.2 GB peak). The decisive point is that the bounded memory is what makes these whole-genome runs *possible* on commodity hardware at all — v1.0.8 could not finish *Arabidopsis* or rice regardless of time (§3.4).

## 4. Discussion

**Why conservative dual-evidence fusion wins.** The central empirical result (§3.1, Fig. 1) is that LiftOn's advantage over the best single method *grows with divergence and is largest on the most divergent pairs*. This is the expected behavior of a correct fusion: on near-identical genomes the DNA alignment is already essentially perfect and there is nothing to add, whereas on divergent genomes the DNA model frameshifts while the protein model survives, and the per-chunk maximization of §2.3 keeps whichever signal is locally better. The best-of-outcome guard of §2.4 is what makes this safe in practice: by adopting the merged model only when it verifiably improves the *emitted* protein, v2.0.0 turns a merge that could backfire on individual transcripts into a strict per-transcript non-regression — visible as the 115/1 and 83/0 improved/regressed counts of §3.2.

**Whole-genome, T2T, and pangenome scale.** The combination of the bounded-memory aligner (§2.2) and the robustness fixes (§2.7) changes what is feasible, not merely how fast it runs. v1.0.8 could not finish full RefSeq *Arabidopsis* or rice at all; v2.0.0 finishes them at ~99.9% coding completeness on commodity memory. As reference catalogs move to T2T assemblies and pangenome graphs multiply the number of genomes to annotate, an annotation lifter that is both memory-bounded and crash-robust on full genomes is a prerequisite, not a convenience. The accurate recovery of pseudogenes and non-coding RNA genes (§3.5) further matters because these features are increasingly central to regulatory and disease genomics yet are exactly what coding-only lifters discard.

**Implications for downstream analysis.** More accurate, in-frame CDS models reduce spurious frameshift and premature-stop calls that propagate into variant annotation and loss-of-function assessment; more complete non-coding annotation improves regulatory and small-RNA analyses; and consistent transcript structure across a clade improves ortholog assignment and comparative-genomic inference. The tool-neutral re-scoring protocol (§2.8) is itself a reusable contribution: by re-aligning every tool's output with one parasail kernel, it makes cross-tool accuracy claims independent of each tool's self-reported and incomparable identity definitions.

**The byte-identity discipline as a software-engineering contribution.** A recurring failure mode in scientific software is that performance refactoring silently perturbs results. v2.0.0's 24-cell byte-identity matrix (§2.7) makes this impossible by construction for the entire class of intended-neutral optimizations: every combination of streaming, in-process I/O, thread count, and native bindings must reproduce the default output byte-for-byte, so a refactor either preserves the bytes or fails CI. This cleanly separates two kinds of change — output-neutral engineering (free, below the line) and deliberate algorithmic change (opt-out-flagged, above the line) — and is directly transferable to other deterministic bioinformatics pipelines. The reverted concurrent-DB-build experiment (§2.7) illustrates the complementary lesson: thread-based concurrency helps only GIL-releasing work, a distinction the matrix made cheap to test.

## 5. Limitations

The benchmark and the tool are presented with the following honest caveats:

1. **Coding recall at extreme divergence.** Because LiftOn anchors each model on a Liftoff DNA lift, when genome-wide synteny is essentially gone miniprot's unconstrained protein search recovers far more coding transcripts than LiftOn across the whole very-distant tier (§3.3; LiftOn 4.5–24.4% vs miniprot 66.7–85.8%; e.g. *Arabidopsis*→rice 76.7% vs 14.8%, human→zebrafish 82.2% vs 4.5%). LiftOn prioritizes per-transcript accuracy and consistent transcript structure over raw recall.
2. **A marginal recall cost on close/distant mammalian pairs.** On the close and distant cross-species mammalian genomes (human→gorilla/macaque/marmoset, mouse→caroli, dog→cat), LiftOn v2.0.0's coding completeness is fractionally below the DNA-only Liftoff (by ≤ 0.7%; §3.3) — the recall side of the precision/recall trade-off it makes for the per-transcript accuracy lead of §3.1.
3. **Fractional accuracy dips vs v1.0.8 at the most extreme distances.** LiftOn v2.0.0 leads both single-method baselines on all seventeen whole genomes, but on the three most divergent — human→zebrafish ($-0.00199$), human→chicken ($-0.00022$), and human→xenopus ($-0.00027$; §3.1) — it is fractionally below the previous release v1.0.8, each on a transcript set both versions recover at under 25%.
4. **Nested miRNA products.** Mature `miRNA` recovery remains low (1/428 on full *Arabidopsis*; §3.5), a pre-existing limitation of transcript-centric handling of products nested under `primary_transcript`, not addressed in this release.
5. **Validity load on gene-like-heavy whole genomes.** v2.0.0 emits fewer GFF3 errors than v1.0.8 on every genome both produced, but where it lifts the full gene-like feature set onto a divergent or repeat-rich assembly (maize, tomato→potato, dog→cat, *Arabidopsis*→rice) it carries somewhat more than the DNA-only Liftoff's stripped-down output (§3.6).
5. **Full-genome compute cost.** v2.0.0's bounded memory makes full-genome runs feasible, but completing the full gene-like feature set is more total work than v1.0.8's coding-only lift, so a fresh whole-genome run can take longer in wall time even as peak memory falls (§3.7).

## 6. Availability and reproducibility

LiftOn is open-source (GitHub: [github.com/Kuanhao-Chao/LiftOn](https://github.com/Kuanhao-Chao/LiftOn)) with documentation at [ccb.jhu.edu/lifton](https://ccb.jhu.edu/lifton/). It installs via conda/bioconda (`parasail`, `pysam`, `pyfaidx`, `gffutils`, `miniprot`, `minimap2`) and exposes the CLI `lifton` (plus a `gff3-validate` console script). A typical whole-genome invocation is:

```
lifton -g reference.gff3 reference.fa target.fa -o lifton.gff3
```

The benchmark harness, the four-way comparison, the tool-neutral re-scoring evaluator, and all source data and figures are archived under `benchmarks/compare/` (`evaluator.py`, `fourway_results.json`, `version_compare.results.json`, `MASTER_COMPARISON_REPORT.md`, and the figures reproduced here). The byte-identity regression suite, including the 24-cell matrix, ships under `tests/`. Comparisons in this report were run with a pinned conda environment, `PYTHONHASHSEED=0`, v1.0.8 at git tag `e503643d` (`lifton_stable` environment), and v2.0.0 at development HEAD `5a8e382` plus the inverted-coordinate write guard (§2.7) — which includes also the gene-like child-lift fix and the Step-7 error-handling fix (`lifton_devel`).
