This is a companion to *[A faster, verified Wheeler-graph recognizer](/reports/wgt-technical-report/)*. That report is about **deciding** whether a graph is a Wheeler graph. This one is about **using** Wheeler graphs: turning a multiple-sequence alignment into an index you can actually query, and getting back the one thing a pangenome index is for — *which species, and where*.

## 1. The problem: a pangenome index that keeps the species

A pangenome replaces a single reference with many genomes at once. To search it, the standard move is to fold the sequences into a graph and index that graph. The most common construction is a **De Bruijn graph**: slide a window of length $k$ over every sequence and merge identical $k$-mers into one node. It is compact and Wheeler-friendly, and it answers *does this pattern occur*. But the merge is exactly the problem. Once two species share a $k$-mer, that node belongs to *both*; a path through the graph can splice together $k$-mers from different genomes that never co-occurred in any single one. The index then accepts **recombinant** strings that no input sequence contains, and — fatally for pangenomics — it cannot tell you which species an occurrence came from, because the node no longer remembers. The graph answers *count* but has thrown away *locate-with-provenance*.

That is the gap this index closes. We want, for an exact pattern $P$ and a set of aligned genomes, **every** occurrence reported as a tuple — species, source sequence, genomic start–end, strand — with no recombinant false positives and no loss of position. The construction that delivers it is not a De Bruijn graph at all but a **tagged suffix Wheeler graph**: a multi-string Burrows–Wheeler Transform of the aligned sequences, with one structural trick that keeps the species separated.

## 2. The algorithm: a tagged multi-string BWT

### 2.1 One block, one multi-string BWT

The unit of construction is a single **alignment block** — a column-aligned slice of the MSA (a `multiz`/MAF block [6]) in which each row is one species' sequence over that interval. We strip alignment gaps and build the index on the *forward* ungapped sequences $S_0,\dots,S_{a-1}$ (one per species $a$).

The index is the Burrows–Wheeler Transform [5] of the concatenation

$$
T \;=\; S_0\,\#_0\,S_1\,\#_1\,\cdots\,S_{a-1}\,\#_{a-1},
$$

with one separator per sequence (written $\#_i$) and — the key choice — the separators **distinct and ordered below the DNA alphabet**: $\#_i = i$ for document $i$, so that $\#_0 < \#_1 < \cdots < \#_{a-1} < \texttt{A} < \texttt{C} < \texttt{G} < \texttt{T}$. A single shared separator (the textbook multi-string trick) would make the cyclic LF-mapping inconsistent at document boundaries; distinct separators make the BWT's LF-mapping exact and, as a bonus, give every document a unique rank-0 anchor. We compute the **cyclic-rotation suffix array** $\mathrm{SA}$ by prefix doubling in $O(n\log^2 n)$ (a brute rotation sort is kept as the test oracle), then $\mathrm{BWT}[i] = T[(\mathrm{SA}[i]-1)\bmod n]$, the symbol counts $C[\cdot]$, and a per-symbol rank structure — the same FM-index machinery [3] a string uses.

### 2.2 Backward search: the interval *is* the occurrences

To search for a DNA pattern $P$ we run the standard FM-index **backward search** over this BWT, narrowing a half-open interval $[\,\mathrm{lo},\mathrm{hi}\,)$ symbol by symbol. Two properties make this exact and species-clean:

- **Exactness.** When backward search terminates, $[\mathrm{lo},\mathrm{hi})$ contains *exactly* the suffixes of $T$ that start with $P$, so $\mathrm{hi}-\mathrm{lo}$ is the occurrence count and the rows of the interval enumerate the occurrences — no post-filtering.
- **No boundary crossing.** Because the separators are symbols *outside* the DNA alphabet and $P$ is pure DNA, $P$ can never span a $\#_i$. Every occurrence therefore lies wholly inside one species' sequence, and the interval never mixes provenance.

`count`$(P)$ returns $\mathrm{hi}-\mathrm{lo}$ directly.

### 2.3 From an interval to species + position

A hit's row $i\in[\mathrm{lo},\mathrm{hi})$ is turned into a coordinate in two reads:

1. **Which species.** A **document array** $\mathrm{DOC}[i]=\mathrm{doc}[\mathrm{SA}[i]]$ records, for each suffix, the document it belongs to. One lookup gives the species.
2. **Which position.** A **sampled suffix array** recovers the text position $p=\mathrm{SA}[i]$ (LF-walking to the nearest sample), from which the within-species offset is $p-\mathrm{docstart}[\mathrm{doc}]$.

The offset is then pushed through a **genomic coordinate transform** that respects the MSA's strand bookkeeping (UCSC convention). For a block coordinate with `start`, `srcSize`, and `strand`, a hit of length $m$ at ungapped offset `pos` maps to

$$
(+)\;\; [\,\texttt{start}+\texttt{pos},\;\; \texttt{start}+\texttt{pos}+m\,), \qquad
(-)\;\; [\,\texttt{srcSize}-(\texttt{start}+\texttt{pos}+m),\;\; \texttt{srcSize}-(\texttt{start}+\texttt{pos})\,),
$$

so that on the minus strand the genomic interval is the one whose forward strand spells $\mathrm{revcomp}(P)$. Each occurrence comes back as `{species, src, gstart, gend, strand}` — multi-hit, exact, any pattern length.

### 2.4 r-index locate: one toehold, then φ

Holding a full sampled suffix array per block is wasteful when a query has many occurrences. The index therefore supports the **r-index** representation [4]: store the BWT as its $r$ maximal equal-symbol runs and sample the suffix array only at run boundaries. Backward search keeps a single position — the **toehold** $\mathrm{SA}[\mathrm{hi}-1]$ — and the **$\varphi$ function** (a predecessor query over the run-head samples plus a linear offset) walks from one occurrence to the next in $O(1)$ predecessor work each, independent of how long the BWT runs are. The located set is identical to the full-SA locate; only the memory changes. (The cyclic-wrap edge case of $\varphi$ is provably unreachable here: the unique-largest separator forces SA-position 0 to be a run head, so it is always a sampled key — checked over thousands of instances with zero counterexamples.)

## 3. The order is a Wheeler order — certified

The construction rests on a theorem: the **suffix-array rank** of a string (or multi-string) *is* a Wheeler order of its BWT-graph [2]. That is what lets us index the graph with FM-index machinery in the first place, and it is why the whole pipeline needs no solver at scale — the order comes for free from the suffix sort.

We do not just assert this; we **certify** it. On small blocks the index emits its suffix graph as a DOT file and runs the project's Wheeler-graph recognizer on it. The recognizer, which searches the entire space of node orderings (§*[recognizer report](/reports/wgt-technical-report/)*), returns verdict *accept*, its emitted order passes independent axiom re-validation, **and** that order is order-isomorphic to the suffix-array rank the index used (`recognizer_iso_to_sa_rank = true` in the certification test). The recognizer lands on exactly the order the suffix sort produced — closing the loop between the theory (SA rank is a Wheeler order) and the implementation (the order this index queries). Recognition is the *gatekeeper*; this index is what you build once recognition is guaranteed by construction.

## 4. Resolution: keeping the species, with no false positives

**Result:** over 71 multi-species yeast blocks and 653 queries, the suffix index returns the **correct species set on 100% of queries with 0 false positives**; a De Bruijn graph over the same blocks resolves species on **0%** and accepts **every** one of 592 recombinant controls as a false positive. The price is a **~1.2× larger** index (257 vs 211 median nodes per block).

This is the result that motivates the whole construction, measured against a brute-force oracle that enumerates occurrences directly. We took the 71 chrI alignment blocks containing more than one species (4 species each), generated 653 multi-species queries, and asked each index which species contain the pattern. The suffix index, reading its document array, named the correct set every time. The De Bruijn graph cannot answer the question at all — its merged $k$-mer nodes have no species identity — so it scores zero. We then built **recombinant controls**: strings that splice $k$-mers from different species so they exist as a *path* in the De Bruijn graph but in no real sequence. The De Bruijn index accepted all 592; the suffix index, whose interval cannot cross a species boundary, rejected all 592 (0 false positives). RevDet (a column-automaton construction) behaves like De Bruijn here — its accepted language is a superset on all 71 blocks.

The honest cost is size: the suffix index keeps every species' sequence distinct, so it is larger — a median of 257 nodes per block versus 211 for De Bruijn and 117 for RevDet, about 1.2× the De Bruijn graph. That is the trade the rest of the report quantifies: **~1.2× the nodes buys exact, native species-and-position resolution** that the compact constructions structurally cannot provide.

## 5. Architecture: shard, then route

A single Wheeler graph for the whole genome is not an option. The whole-genome multi-string is **43.4 million** suffix nodes — orders of magnitude past the size the recognizer decides and impractical to index as one object. So the index is **sharded**: one small multi-string BWT per alignment block. Each block is tiny (tens to hundreds of bases per species), builds in milliseconds, and is embarrassingly parallel across the 44,063 blocks of the genome.

Sharding moves the cost to query time — a naive genome query would touch all 44,063 blocks — so the index adds a **sound w-mer router**. For each block we record the set of length-$w$ substrings its sequences contain; a global inverted map sends each $w$-mer to the blocks that contain it. A query intersects the maps of its own $w$-mers to get a small candidate set, then runs exact locate only there. The prefilter is **sound by construction**: if $P$ occurs in a block, *every* length-$w$ substring of $P$ occurs in that block, so a true container is never skipped; the per-block exact locate then removes any false survivors. The result is a query whose cost is independent of genome size — it scales with the number of blocks a pattern actually lands in, not with how many blocks exist (§6, §7).

Two engineering pieces make this deployable. A **C++ port** (`index/cpp/wg_suffix`) reimplements the per-block backward search and locate; on a real block it runs **90× faster** than the Python reference (0.39 vs 35.3 µs per locate, 200k repetitions). And a **serialization** layer writes each block's arrays to disk, so the genome index is stored once and **loaded on demand**: keep the small router in memory, leave the 44k block indexes on disk, and pull only the blocks a query routes to. That is the architecture the human-scale projection (§9) requires.

## 6. How to query

The index ships three interfaces over the same algorithm. All Python commands use the Biopython environment (`~/miniconda3/envs/myenv/bin/python`); paths are relative to the repository root.

### 6.1 Command line (Python)

`index/query.py` builds an index from a block FASTA (or loads a recognizer `out__` directory) and searches a pattern. With `--locate` it reports the genomic tuple for every occurrence (it reads the `<fasta>.coords.json` sidecar that the genome builder writes):

```bash
# count + locate a pattern in one block, with species + genomic coordinates
~/miniconda3/envs/myenv/bin/python index/query.py block.fa --pattern CATTACCC --locate -a 4

# query an existing recognizer output directory instead of building from FASTA
~/miniconda3/envs/myenv/bin/python index/query.py --iol out__block --pattern CATTACCC --no-reverse
```

Flags: `--pattern P` (required), `--locate` (emit `species, src, gstart, gend, strand`), `-a` (rows/species to index), `-l` (cap sequence length), `-k` (De Bruijn $k$ when comparing constructions), `--no-reverse` (search $P$ as given; the default reverses it because recognizer paths spell sequences in reverse).

### 6.2 Python API

`SuffixIndex` is one block; `PangenomeIndex` is the sharded, routed genome index.

```python
from index.suffix_index import SuffixIndex
from index.pangenome_index import PangenomeIndex

# one block: count, then locate with species + genomic position
idx = SuffixIndex.from_fasta("block.fa", a=4, s=4, sample="runs")   # 'runs' = r-index locate
n   = idx.count("CATTACCC")                       # number of occurrences
for h in idx.locate("CATTACCC"):                  # each hit is a dict
    print(h["species"], h["src"], h["gstart"], h["gend"], h["strand"])

# a whole chromosome, routed: build the global w-mer map once, then locate genome-wide
pg = PangenomeIndex(block_fastas, a=4, s=4, w=8).build_global()
hits = pg.locate_routed("CATTACCC")               # touches only candidate blocks
species = pg.species_of("CATTACCC")               # set of species containing P
```

A located hit carries `record_idx` and `ungapped_pos` always, and `species, src, gstart, gend, strand` when coordinates are available; `PangenomeIndex` adds the `block` it came from. `locate` and `locate_routed` return the same set — the router only changes *which* blocks are touched, never the answer.

### 6.3 C++ (per block)

```bash
make -C index/cpp wg_suffix
index/cpp/wg_suffix block.fa --a 4 --l -1 --s 4 --coords block.coords.json --locate CATTACCC
index/cpp/wg_suffix block.fa --a 4 --l -1 --s 4 --queries patterns.txt      # batch
index/cpp/wg_suffix block.fa --a 4 --l -1 --s 4 --bench 200000              # us_per_locate
index/cpp/wg_suffix block.fa --a 4 --l -1 --s 4 --info                       # n, r, samples, sigma
```

## 7. Indexing the whole yeast genome

**Result:** the complete sacCer3 genome — **17 sequences, 44,063 MAF blocks, 43.4 M suffix nodes** — indexes in **318 s** single-threaded (≈5.3 min; embarrassingly parallel per chromosome) into **334 MB on disk**, which is **33× smaller** than the **11 GB** needed to hold every block index in memory at once. A routed genome-wide locate runs in **~2.6 ms**, and the build is verified end-to-end: **0 mismatches** across all 17 chromosomes, with **8,687** located hits cross-checked against the real chromosome sequences.

The per-chromosome cost is linear in block count — chrIV (the largest, 5,884 blocks) takes 41 s and 1.27 GB peak RAM, which bounds the per-chromosome build; chrM (318 blocks) takes about a second. The whole genome builds **directly from each block's FASTA** — no De Bruijn generator, no recognizer in the loop — so it finishes in minutes, not the hours a generate-and-recognize pipeline would need.

The chrI sweep shows the two knobs. The species count $a$ scales build time and memory roughly linearly (more sequence ⇒ a bigger index). The suffix-array sample rate $s$ is a **size↔speed dial**: raising $s$ from 1 to 16 shrinks the on-disk arrays (7.6→5.0 MB on chrI at $a{=}4$) while slowing routed locate (0.86→1.37 ms) — you choose where to sit. The router touches a median of about 8–18 of 992 blocks per query across the sweep, keeping routed locate well under 2 ms throughout.

On memory, the audit-driven design matters: replacing the dense $O(\sigma n)$ rank arrays with block-rank $O(\sigma n/64)$ arrays (57× smaller on a real block) and dropping the full suffix array in `runs` mode leaves the **router** — the per-block $w$-mer sets and the global map — as the memory-binding structure. Holding everything in RAM is ~11 GB; the on-disk arrays serialization is 334 MB and reloads without re-sorting suffixes. Hence the practical shape: a small in-memory router over an on-disk, load-on-demand block index.

## 8. Correctness: an adversarial audit

**Result:** an adversarial audit checks **8 pattern-matching algorithms** against independent brute oracles over **more than 4,000,000 cases**, finds and fixes exactly **one** degenerate bug (`DAWG.count("")` returned the text length; fixed to 0), and otherwise records **0 mismatches** — including at full-genome scale (44,063 blocks, 8,687 reference-genome checks).

Every algorithm is tested against a separate, obviously-correct procedure, not against itself: a brute cyclic-SA sort, a brute BWT/rank, a hand-built inverse suffix array, a brute substring scan, and an independently-written coordinate transform. The De Bruijn `WGIndex` is checked on **2,067,185** patterns over **8,000** random graphs; the coordinate transform on **~1.6 M** cases over both strands, genome-anchored to the real chromosomes; the r-index $\varphi$/toehold on **>160,000** assertions (including the wrap-edge case proven structurally unreachable); the C++ port on an **exhaustive 27,280-case** parity sweep against the Python reference; and a cross-engine stress test confirms the suffix, r-index, and DAWG engines return identical occurrence sets on **13,858** queries. Each scale/speed benchmark above also carries a built-in `locate == locate_routed == ∪oracle` gate — which is what lets the §7 numbers be reported as verified rather than merely produced.

## 9. Toward human scale

A human-scale MSA is roughly **262×** the yeast genome by aligned length, which sets the projection: about **11.6 M** alignment blocks, a single-core build of **~23 h** (minutes in parallel — the build is per-block independent), **~86 GB** on disk, and **~2.8 TB** if every block index were held in memory at once. The routed query stays essentially flat, because its cost depends on how many blocks a pattern lands in, not on how many blocks exist.

The projection makes the binding constraint explicit and answers it. Holding the whole index in RAM is infeasible at ~2.8 TB; ~86 GB on disk is entirely practical. The serialization and load-on-demand router built here are exactly the path: keep the small router resident, store the block indexes on disk, and load only the candidate blocks per query. The next compression target is the router itself — its global $w$-mer map is linear in genome size — and a **minimizer/sketch** of that map is the recommended step before a full human run, alongside porting the r-index $\varphi$-locate and the router to C++.

## 11. Conclusion

A De Bruijn pangenome graph answers *whether* a pattern occurs but forgets *who* and *where*. Reframing the index as a **tagged multi-string BWT** — distinct per-species separators, a document array for species, an r-index sampled suffix array for position — recovers exact, native species-and-position resolution at a modest ~1.2× size cost, with the suffix order guaranteed (and independently recognizer-certified) to be a Wheeler order. The construction shards cleanly to the whole yeast genome (44,063 blocks, 318 s, 334 MB on disk, ~2.6 ms routed) and is verified against independent oracles over more than four million cases with zero mismatches. The same design — a small in-memory router over an on-disk, load-on-demand block index — projects to human scale, where it is the practical path. Recognition decides *can this graph be indexed*; this index is what you build, and query, once the answer is yes.
