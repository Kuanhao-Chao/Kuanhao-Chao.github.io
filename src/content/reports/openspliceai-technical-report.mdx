## 1. Introduction

### 1.1 Splice-site prediction and SpliceAI

Pre-mRNA splicing removes introns and joins exons, and the choice of splice sites — the donor (5′) and acceptor (3′) boundaries — determines the protein a gene encodes. Predicting those boundaries from primary sequence alone is a foundational genomics problem with direct clinical stakes: a large fraction of disease-causing variants act by creating or destroying splice sites, often far from the canonical dinucleotides. SpliceAI [2] established the state of the art by framing the task as dense, per-nucleotide three-way classification — every position is *not-a-site*, *acceptor*, or *donor* — solved with a deep residual dilated 1-D convolutional network that reads up to 10,000 nt of flanking context. Its variant module, which scores how a substitution shifts predicted splicing, became a standard component of clinical variant-interpretation pipelines.

### 1.2 Why a reimplementation

SpliceAI is influential but, as a software artifact, closed in several practical ways:

1. **Runtime lock-in.** The released model is TensorFlow/Keras weights. Modern training, profiling, and deployment tooling has consolidated around PyTorch [3], and the original code is not maintained against it.
2. **No supported retraining.** The published weights are human-only, and there is no first-class, documented path to retrain the architecture on a new species, a new annotation, or a different context size — exactly what comparative and non-model-organism genomics needs.
3. **A pipeline reduced to one entry point.** Variant scoring is exposed, but the steps that produce a model — building one-hot windowed training data from a genome and annotation, training, calibrating probabilities, and running genome-scale prediction — are not packaged as a usable, composable workflow.
4. **Scaling friction.** The original variant scorer processes one variant at a time, which is adequate for a clinical VCF but far too slow for genome-wide studies that score billions of SNVs.

### 1.3 What OpenSpliceAI adds

OpenSpliceAI [1] is a from-scratch PyTorch reimplementation and extension that keeps the SpliceAI architecture intact while opening the entire lifecycle. It is distributed as a single `openspliceai` console command with six composable subcommands forming an end-to-end pipeline — **create-data → train / transfer → calibrate → predict / variant** (Fig. 1B). The same `SpliceAI` network class is built by every subcommand, parameterized by one central knob (the flanking/context size), so datasets, checkpoints, and inference runs are guaranteed mutually consistent. On top of architectural parity it adds: documented **training and transfer learning** (including layer-freezing fine-tuning across species), **probability calibration**, **genome-scale prediction** with automatic long-sequence tiling, a **batched, reference-deduplicated variant scorer** for whole-genome throughput, a **model zoo** of five-seed ensembles for five species at four context sizes, and a tested, lint-clean codebase.

### 1.4 Contributions

This report documents OpenSpliceAI and validates it against the original SpliceAI on real data. The principal points are:

- A faithful PyTorch reimplementation of the SpliceAI residual dilated-CNN whose context length is tied to the dilation schedule by the exact identity $CL = 2\sum_i AR_i(W_i-1)$, reused across all six subcommands (§2).
- A head-to-head benchmark on the human MANE test set showing OpenSpliceAI **at or above** the original SpliceAI on every top-kL accuracy threshold, AUPRC, precision, and F1 for both donors and acceptors, while being **1.47× faster** on identical hardware (§3.1–3.2).
- A batched, reference-deduplicated variant-scoring engine (~**6.7×** over the per-variant loop) that makes genome-wide variant effect prediction tractable (§2.7, §3.2).
- A genome-wide campaign that re-scores **every human GRCh38 SNV** with OpenSpliceAI alongside SpliceAI's existing scores — a strictly mask-matched, tool-neutral overlay — engineered as an idempotent, GPU-capped, good-citizen SLURM workload, reported here with its current live status (§4).

## 2. Architecture and pipeline

### 2.1 The model

OpenSpliceAI defines one network, the `SpliceAI` module, and reuses it everywhere (Fig. 1A). It is a deep residual, dilated 1-D CNN that consumes **4-channel one-hot DNA** (A, C, G, T) and emits, for each output position, a **3-class softmax** over three classes — not-a-site, acceptor, and donor. Its building blocks are:

- **ResidualUnit** — a pre-activation block applying BatchNorm → LeakyReLU(0.1) → Conv1d **twice**, with symmetric padding $(W-1)\cdot AR/2$ that preserves length, then a residual add of the block's input. Convolution window $W$ and dilation (atrous) rate $AR$ are the two per-unit hyperparameters.
- **Skip connections** — a $1\times1$ convolution taps the representation after the input stem and after **every four** residual units, and these taps are summed into an accumulator. This is the mechanism by which shallow and deep features are combined before the head.
- **Cropping1D** — the accumulated signal is trimmed by $CL/2$ positions on each end, converting a window of length $SL + CL$ down to the $SL$ central positions that actually receive predictions.
- **Output head** — a final $1\times1$ convolution to 3 channels followed by softmax over the class dimension.

Two global constants fix the geometry (`openspliceai/constants.py`): the maximum context $CL_{max} = 10{,}000$ (padded onto both ends) and the prediction window $SL = 5{,}000$. Every input window is therefore $SL + CL_{max} = 15{,}000$ positions wide; the model is fed tensors of shape $(N, 4, 15000)$ and returns $(N, 3, 5000)$.

### 2.2 Flanking size: the one central parameter

The single most important parameter is the **flanking (context) size**, valid only in $\{80, 400, 2000, 10000\}$. It selects a fixed schedule of residual units and their $(W, AR)$ values (Table 2). The schedule is constructed so that the **receptive-field context length equals the flanking size exactly**, through the identity

$$CL = 2\sum_{i} AR_i\,(W_i - 1).$$

For example, the 10,000-nt model stacks 16 residual units in four dilation tiers, and $2\,[\,4\cdot1\cdot10 + 4\cdot4\cdot10 + 4\cdot10\cdot20 + 4\cdot25\cdot40\,] = 10{,}000$. Because the flanking size must agree between the dataset, the model definition, and the checkpoint, a mismatch surfaces immediately as a tensor-shape error rather than as silently wrong predictions.

| 3 | `transfer` | Fine-tune a pretrained checkpoint on a new dataset/species, optionally freezing all but the last *k* residual units. |

### 2.3 Data representation (`create-data`)

`create-data` runs two stages in one invocation. **Stage 1** (`create_datafile`) walks the annotation with `gffutils`, extracts each gene's sequence from the FASTA, **reverse-complements genes on the minus strand**, and marks splice labels (donor = 2, acceptor = 1, else 0), optionally gating sites to canonical motif pairs (GT-AG / GC-AG / AT-AC) under `--canonical-only`. The result is a per-gene HDF5 (`datafile_*.h5`) holding string datasets `SEQ`, `LABEL`, `STRAND`, `TX_START`, `TX_END`. **Stage 2** (`create_dataset`) one-hot encodes those genes into fixed-length windows and writes chunked tensor datasets `X0, X1, …` and `Y0, Y1, …` (100 genes per chunk) in `dataset_*.h5`. Encoding uses two fixed maps: nucleotides A, C, G, T → 1, 2, 3, 4 and N/padding → 0 (`IN_MAP`); labels 0, 1, 2 (`OUT_MAP`). Training reads these shards and transposes them to channels-first $(N, 4, L)$ before the model.

Chromosome assignment follows one of two policies. `--split-method human` reproduces SpliceAI's fixed train/test partition (test = chr1, 3, 5, 7, 9); `random` splits by cumulative length to a target `--split-ratio`. The training portion is then split 90:10 into train/validation.

### 2.4 Training and transfer

`train` and `transfer` build the model, then share one `train_model` loop (`openspliceai/train_base/`). Data is streamed shard-by-shard from HDF5, and every batch is cropped to the model's required context before the forward pass. The loss is either a two-dimensional categorical cross-entropy (default) or a focal loss [5] ($\gamma = 2$) for class imbalance, selectable with `--loss`. The optimizer is **AdamW** [4] (learning rate 1e-3 for training, 1e-4 for transfer), with either a `MultiStepLR` schedule (×0.5 over the final epochs) or `CosineAnnealingWarmRestarts` ($T_0 = 5$). Checkpoints are saved as **plain `state_dict`s** (`model_{epoch}.pt`, `model_best.pt`). Per-epoch metrics — top-k accuracy and AUPRC for donor and acceptor, plus per-class precision/recall/F1 — are appended to log files under a structured output tree.

`transfer` additionally loads a pretrained checkpoint, filtering out size-mismatched keys, and can **freeze all but the last `--unfreeze` residual units** so that fine-tuning on a new species adapts only the head. This is the mechanism behind OpenSpliceAI's cross-species results in the published study [1].

> **A note on `N_GPUS`.** The hyperparameter tables hardcode `N_GPUS = 2`, but the code does **not** wrap the model in `DataParallel`; it runs on a single device chosen by preference order CUDA → MPS → CPU. `N_GPUS` only scales the batch size and the remainder-trimming in batch clipping. This is a known quirk, not multi-GPU training.

### 2.5 Genome-scale prediction (`predict`)

`predict` is staged to scale to whole genomes. It optionally restricts to gene regions when given an annotation (`-a`), splits any FASTA entry longer than `--split-threshold` (default 1.5 Mb) into tiles **with flanking overlap so that predictions are seamless across tile boundaries**, one-hot encodes, runs the model, and writes `donor_predictions.bed` and `acceptor_predictions.bed`. There are two modes: the default **turbo** path streams predictions straight to BED with no intermediate file, while `--predict-all` materializes a `predict.h5` first. **If `--model` is a directory, every checkpoint in it is ensembled by averaging predictions** — the intended way to use the five-seed model zoo.

### 2.6 Variant effect scoring (`variant`)

`variant` annotates a VCF with splicing delta scores. An `Annotator` loads the reference genome (`pyfaidx`), a gene-annotation table (built-in `grch37`/`grch38`, now packaged with the tool, or a custom TSV), and one or more models (PyTorch by default, or the original Keras weights with `--model-type keras`). For each variant, within `--distance` (default 50) of each overlapping gene, it constructs reference and alternate one-hot windows (reverse-complemented on the minus strand), predicts both, and computes four **delta scores** as the maximum per-position change in the relevant class probability:

$$
\begin{aligned}
DS_{AG} &= \max\big(y^{alt}_{\text{acc}} - y^{ref}_{\text{acc}}\big), &
DS_{AL} &= \max\big(y^{ref}_{\text{acc}} - y^{alt}_{\text{acc}}\big), \\
DS_{DG} &= \max\big(y^{alt}_{\text{don}} - y^{ref}_{\text{don}}\big), &
DS_{DL} &= \max\big(y^{ref}_{\text{don}} - y^{alt}_{\text{don}}\big),
\end{aligned}
$$

for acceptor/donor gain/loss, together with their argmax **positions** $DP_{*}$. Results are written to an `OpenSpliceAI` INFO field with the format `ALLELE|SYMBOL|DS_AG|DS_AL|DS_DG|DS_DL|DP_AG|DP_AL|DP_DG|DP_DL`. The `-M/--mask` flag zeroes scores that contradict the annotation (gains at already-annotated sites; losses away from annotated sites), matching the convention of SpliceAI's *masked* precompute.

### 2.7 The batched variant scorer

The original variant path scores one window at a time — fine for a clinical VCF, far too slow for billions of SNVs. OpenSpliceAI adds `get_delta_scores_batched` (`openspliceai/variant/utils.py`), enabled by `-b/--batch-size > 1` on the PyTorch backend, with **identical per-variant math** to the scalar path but two throughput wins:

1. **Within-position reference deduplication.** The reference window at a locus depends only on (chromosome, position, gene) — *not* on the alternate allele — so the (typically three) alternate alleles at a site share a single reference forward pass. The batch is assembled as `[unique refs][all alts]`, removing the redundant reference passes (≈ 1.3–1.5× fewer windows).
2. **Batched inference.** All collected windows are run through the ensemble in sub-batches of `batch_size` under `torch.inference_mode()`, with the ensemble prediction taken as the mean across seed models. Two env-tunable knobs accelerate this on A100-class GPUs: cuDNN autotuning (`OSAI_CUDNN_BENCH`, on by default, beneficial because all windows share one length) and TF32 matmuls (`OSAI_TF32`, on by default, ~2× with ~1e-3 numeric noise that is far below the reported score resolution; set to 0 for bit-reproducible fp32).

In aggregate this is about **6.7× faster** than the per-variant loop (Fig. 3B), and it is the engine that makes the genome-wide campaign of §4 feasible.

### 2.8 Calibration (`calibrate`)

Softmax outputs are not guaranteed to be well-calibrated probabilities. `calibrate` applies post-hoc **temperature scaling** [6]: a single scalar temperature $\tau$ is fit on the validation set by minimizing NLL, and predictions become $\mathrm{softmax}(\text{logits}/\tau)$. It reports Expected Calibration Error, NLL, and Brier scores before and after, writes calibration-curve plots, and saves `temperature.pt`/`.txt` plus a full `calibrated_model.pt`.

## 3. Results

### 3.1 Per-position accuracy vs. the original SpliceAI

We benchmarked OpenSpliceAI against the original SpliceAI on the **human MANE test set** [7] — the paralog-removed test split following SpliceAI's own protocol — streaming the full test set (no sub-sampling) through both **five-model ensembles at 10,000 nt** and computing the canonical SpliceAI metrics. Both backends scored the same **420.58 million positions** (53,849 true donors and 53,849 true acceptors), with channel order fixed identically. OpenSpliceAI is **at or above** SpliceAI on every threshold accuracy (Top-0.5L/1L/2L/4L), on AUPRC, on precision, and on F1, for **both** donors and acceptors (Table 3, Fig. 2). The clearest gaps are at the operating point that matters most — Top-1L accuracy (donor +0.0245, acceptor +0.0254) and AUPRC (donor +0.0151, acceptor +0.0139) — and in precision (+0.11 each), which lifts F1 by ~0.05 despite a small recall give-back.

The recall/precision trade is worth stating plainly: OpenSpliceAI calls splice sites **more conservatively** than SpliceAI — slightly lower recall, substantially higher precision — which nets out to higher F1 and AUPRC and, as §3.3 shows, recurs in the variant comparison as better specificity. These are the *retrained* PyTorch models, which differ from the original weights; at flanking 10,000 the architectures are equivalent enough that the Keras weights can be loaded and reproduced bit-exactly through the `variant --model-type keras` path, but for accuracy the PyTorch zoo is a genuinely different, retrained model (with the largest gains, per the published study, appearing on non-human species [1]).

### 3.2 Inference speed

On the same benchmark run and the same GPU, scoring all 420.58 M positions took OpenSpliceAI **1,491 s** versus SpliceAI's **2,185 s** — a **1.47×** speedup from the PyTorch runtime alone (Fig. 3A). The larger lever is the batched variant scorer of §2.7: on the genome-wide workload it is about **6.7× faster** than the original per-variant loop (Fig. 3B), running at roughly **122 s per 34,334-SNV chunk** on a shared A100. That factor is the difference between a genome-wide SNV study being a multi-year proposition and a managed multi-month one.

### 3.3 Variant delta-score comparison

Beyond per-position accuracy, the clinically relevant question is whether the two tools *agree on the effect of a variant*. We compared OpenSpliceAI and SpliceAI delta scores over a set of **452,053 paired human SNVs** (the VCF INFO carries both an `OpenSpliceAI=` and a `SpliceAI=` field). The headline is **"tied power, better specificity"** (Table 4, Fig. 4): the two tools' maximum delta scores are moderately correlated (Pearson 0.76; agreement at the 0.5 threshold has $\kappa = 0.67$ once the shared, overwhelming negative mass is accounted for, with a positive-class Jaccard ≈ 0.50), **losses agree more than gains** (AL/DL ≈ 0.84 vs AG/DG ≈ 0.64), and OpenSpliceAI is markedly more **specific** (splice-loss AUROC 0.95 vs 0.81, roughly 3× fewer false positives) while **detection power is essentially a tie** (max-DS AUPRC 0.458 vs 0.417). OpenSpliceAI is the more conservative caller, and the entire gap lives on the *gain* side — consistent with the higher-precision behavior seen in §3.1.

| Agreement @ 0.5 ($\kappa$) | 0.67 | inflated by shared negative mass |
| Positive-class Jaccard | ≈ 0.50 | real concordance is moderate |
| Splice-loss AUROC | **0.95** (OSAI) vs 0.81 (SpliceAI) | OSAI more specific (~3× fewer FP) |
| Max-DS AUPRC (detection power) | 0.458 (OSAI) vs 0.417 (SpliceAI) | ≈ tie |
| Loss agreement (AL/DL) | 0.84 | losses agree |
| Gain agreement (AG/DG) | 0.64 | gains agree less |

**An honest, load-bearing caveat.** This bulk comparison pairs OpenSpliceAI scores computed **unmasked** against SpliceAI's **masked** precompute, and masking acts precisely on the *gain* side — so the gain/loss asymmetry above cannot be cleanly attributed to the *model* versus the *mask*. A small mask-matched re-scoring confirms the conservatism is real at decision-relevant thresholds, but resolving the attribution at scale requires a **strictly mask-matched re-scoring of the whole SNV space**. That requirement is the direct motivation for the campaign in §4.

## 4. Genome-wide SNV re-scoring campaign

### 4.1 Motivation

To settle the §3.3 attribution and produce a permanent, tool-neutral resource, we are re-scoring **every human GRCh38 SNV** with OpenSpliceAI and writing an `OpenSpliceAI=` delta-score field next to SpliceAI's existing field at every position — a strictly **mask-matched** overlay (OpenSpliceAI run with `-M 1` against SpliceAI's masked precompute). The result is an apples-to-apples comparison at every SNV in the genome, not a sampled subset, and a reusable precompute that others can query without a GPU.

### 4.2 Scale and cluster constraints

The input is partitioned into **100,000 chunks** of ~34,334 SNVs each — about **3.43 billion SNVs per model seed**. Each seed (we are running two independently drawn MANE seeds, **rs10** and **rs13**, scored as single models, not the 5-seed ensemble) is therefore a multi-thousand-GPU-hour job. Two hard limits on the shared cluster shaped the design (Fig. 5): the GPU QOS caps the account at **10 GPUs** (shared across all lab members), and the controller's `MaxJobCount` is ≈ 100,000 total array tasks (with `MaxArraySize` = 15,000). A naïve one-task-per-chunk submission (100,000 tasks for a single seed) alone exceeds the job-count cap.

### 4.3 Design

The campaign is driven by a single `campaign.sh {status|submit|resume|pause}` script over a per-chunk SLURM array, with four design choices:

- **`PER_TASK = 20`, sequential.** Each array task scores 20 chunks **one at a time** on a single GPU (batched inference already saturates the A100, so fanning out within a task would only OOM). This yields 5,000 tasks per seed — comfortably under both array and job-count caps.
- **Idempotent at chunk granularity.** A chunk is skipped if its output already has the correct record count; a partial file from a killed task is detected by count mismatch, deleted, and redone. This makes `pause`/`resume` lossless: only the handful of chunks in flight on the GPUs at pause time are ever repeated.
- **Whole-campaign GPU cap by construction.** SLURM's `%N` array throttle is *per array job*, so two seed-arrays each at `%4` could use 8 GPUs. Instead, `submit` **serializes the seeds into a dependency chain** (`seed[i] --dependency=afterany:seed[i−1]`), so only one seed-array is ever runnable. One runnable array throttled at `%CONC` ⇒ **total concurrent GPUs ≡ CONC**. We set **`CONC = 4`** — a deliberately good-citizen cap that leaves ≥ 6 of the shared 10-GPU account free for labmates. The cap can be retuned on a *live* campaign without losing work via `scontrol update … ArrayTaskThrottle=N`.

## 5. Discussion

**Reimplementation as a platform.** The value of OpenSpliceAI is not a single new number but the conversion of a closed, human-only artifact into an open, end-to-end, PyTorch-native platform. Because one model class is parameterized by the flanking size and reused across `create-data`, `train`, `transfer`, `calibrate`, `predict`, and `variant`, a user can take the architecture to a new genome, retrain or fine-tune it, calibrate it, and score variants — none of which was supported by the original release. The published study [1] shows this most clearly on non-human species, where transfer learning yields the largest gains.

**Accuracy parity-plus, with a specificity tilt.** On human MANE the retrained models are not merely on par with SpliceAI; they exceed it on the canonical threshold-accuracy and AUPRC metrics while being more conservative (higher precision, slightly lower recall). The same conservatism appears in the variant comparison as ~3× fewer false positives at tied detection power. Whether that tilt is desirable is application-dependent — a screen prioritizing recall may prefer SpliceAI's operating point — but for precision-sensitive uses it is an advantage, and calibration (§2.8) plus the per-class delta scores let a user pick an explicit threshold rather than inherit one.

**Throughput as an enabler.** The batched, reference-deduplicated scorer is what turns "score every SNV in the genome" from a thought experiment into a managed campaign. The same engineering that makes the campaign fast also makes it a *good citizen*: by serializing seeds into a dependency chain, the whole multi-seed job provably never exceeds a four-GPU footprint on a shared account, and by being idempotent at chunk granularity it can be paused and resumed without waste. The genome-wide, strictly mask-matched overlay it produces is, itself, a reusable contribution: it will let any downstream user compare the two tools at a SNV of interest without recomputation, and it removes the unmasked-vs-masked confound that limits the current variant comparison.

## 6. Limitations

1. **Recall trade-off.** OpenSpliceAI's higher precision comes with slightly lower recall than SpliceAI (§3.1); applications that weight sensitivity above specificity should choose the operating point deliberately.
2. **The variant comparison is not yet fully mask-matched at scale.** The 452 K-SNV comparison pairs unmasked OpenSpliceAI against masked SpliceAI, so the gain/loss attribution is confounded until the §4 campaign completes (§3.3).
3. **Single-device training.** Despite a hardcoded `N_GPUS = 2` in the hyperparameter tables, the code does not use `DataParallel`; training runs on one device, and `N_GPUS` only scales batch size (§2.4).
4. **Maintenance hazard.** The $(W, AR, \text{batch size})$ schedule is duplicated across several subcommands; a change in one must be mirrored in the others. This is the most likely source of subtle bugs and is flagged in the codebase.
5. **Campaign in progress.** The genome-wide overlay is not finished — at the four-GPU good-citizen cap the full two-seed run projects to ~2.8 months — so its scientific payoff (the clean mask-matched comparison) is pending completion.

## 7. Availability and reproducibility

OpenSpliceAI is open-source ([github.com/Kuanhao-Chao/OpenSpliceAI](https://github.com/Kuanhao-Chao/OpenSpliceAI)) with documentation at [ccb.jhu.edu/openspliceai](https://ccb.jhu.edu/openspliceai/). It installs from PyPI (`pip install openspliceai`, v0.0.6; Python ≥ 3.9, PyTorch ≥ 2.2.1) and exposes one console command with six subcommands. Typical invocations:

```
# build training data, then train a 10,000-nt model
openspliceai create-data --annotation-gff anno.gff --genome-fasta genome.fa --output-dir data/
openspliceai train --flanking-size 10000 --train-dataset data/dataset_train.h5 --output-dir run/

# fine-tune onto a new species (freeze all but the last residual units)
openspliceai transfer --pretrained-model run/models/model_best.pt --flanking-size 10000 ...

# score a genome and annotate a VCF (directory => 5-seed ensemble)
openspliceai predict --model models/openspliceai-mane/10000nt/ --input-sequence genome.fa ...
openspliceai variant -R genome.fa -A grch38 --model models/openspliceai-mane/10000nt/ -b 128 < in.vcf > out.vcf
```

The **model zoo** ships five random-seed checkpoints per setting — `models/openspliceai-{mane,mouse,zebrafish,arabidopsis,honeybee}/{80,400,2000,10000}nt/model_*nt_rs{10..14}.pt` — intended to be ensembled by pointing `predict`/`variant` at the directory; `models/spliceai/` holds the original Keras weights for the `--model-type keras` path. The accuracy benchmark harness streams a MANE `dataset_test.h5` through both ensembles and computes the canonical top-kL/AUPRC statistics on the full test set. The genome-wide campaign is driven by `campaign.sh {status|submit|resume|pause}` over a per-chunk SLURM array, with all parameters (paths, `-M 1 -f 10000 -D 50 -b 128`, seeds, `PER_TASK`, `CONC`) centralized in one `config.sh`; the campaign is idempotent and pause/resume-safe by construction. The figures in this report are regenerated from the on-disk benchmark JSONs and live campaign status by a single self-contained plotting script.
